## 핵심 기술 제약 요약
- JS 내 `${variable}` → 반드시 `\${variable}` 이스케이프
- 가상 태그(`{$...}`, `<!-- -->`, `<makeshop:>`) 절대 수정/삭제 금지
- CSS는 반드시 고유 컨테이너로 스코핑 (전역 셀렉터 금지)
- JS는 IIFE 패턴 필수
- UTF-8 이모지 사용 금지
- 외부 라이브러리는 CDN `<script>` 태그로만

## 프로젝트 구조
- 메인페이지: `메인페이지/Index.html`, `css.css`, `js.js`

## 2026-03-10 파트너클래스 UX 기준

- 파트너클래스 UI의 1순위 설계 대상은 수강생이다.
- 목록과 상세는 예약 전환보다 먼저 신뢰 형성과 불안 해소를 책임져야 한다.
- 파트너신청 페이지는 신청 폼이 아니라 `강의만 잘하면 되게 해주는 서비스`를 파는 제휴 입점 페이지여야 한다.
- 협회 레이어는 같은 허브 안에서 탐색만 분리하고, `협회/세미나`와 `협회원 혜택`이 클래스 레이어와 자연스럽게 이어져야 한다.
- 용어는 목록, 상세, API, n8n, NocoDB에서 완전히 통일해야 한다.
- 현재 문서 진입점은 `docs/파트너클래스/README.md` 이다.

## 2026-03-10 S1-1 UI 메모

- 강의 등록 키트 UI는 `상품명 / 자사몰 상품 링크 / 예상 판매가 / 1인 기준 수량` 4필드로 통일한다.
- 링크 입력 UX는 검색창보다 먼저 `상품 링크 또는 branduid 입력` 흐름으로 단순화한다.
- 상세 페이지 키트 섹션 제목은 `이 수업에 사용되는 재료`를 사용한다.
- 상세 키트 카드는 `상품 보기`와 `장바구니 담기`, 상단 `재료 한번에 담기` CTA를 함께 제공한다.
- 이후 S1-2는 상단 신뢰 요약과 CTA 계층을 이 키트 영역과 끊기지 않게 연결해야 한다.

## 2026-03-10 S1-2 상세 UX 메모

- 상세 상단에는 스크롤 중에도 남는 `Trust Summary Bar`를 둔다.
- 포함 내역 섹션은 아이콘 나열보다 `항목 / 상태 / 설명` 카드 구조가 이해가 빠르다.
- CTA 계층은 시각적으로도 확실히 갈린다:
  - 예약하기 = 채운 버튼
  - 선물하기 = 보조 버튼
  - 키트만 구매 = 링크
- 모바일 고정 바도 같은 계층을 유지해야 하며, 금액은 선택 인원에 따라 같이 갱신한다.

## 2026-03-10 S1-3 목록 UX 메모

- 목록 상단에는 사이드바 필터와 별개로 빠른 결정용 `퀵 필터 레일`을 둔다.
- 퀵 필터는 `카테고리 / 지역 / 형태 / 가격대 / 난이도` 5그룹으로 고정한다.
- 카드 제목 아래 신뢰 배지는 최대 3개만 보여 과밀해지지 않게 한다.
- 오프라인 카드는 `가까운 공방 보기` CTA를 카드 내부에 같이 노출하고, 온라인 카드는 지도 CTA를 숨긴다.
- 필터 상태가 새로고침 후 그대로 복원되어야 수강생의 탐색 흐름이 끊기지 않는다.

## 2026-03-10 S1-4 마이페이지 UX 메모

- 마이페이지는 예약 조회 화면이 아니라 `수강 후 다음 행동을 제안하는 화면`으로 해석한다.
- 완료 카드는 요약 정보 밑에 `후기/혜택 히어로 -> 키트 재구매 -> 같은 강사 추천` 3단 구조를 유지한다.
- 키트 재구매 UI는 장바구니보다 먼저 `상품 다시 보기` 성격의 칩 카드가 적합하다.
- 추천 클래스는 같은 강사라는 신뢰 근거가 명확해야 하므로 일반 인기 클래스 섞기를 피한다.
- 메이크샵 페이지 스타일은 `#partnerclass-my-bookings` 컨테이너 ID 기준으로만 스코핑한다.

## 2026-03-10 S1-5 어드민 정산 UX 메모

- 정산 탭은 단순 완료 처리 표가 아니라 `정산 실행 컨트롤 + 이력 확인`이 한 화면에서 이어져야 한다.
- 실행 컨트롤은 `상태 필터 / 월 / 반기 / 정산 실행 / 수동 일괄 정산` 순서로 배치한다.
- 정산서 발송 이력은 `정산 구간 / 파트너 / 등급 / 수업 수 / 총 주문 금액 / 총 적립금 / 최근 발송` 컬럼으로 충분하다.
- 운영 장애가 있을 때는 조용히 실패시키지 않고 토스트로 직접 에러 이유를 보여줘야 한다.

## 2026-03-11 S1-6 상세 FAQ UX 메모

- 상세 FAQ는 `문의용 부록`이 아니라 예약 직전 불안을 줄이는 탐색형 패널로 본다.
- 구조는 `검색 -> 카테고리 -> 결과 요약 -> 아코디언 -> 문의 전환` 순서가 가장 이해가 빠르다.
- 카테고리는 `전체 / 수강 / 키트·배송 / 파트너 / 정산 / 기타` 6개로 고정한다.
- 질문 수는 15개까지 늘리되, 한 번에 1개만 열리게 해서 스캔 속도를 유지한다.
- 결과가 없을 때는 빈 상태 문구만 끝내지 말고 바로 문의 채널로 이어줘야 한다.

## 2026-03-11 S1-7 파트너 온보딩 UX 메모

- 파트너 대시보드는 첫 화면부터 `무엇을 해야 첫 예약을 받을 수 있는지`가 보여야 한다.
- 온보딩 구조는 헤더 아래 요약 카드보다 먼저 `진행 카드 -> 자동 오픈 모달 -> 완료 모달` 3단 구성이 가장 직관적이다.
- 단계 체계는 `프로필 / 교육 / 강의 / 일정 / 키트` 5단계로 고정하고, CTA는 항상 현재 단계 한 개만 강하게 노출한다.
- 일정 단계는 일정 관리 탭으로 보내는 것만으로 끝내지 말고 첫 강의를 자동 선택하고 일정 폼까지 열어줘야 체감이 좋다.
- 완료 상태에서는 체크리스트를 계속 남기지 말고 카드 숨김 + 완료 모달 1회 노출로 전환하는 편이 깔끔하다.
