> **목적**: 새 세션에서 HR-001 `/start` 미작동 문제를 근본 해결하기 위한 킥오프 문서
> **전 세션**: 3시간 동안 3차례 표면 수정 시도 → 전부 실패. 팀미팅에서 근본 원인 확정.
> **결론**: `n8n-nodes-base.httpRequest` → `n8n-nodes-base.telegram` 노드 전면 교체

## 1. 한 줄 요약

**HR-001의 HTTP Request 노드가 `{{ $credentials.telegramApi.accessToken }}` 표현식으로 봇 토큰을 참조 중인데, n8n `$credentials` 변수는 HTTP Request 노드 컨텍스트에 주입되지 않아 항상 `undefined` → URL이 `https://api.telegram.org/botundefined/sendMessage` → HTTP 404.**

---

## 2. 소스 증거 (CTO 진단)

### 증거 1: `$credentials`는 routing-node 경로에서만 주입됨
**파일**: `/Users/jangjiho/Desktop/n8n-main/packages/core/src/execution-engine/routing-node.ts:142-172`

```
{ $credentials: credentials, $version: node.typeVersion }
```

이 주입 블록은 n8n이 "declarative 노드"(Telegram, Slack, Notion 등 서비스 전용 노드)를 실행할 때만 사용하는 `routing-node` 경로에만 존재.

### 증거 2: HttpRequestV3는 routing 경로를 전혀 안 씀
**파일**: `/Users/jangjiho/Desktop/n8n-main/packages/nodes-base/nodes/HttpRequest/V3/HttpRequestV3.node.ts`
- `execute()` 함수 (170~290라인): authentication 타입별로 `this.getCredentials(...)` 를 **코드로 직접 호출**하여 요청 옵션 조립
- `$credentials` 변수는 이 경로에 **한 줄도 주입되지 않음**
- `HttpRequestV3.node.ts:864` 의 `throw error;` 는 404 응답을 NodeApiError로 감싸 던지는 지점 (근본 원인 아님)

### 증거 3: Telegram 노드는 `requestWithAuthentication` 사용
**파일**: `/Users/jangjiho/Desktop/n8n-main/packages/nodes-base/nodes/Telegram/Telegram.node.ts:1393-1534`
**파일**: `/Users/jangjiho/Desktop/n8n-main/packages/nodes-base/nodes/Telegram/GenericFunctions.ts:70-186`
- credential은 node configuration에 바인딩만 하면 `requestWithAuthentication` 이 토큰을 자동 삽입
- URL 수동 작성 불필요
- `reply_markup`은 `replyKeyboard` fixedCollection으로 설정

## 3. 과거 3차례 수정 히스토리 (전부 표면 증상)

| 차수 | 시도 | 결과 |
|------|------|------|
| 1차 14:56 | 정규식 escape 유실 → regex 제거 + indexOf | 명령어 파싱 정상 진입 |
| 2차 15:04 | JS string literal 내 literal newline → 배열 `.join('\n')` | "invalid syntax" → "404" 에러 변화 |
| 3차 15:35 | `$credentials.telegramApi.token` → `accessToken` | **여전히 404** (필드명 문제가 아니었음) |

**교훈**: 3차 수정은 "필드 이름 문제"가 아니라 "변수 자체가 존재하지 않음"을 간과한 것.
