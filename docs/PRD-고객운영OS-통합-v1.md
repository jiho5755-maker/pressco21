## 1. 프로젝트 정의

### 1.1 한 줄 정의

PRESSCO21 고객운영 OS는 흩어진 고객, 주문, 상담, 입금, 출고, 재구매 데이터를 내부 기준으로 통합해 `하나의 고객 기준`, `하나의 주문 기준`, `하나의 운영 화면`으로 관리하는 전사 운영 시스템이다.

### 1.2 왜 지금 필요한가

현재 고객 운영은 아래처럼 분산되어 있다.

- 자사몰 주문과 회원 정보: 메이크샵 관리자
- 오픈마켓 주문: 스마트스토어/쿠팡/11번가 각 관리자
- 통합 주문/송장/재고 운영: 사방넷
- 전화주문/명세표/입금/미수: `offline-crm-v2`
- 상담: 채널톡 무료 플랜 중심
- 알림/자동화: 메이크샵 기본 기능 + 수동 발송 + 일부 n8n

이 구조는 운영은 가능하지만, 아래 문제가 누적된다.

- 고객 1명을 여러 시스템에서 따로 본다.
- 총매출, 실결제액, 실수금액이 시스템마다 다르게 보인다.
- 전화주문과 온라인주문이 분리되어 고객 이력이 끊긴다.
- 사방넷은 운영 허브지만 분석/고객 관계 관점의 데이터는 약할 수 있다.
- 상담 이력과 주문/결제/재구매가 연결되지 않는다.
- 채널/서비스를 바꾸면 과거 데이터와 운영 지식이 쉽게 분산된다.

### 1.3 이 PRD의 목표

이 문서는 `CRM v2를 거대하게 키운다`가 아니라, `Customer OS Core`라는 별도 중심 허브를 정의하고 기존 `offline-crm-v2`를 그 허브에 연결되는 안정적 실무 앱으로 유지하기 위한 기준선을 잡는다.

## 2. 제품 비전

### 2.1 최종 비전

직원이 어떤 채널에서 고객을 만나든 아래가 한 화면에서 이어져야 한다.

1. 고객 식별
2. 최근 주문/상담/입금/미수 확인
3. 전화주문 또는 후속 조치 실행
4. 메시지 발송
5. 출고/수금 상태 추적
6. 재구매와 리텐션까지 연결

### 2.2 핵심 문장

`외부 서비스는 채널과 운영을 담당하고, 고객 기준과 운영 지식은 내부가 소유한다.`

### 2.3 시스템 경계 재정의

이 프로그램은 2계층 구조를 전제로 한다.

- `Customer OS Core`
  - 통합 고객, 통합 주문, 통합 이벤트, 분석, 자동화의 기준 허브
- `Execution Apps`
  - `offline-crm-v2`, 사방넷, 메이크샵 관리자, 상담툴 등 실제 업무를 수행하는 화면

즉 `Customer OS Core`가 데이터를 통합하고, `CRM v2`는 그 데이터를 활용해 명세표/입금/미수/전화주문 같은 실무를 처리한다.

## 4. 현재 제약 조건

### 4.1 외부 시스템 제약

- 쿠팡 Open API는 판매자 기준 API 키 제약으로 인해 사방넷과 직접 개발을 동시에 병행하기 어렵다.
- 사방넷 API와 스케줄러는 유료 기능이므로 초기 단계에서는 저비용 우회 구조가 필요하다.
- 메이크샵은 주문/회원/적립금/상품 데이터가 중요하지만 개인정보 보관 및 운영 화면 제약이 있다.
- 오픈마켓은 각 채널 화면에 의존하면 통합 분석이 어렵다.

### 4.2 내부 운영 제약

- 직원은 여러 관리자 화면을 오가며 주문, 고객, 입금을 따로 처리한다.
- 전화주문은 반복되지만 시스템 흐름에 완전히 편입되어 있지 않다.
- `offline-crm-v2`는 이미 강한 기반이 있으므로, 중심 허브 기능까지 한 번에 밀어 넣으면 현재 잘 쓰는 회계/명세표 흐름을 망가뜨릴 위험이 있다.

### 4.3 비용 제약

- 무료 또는 아주 저비용 구조가 우선이다.
- 유료 SaaS 추가보다는 기존 시스템 + 내부 자동화가 우선이다.

## 5. 제품 원칙

1. `내부 기준 우선`
   - 고객, 주문, 거래, 메시지 기준은 내부 DB가 가진다.
2. `운영 실용성 우선`
   - 완전한 실시간보다, 믿을 수 있는 일일/반일 단위 통합이 우선이다.
3. `원본 보존`
   - 수집 파일과 원본 응답은 그대로 저장한다.
4. `자동 병합 최소화`
   - 고객 병합은 보수적으로 하고, 애매하면 검토 큐로 보낸다.
5. `기존 운영 허브 존중`
   - 사방넷은 주문/송장/재고 운영 허브로 유지한다.
6. `전화주문 내재화`
   - 전화주문은 내부 시스템의 일급 흐름으로 편입한다.
7. `안정 앱 보호`
   - `offline-crm-v2`는 우선 보호 대상이며, 허브와의 연결은 읽기/복제 중심으로 시작한다.

## 6. 전략적 결정 초안

### 6.0 시스템 경계

- 기준 구조: `Customer OS Core + CRM v2 연동`
- 추천 결정: `offline-crm-v2`를 허브의 하위 실행 앱으로 두고, 통합 기준 데이터는 별도 허브에서 관리한다.

### 6.1 기준 고객

- 기준 시스템: `internal customer master`
- 추천 결정: 채널톡/해피톡/메이크샵 고객 테이블이 아니라 내부 고객 마스터를 기준으로 삼는다.

### 6.2 기준 주문

- 기준 시스템: `unified order master`
- 추천 결정: `사방넷 우선 + 메이크샵 직접 보강 + 전화주문 직접 생성`

### 6.3 기준 거래/수금

- 기준 시스템: `offline-crm-v2`
- 추천 결정: 입금, 미수, 예치금, 환불대기는 CRM v2를 기준으로 삼는다.

### 6.4 기준 저장소

- 추천 결정: `Postgres`를 기준 저장소로 두고, `NocoDB`는 운영 테이블 관리와 빠른 내부 뷰에 사용한다.

### 6.5 수집 주기

- 추천 결정: `매일 1회 전체 수집`을 기본으로 하고 필요 시 `낮 1회 추가 수집`을 붙인다.

## 7. 목표 아키텍처

### 7.1 전체 구조

```text
메이크샵 / 스마트스토어 / 쿠팡 / 11번가 / 사방넷 / 상담툴 / CRM v2
                         ↓
                수집기 (n8n + 브라우저 자동화 + 파일 파서)
                         ↓
             Customer OS Core (Raw + Unified DB + Rules)
                         ↓
        Customer OS Portal / 대시보드 / 메시지 자동화 / 리포트
                         ↓
              CRM v2 / 운영 Deep Link / 후속 실행 앱
```

### 7.2 역할 분리

- `Customer OS Core`
  - 통합 기준 데이터 저장
  - 고객/주문/결제/출고/상담 이벤트 정규화
  - 분석/자동화/통합 타임라인 제공
- `사방넷`
  - 주문수집, 송장전송, 재고운영 실무 허브
- `메이크샵`
  - 자사몰 주문/회원/상품 기준 보강 원천
- `offline-crm-v2`
  - 전화주문, 명세표, 입금, 미수, 고객 실무 실행 앱
- `n8n`
  - 수집, 정규화, 적재, 메시지 자동화
- `메시징 공급자`
  - 알림톡/SMS/RCS 발송 레일

### 7.3 CRM v2와 허브의 책임 분리

| 영역 | Customer OS Core | CRM v2 |
|------|------------------|--------|
| 고객 기준 ID | 소유 | 참조 |
| 통합 주문 | 소유 | 조회/실행 링크 |
| 명세표/입금/미수 | 요약 복제 | 소유 |
| 전화주문 | 타임라인/통계 수집 | 생성/수정/처리 |
| 상담 이벤트 | 소유 | 선택 표시 |
| 분석/대시보드 | 소유 | 일부 표시 가능 |
| 자동화 트리거 | 소유 | 실행 결과 반영 |

## 8. 통합 데이터 모델 v1

### 8.1 핵심 테이블

| 테이블 | 목적 | 비고 |
|------|------|------|
| `customers` | 내부 기준 고객 마스터 | 한 고객당 1행 |
| `customer_identities` | 외부 식별자 연결 | 전화, 회원ID, 채널 식별자, 입금자명 별칭 |
| `customer_consents` | 수신동의/광고동의 | 알림 규정 대응 |
| `orders_raw` | 원본 주문 파일/JSON 보관 | 사방넷 export, 메이크샵 API 응답 등 |
| `orders_unified` | 내부 기준 주문 마스터 | 채널 무관 공통 스키마 |
| `order_items_unified` | 주문 상품 라인 | SKU/옵션/수량/금액 |
| `payments` | 결제/입금 이벤트 | 실결제, 실수금, 예치금 포함 |
| `receivables` | 미수/회수 상태 | CRM v2 연동 |
| `shipments` | 출고/송장 상태 | 사방넷 기준 |
| `conversations` | 상담 이벤트 | 상담툴 이벤트 적재 |
| `phone_orders` | 전화주문 메타 | CRM v2에서 직접 생성 |
| `customer_timeline` | 고객 사건 로그 | 주문/상담/입금/발송 통합 뷰 |

### 8.2 기준 키

| 키 | 의미 |
|----|------|
| `customer_id` | 내부 기준 고객 ID |
| `identity_id` | 외부 식별자 ID |
| `unified_order_id` | 내부 기준 주문 ID |
| `channel_order_id` | 채널 원본 주문번호 |
| `payment_event_id` | 입금/결제 이벤트 ID |
| `shipment_event_id` | 출고/배송 이벤트 ID |

### 8.3 고객 매칭 우선순위

1. 휴대폰 번호
2. 전화번호
3. 메이크샵 회원 ID
4. 수취인 이름 + 전화번호
5. 입금자명 별칭
6. 주소 일부 + 이름

애매한 경우 자동 병합하지 않고 `review queue`로 보낸다.

## 9. 채널별 수집 전략

### 9.1 메이크샵

- 1순위: Open API
- 2순위: 관리자 엑셀 export
- 용도: 자사몰 주문/회원/상품/고객 보강

### 9.2 사방넷

- 1순위: 주문 export 자동 다운로드
- 용도: 통합 주문 운영 기준
- 원칙: API 유료 전에는 브라우저 자동화로 export 수집

### 9.3 쿠팡

- 원칙: 직접 API 연동을 새로 붙이지 않는다.
- 사방넷을 통해 수집된 주문을 기준으로 본다.

### 9.4 스마트스토어 / 11번가

- 초기: 사방넷 수집 기준
- 보강 필요 시: 채널 export 또는 직접 API 추가 검토

### 9.5 상담툴

- 대화 UI는 외부 툴을 사용하되, 이벤트/메타데이터는 내부로 적재한다.

### 9.6 전화주문

- CRM v2에서 직접 생성한다.
- 외부 원천이 아니라 내부 원천이다.

## 10. 기능 요구사항

### 10.1 Customer 360

직원이 고객 1명을 열었을 때 아래를 봐야 한다. 이 화면은 우선 `Customer OS Core`에 두고, 안정화 후 CRM v2에 필요한 일부만 연결한다.

- 기본 프로필
- 최근 주문
- 최근 상담
- 미수/예치금/환불대기
- 최근 출고
- 재구매 가능성
- 수신동의 상태

### 10.2 Unified Order Hub

직원이 채널별 관리자 화면을 돌지 않고 아래를 볼 수 있어야 한다.

- 주문 채널
- 결제 상태
- 송장 상태
- 취소/반품/교환 상태
- 채널별/기간별 주문 건수
- 전화주문 포함 통합 주문 흐름

### 10.3 Phone Order Desk

전화주문 흐름은 아래를 지원해야 한다.

- 발신번호 기반 고객조회
- 최근 주문 복사
- 자주 쓰는 주소 선택
- 단가/메모/결제안내 반영
- 주문 생성 후 입금 대기 또는 즉시 정산 연결

초기 구현 원칙:

- 전화주문 생성/처리는 `CRM v2`
- 전화주문 통합 조회/성과 분석은 `Customer OS Core`

### 10.4 Payment / Receivables Console

- 자동입금 후보 매칭
- 수동 검토 큐
- 예치금/초과입금 처리
- 미수 추적
- 정산 로그 감사

### 10.5 Analytics Console

최소 아래 지표를 통합 집계해야 한다.

- 주문총액
- 실결제액
- 순매출
- 실수금액
- 채널별 매출
- 고객별 누적 매출
- 재구매율

## 13. 다른 계획과의 관계

이 프로그램은 독립 프로젝트가 아니라 아래 계획들의 기반이다.

- CRM v2 고도화
- 전화주문 간소화
- 상담 허브 전환 또는 유지 판단
- 사방넷 재고/물류 운영 정착
- 원가마진분석과 실제 매출 연결
- 고객 리텐션 자동화

즉 `고객운영 OS`는 별도 옵션이 아니라, 앞으로의 고객/매출/운영 자동화 계획을 묶는 공통 기반이며 `CRM v2`를 대체하기보다 상위에서 연결하는 프로그램이다.
