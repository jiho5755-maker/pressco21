## 1. 로드맵 목적

이 로드맵은 고객운영 OS를 `큰 비전`이 아니라 `실행 가능한 단계`로 쪼개기 위한 문서다.

핵심은 세 가지다.

1. 한 번에 다 만들지 않는다.
2. 데이터 기준을 먼저 세운다.
3. 기존 운영을 멈추지 않고 옆에서 붙인다.

## 2. 실행 원칙

1. `사방넷을 버리지 않는다`
   - 주문/송장/재고 운영은 기존 허브를 유지한다.
2. `CRM v2를 버리지 않는다`
   - 전화주문, 명세표, 수금 운영의 핵심 콘솔로 키운다.
3. `원본을 먼저 저장한다`
   - export 자동화는 항상 원본 파일 보관을 동반한다.
4. `의사결정 게이트를 둔다`
   - 사방넷 export 충분성 검증 전에는 대규모 구현을 시작하지 않는다.
5. `분석보다 운영을 먼저 살린다`
   - 전화주문, 수금, 통합 고객 조회가 우선이다.
6. `CRM v2를 먼저 보호한다`
   - 기존 명세표/입금/미수 흐름은 바로 뜯지 않고, 허브와 읽기 중심으로 연결한다.

## 3. 워크스트림

### W1. 데이터 통합

- 통합 고객/주문/결제/출고 스키마
- raw 저장소
- 정규화 파이프라인

### W2. 수집 자동화

- 사방넷 export 자동 다운로드
- 메이크샵 API/export 수집
- 수동 업로드 백업 경로

### W3. CRM v2 확장

- CRM v2 연동 브리지
- Phone Order Desk 연동
- 고객/주문 deep link
- 요약 카드 반영

### W4. Customer OS Portal

- Customer 360
- 통합 주문 조회
- 통합 타임라인
- 운영 대시보드

### W5. 수금/정산

- 자동입금 후보
- 검토 큐
- 미수/예치금/환불대기

### W6. 메시징/운영 자동화

- 결제안내
- 출고안내
- 리뷰 요청
- 리텐션

### W7. 분석/리포트

- 총매출/실결제/순매출/실수금
- 채널별 성과
- 고객별 매출/LTV

## 4. 단계별 로드맵

### Phase 0. 기준선 고정

기간: 1주

#### 목표

- 프로그램 범위와 기준 지표를 고정한다.
- 데이터 원천과 우선순위를 확정한다.

#### 작업

- 고객 기준/주문 기준/매출 기준 확정
- 사방넷 export 검증 항목 정의
- 메이크샵 보강 항목 정의
- 통합 테이블 1차 설계
- 운영/보관/백업 경로 정의

#### 산출물

- PRD 확정본
- 로드맵 확정본
- 데이터 모델 v1 초안
- 검증 체크리스트

#### 게이트

- `고객 기준`, `주문 기준`, `매출 기준` 3개가 합의되어야 다음 단계로 간다.

### Phase 1. 수집 PoC + 섀도우 런

기간: 2주

#### 목표

- 사방넷 export 자동화가 실제로 가능한지 검증한다.
- 사방넷 기준 데이터가 통합 운영에 충분한지 확인한다.

#### 작업

- Playwright 기반 사방넷 export 자동 다운로드 구현
- 파일 archive 저장
- Oracle 적재 파이프라인 초안 연결
- 메이크샵 같은 날짜 데이터와 비교
- 누락 필드/오차율 기록

#### 산출물

- 사방넷 export 자동화 스크립트
- raw 파일 저장소
- 섀도우 비교 리포트
- 부족 필드 목록

#### 완료 기준

- 최소 7일 이상 자동 다운로드 성공
- 주문 건수/주요 필드 비교표 확보
- `사방넷으로 충분한 영역`과 `직접 보강이 필요한 영역`이 구분됨

### Phase 2. 통합 데이터층 구축

기간: 2주

#### 목표

- 내부 기준 고객/주문/결제/출고 테이블을 만든다.

#### 작업

- `customers`, `customer_identities`, `orders_raw`, `orders_unified` 생성
- 사방넷 export 파서 구현
- 메이크샵 보강 수집 연결
- 고객 매칭 규칙 엔진 구현
- review queue 구현
- CRM v2 요약 복제 경로 정의

#### 산출물

- 통합 DB 스키마 v1
- 정규화 파이프라인
- 고객 매칭 로그
- 관리자 검토 큐

#### 의사결정 필요

- Postgres 단독 vs Postgres + NocoDB 관리뷰
- 고객 자동 병합 임계값

### Phase 3. Customer OS Portal 1차

기간: 2주

#### 목표

- 기존 CRM v2를 건드리지 않고도 통합 고객/주문을 읽을 수 있는 새 허브 화면을 만든다.

#### 작업

- Customer 360 화면
- 통합 주문 조회 화면
- 주문/상담/입금/송장 타임라인
- 주요 KPI 카드
- CRM v2 deep link 연결

#### 산출물

- Customer OS Portal 읽기 화면
- 통합 주문 리스트
- CRM v2 연결 포인트

#### 완료 기준

- 직원이 CRM v2를 버리지 않고도, 허브에서 고객 1명 기준 기본 판단 가능

### Phase 4. CRM v2 브리지 + Phone Order Desk 연결

기간: 2주

#### 목표

- CRM v2의 안정성을 유지하면서 허브와 실무 실행을 연결한다.

#### 작업

- Customer OS -> CRM v2 deep link
- CRM v2 고객/거래 요약 카드 수신
- 발신번호 기반 고객 조회
- 최근 주문 복사
- 빠른 전화주문 생성
- 결제안내 발송
- 입금 대기/반영 연동

#### 산출물

- CRM v2 연동 브리지
- Phone Order Desk
- 전화주문 테이블
- 전화주문 -> CRM 수금 흐름

#### 완료 기준

- 전화주문이 수기 메모가 아니라 주문 이벤트로 기록됨

### Phase 5. 분석/자동화 1차

기간: 2주

#### 목표

- 허브 기준 데이터로 매출/고객/운영 자동화의 첫 실효성을 만든다.

#### 작업

- 총매출/실결제/순매출/실수금 대시보드
- 채널별 매출 리포트
- 리뷰 요청/결제안내/미수 리마인드 자동화
- 휴면고객 세그먼트 1차

#### 산출물

- 운영 대시보드
- 자동화 시나리오 3~5개
- KPI 리포트

### Phase 6. 재고/물류/리텐션 확장

기간: 4주+

#### 목표

- 사방넷 재고 운영과 고객 운영을 더 깊게 연결한다.

#### 작업

- 재고 스냅샷 연동
- 품절/입고예정 알림
- 반품/교환 상태 연결
- 재구매/리텐션 자동화 고도화

## 8. 리스크 체크포인트

1. 사방넷 export 컬럼이 충분하지 않으면 메이크샵 직접 보강 범위를 늘린다.
2. 브라우저 자동화가 불안정하면 수동 업로드 Inbox를 유지한다.
3. 고객 병합 오판이 늘면 자동 병합 임계값을 낮춘다.
4. 실시간 요구가 커지면 사방넷 유료 기능 또는 직접 API를 재검토한다.
