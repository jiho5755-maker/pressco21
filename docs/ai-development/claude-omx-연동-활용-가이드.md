# Claude Code + OMX 연동 활용 가이드

> 대상: 비전공자 대표/운영자  
> 목적: Claude Code와 Codex/OMX를 **서로 다른 AI**가 아니라 **같은 회사 직원이 다른 작업실에서 일하는 구조**로 이해하고, 실제 업무에서 가장 효율적으로 쓰기 위한 운영 가이드

---

## 1. 이 작업이 실제로 바꾸는 것

이 연동 작업의 핵심은 단순히 설정 파일 몇 개를 바꾸는 것이 아닙니다.

우리가 만드는 구조는 다음과 같습니다.

- **Claude Code = 회의실 / 비서실 / frontdoor**
  - 방향 정리
  - 설명
  - 전략 논의
  - 대표 친화적 대화
- **Codex/OMX = 실행실 / 공장 / 검증실**
  - 코드 수정
  - 테스트
  - 병렬 실행
  - 장시간 작업
  - 검증 루프

즉, **같은 직원이 다른 작업실에서 일하는 구조**로 만드는 것이 목적입니다.

한 줄로 요약하면:

> 같은 직원, 다른 런타임. 같은 기억, 다른 실행 근육.

---

## 2. 기존보다 좋아진 점

## 2.1 같은 직원처럼 보이기 시작함

예전에는 Claude 쪽 에이전트와 OMX 쪽 role 이름이 따로 노는 느낌이 있었습니다.

지금은 다음 shared-kernel 문서를 중심 기준으로 삼습니다.

- `docs/ai-native-upgrade/shared-agent-kernel/agents.v1.yaml`
- `docs/ai-native-upgrade/shared-agent-kernel/founder-facing-roster-v1.md`

즉, 대표에게 보이는 직원 정체성은 하나로 맞추고,
Claude와 OMX는 그 정체성을 각자 다른 방식으로 수행합니다.

### 대표 체감 효과
- “이건 Claude용 인격, 저건 OMX용 인격”처럼 느껴지는 현상이 줄어듭니다.
- 도구보다 **누구와 일하는지** 중심으로 이해할 수 있습니다.

---

## 2.2 세션 연속성(continuity)이 좋아짐

지향 구조상 세션 시작/종료 시 다음 정보가 이어지도록 설계합니다.

- latest handoff
- recent learnings
- open risks
- next step
- relevant playbook

즉, Claude에서 논의한 내용이 OMX 실행으로 넘어가고,
OMX 실행 결과가 다시 Claude 요약/의사결정으로 돌아올 수 있는 흐름을 만듭니다.

### 대표 체감 효과
- 매 세션마다 처음부터 다시 설명하는 비용이 줄어듭니다.
- “어제 얘기한 걸 오늘 다른 도구가 이어받는 느낌”이 생깁니다.

---

## 2.3 도구마다 잘하는 일만 맡기게 됨

이 연동 작업의 중요한 성과는 “역할 분리”가 명확해진 것입니다.

### Claude가 잘하는 것
- 전략 정리
- 사람 친화적 설명
- 회의
- 맥락 구조화
- 대표용 요약

### OMX가 잘하는 것
- 실제 구현
- 테스트
- 병렬 실행
- 검증
- repo-grounded 작업

### 대표 체감 효과
- 한 도구에 억지로 모든 걸 시키지 않게 됩니다.
- “논의는 Claude, 실행은 OMX” 같은 자연스러운 업무 분업이 가능해집니다.

---

## 2.4 문서/조직도 drift가 줄어듦

예전에는 문서마다 에이전트 설명과 조직 구조가 조금씩 어긋날 가능성이 있었습니다.

지금은 shared-agent-kernel을 기준 소스로 두고:

- `CLAUDE.md`
- `HARNESS.md`
- OMX prompt / founder-facing output spec

이 같은 기준을 참조하도록 정렬하는 방향입니다.

### 대표 체감 효과
- 문서마다 말이 다르다는 느낌이 줄어듭니다.
- AI 팀이 “정리된 회사”처럼 보이기 시작합니다.

---

## 2.5 handoff와 founder-facing 보고 품질이 좋아짐

연동 작업은 단순 대화 저장이 아니라 **인수인계 가능한 형식**을 만드는 데 초점을 둡니다.

현재 준비된 방향:
- Claude → OMX handoff 파싱
- latest.md bridge
- founder-facing output wrapper spec
- cross-runtime smoke test

즉, “잘 대화했다”가 아니라 “다음 세션/다음 작업실이 바로 이어받을 수 있다”가 목표입니다.

### 대표 체감 효과
- 결과 보고가 더 읽기 쉬워집니다.
- 세션이 바뀌어도 맥락 손실이 줄어듭니다.

---

## 3. 성능이 좋아졌다는 뜻

여기서 성능은 단순히 모델 답변 속도가 아니라 **대표 입장에서의 운영 생산성**입니다.

### 실제로 좋아진 성능
- 설명 반복 비용 감소
- 도구 전환 비용 감소
- 판단 일관성 증가
- 장기 학습 가치 증가

즉,

> 한 번 좋은 판단과 handoff가 쌓이면 다음번에 더 잘할 가능성이 높아지는 구조

가 생긴다는 뜻입니다.

---

## 4. 아직 남아 있는 일

문서/프롬프트/브리지/스모크 테스트 준비는 많이 진행됐지만,
아직 남은 핵심은 다음입니다.

- **실제 OMX runtime output chain에 founder-facing wrapper를 live로 연결하는 일**

쉽게 말하면:
- 설계도와 기준 문서는 상당 부분 준비됨
- 실제 운영 흐름에 자동 래핑을 꽂는 마무리 배선 작업이 남아 있음

즉, 현재 상태는:

> “구조 설계와 준비는 많이 끝났고, live 연결 마무리가 남아 있는 상태”

로 이해하면 됩니다.

---

## 5. 대표가 가장 잘 쓰는 운영 방식

## 5.1 도구가 아니라 작업실로 생각하기

앞으로는 이렇게 생각하는 것이 가장 쉽습니다.

- **Claude = 회의실**
- **OMX = 실행실**

### 추천 판단 기준
- 방향/구조/전략/설명 → Claude
- 수정/테스트/병렬작업/검증 → OMX

---

## 5.2 도구를 바꿀 때는 항상 handoff를 의식하기

도구를 바꾸는 순간을 “채팅방 이동”이 아니라 **업무 인수인계**로 생각하면 좋습니다.

### 추천 표현
- “지금까지 내용을 실행용 handoff로 정리해줘.”
- “대표 보고용 handoff로 다시 써줘.”
- “다음 세션이 바로 이어받을 수 있게 요약해줘.”

---

## 5.3 좋은 판단은 저장 가능한 자산으로 만들기

반복해서 쓸 가치가 있는 내용은 이렇게 요청합니다.

### 추천 표현
- “이건 playbook으로 남길 만한지 봐줘.”
- “이건 founder preference로 승격할 가치가 있는지 봐줘.”
- “반복되는 패턴이면 reusable handoff로 정리해줘.”

중요한 건 모든 대화를 저장하는 게 아니라,
**다음 행동을 더 잘하게 만드는 정보만 남기는 것**입니다.

---

## 5.4 세션 끝날 때는 항상 다음 사람이 바로 이어받게 만들기

세션 종료 전에는 아래 형식으로 정리시키면 좋습니다.

### 추천 표현
- “what changed / what worked / what failed / next step 으로 정리해줘.”
- “다음 세션이 바로 이어받게 마감 정리해줘.”
- “리스크와 다음 결정 포인트까지 같이 적어줘.”

---

## 6. 비전공자가 꼭 알아두면 좋은 개념

## 6.1 handoff

handoff는 쉽게 말해:

> 다음 사람이 바로 이해하고 이어서 일할 수 있게 남기는 업무 메모

입니다.

---

## 6.2 continuity

continuity는:

> 세션이 바뀌어도 맥락이 이어지는 상태

입니다.

즉, “어제 회의실에서 정리한 내용이 오늘 실행실에서도 살아있는 상태”를 목표로 합니다.

---

## 6.3 memory

memory는 단순 기록 저장소가 아닙니다.

이 구조에서 memory는:

> 다음 행동을 더 잘하게 만드는 판단 자산

입니다.

예:
- 대표가 선호하는 보고 방식
- 브랜드/사업 판단 기준
- 반복되는 실수
- 재사용 가능한 playbook

---

## 6.4 worktree

worktree는 기술적으로는 Git 기능이지만,
비전공자 관점에서는 그냥 이렇게 이해하면 됩니다.

> 프로젝트별로 따로 쓰는 작업방

즉, 여러 업무를 한 책상에 섞지 않고,
각 프로젝트마다 별도 작업 테이블을 두는 구조입니다.

---

## 6.5 verification

verification은:

> “다 했어요”가 아니라 “무엇으로 확인했는가”

를 보는 습관입니다.

대표가 자주 요청하면 좋은 표현:
- “검증 결과도 같이 보여줘.”
- “확인한 것과 아직 확인 못 한 것을 나눠줘.”
- “증거 중심으로 요약해줘.”

---

## 7. 바로 써먹는 운영 패턴

## 패턴 A — 방향은 Claude에서

예시:

> “이번 작업 목적과 내가 결정해야 할 포인트를 먼저 정리해줘.”

---

## 패턴 B — 실행은 OMX에서

예시:

> “이제 이 내용을 실행용 handoff로 정리해줘.”

그 다음 OMX에서:

> “이 handoff 기준으로 실제 수정/검증까지 진행해줘.”

---

## 패턴 C — 마지막 요약은 다시 대표 보고용으로

예시:

> “대표 보고용으로 바꿔줘. 변경점 / 검증 / 리스크 / 다음 결정만 알려줘.”

---

## 8. 새로운 세션에서 바로 해볼 추천 루틴

## 8.1 OMX 실행 점검

```bash
p21-omx doctor
```

이상 없으면:

```bash
p21-omx
```

또는 강하게 실행하려면:

```bash
p21-omx turbo
```

---

## 8.2 Claude ↔ OMX 연동을 시험하는 가장 쉬운 흐름

### 1단계 — Claude에서 물어보기

예시:

> “이번 작업을 Claude 회의실 기준으로 먼저 구조화해줘.  
> 목표 / 제한 / 완료기준 / OMX로 넘길 handoff 초안까지 같이 줘.”

### 2단계 — OMX에서 실행시키기

예시:

> “이 handoff 기준으로 실제 실행실 작업처럼 진행해줘.  
> 끝나면 변경점 / 검증 / 리스크 / 다음 단계만 알려줘.”

### 3단계 — 다시 대표 보고용 요약 받기

예시:

> “이 결과를 창업자용 보고로 다시 정리해줘.”

---

## 9. 대표용 최소 암기 버전

이것만 기억해도 충분합니다.

### 핵심 개념
- 같은 직원, 다른 작업실
- 도구 전환 = handoff
- memory = 행동 개선 자산

### 핵심 명령
```bash
p21-omx
p21-omx turbo
p21-omx doctor
p21-omx repair
```

### 핵심 요청 문장
- “실행용 handoff로 정리해줘.”
- “대표 보고용으로 다시 써줘.”
- “다음 세션이 바로 이어받게 마감 정리해줘.”
- “검증 결과와 남은 리스크를 같이 알려줘.”
