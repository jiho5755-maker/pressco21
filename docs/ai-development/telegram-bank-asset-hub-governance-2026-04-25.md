# 원장님 은행·자산 관제방 설계안

- 작성일: 2026-04-25
- 작업 브랜치: `work/workspace/telegram-bank-room-governance`
- 상태: 실행 준비안 / 라이브 Telegram·n8n 변경 없음
- 목적: 기존 직원 포함 `은행 알림` 방은 유지하고, 원장님+장지호 전용의 계좌별 토픽 기반 자산 관제방을 새로 설계한다.

## 1. 결론

새 방을 만드는 방향이 맞다.

기존 `은행 알림` 방은 직원과 함께 보는 **입금·미수관리 실무방**으로 유지한다. 새 방은 원장님과 장지호만 보는 **은행·자산 관제방**으로 분리한다. 이 분리는 단순히 방을 하나 더 만드는 것이 아니라, `CRM 처리`와 `경영/자산 분석`의 책임을 분리하는 안전장치다.

권장 최종 구조:

| 구분 | 기존 방 | 신규 방 |
|---|---|---|
| 목적 | 입금 확인, CRM 미수/반영 검토 | 원장님 전체 계좌의 입출금·잔액·자산 흐름 관찰 |
| 멤버 | 원장님 + 장지호 + 직원 + Bank bot | 원장님 + 장지호 + 전용 Bank/Finance bot |
| 데이터 범위 | CRM 관련 입금/출금 알림 중심 | 원장님이 쓰는 모든 주요 계좌 |
| 공개 수준 | 직원 실무 공유 가능 범위 | 매우 민감한 금융/자산 정보 |
| 자동화 책임 | 미수관리/입금반영 판단 | 자산 모니터링/리포팅/이상거래 검토 |
| 변경 방침 | 변경 없음 | 신규 구축 |

## 2. 기존 방을 그대로 두는 이유

현재 문서상 기존 `은행 알림` 그룹은 `원장님 + 장지호 + 이재혁 + Bank_bot` 구성이며, 농협 입금 실시간 및 CRM 검토 용도로 정의되어 있다.

- `docs/ai-native-upgrade/P1-bot-wf-matrix.md`
- `docs/ai-native-upgrade/P1-telegram-bot-reorganization.md`
- `docs/offline-crm-bank-automation-ops-guide-2026-03-15.md`

특히 기존 CRM 은행 자동화 운영 규칙은 다음 원칙을 가진다.

- 입금은 CRM과 매칭해 `자동반영 / 검토 / 미매칭`으로 분기한다.
- 출금은 CRM 자동반영 대상이 아니다.
- 같은 입금 건에 대해 원본 은행 알림과 CRM 처리 결과를 따로 두 번 보내지 않는다.

따라서 기존 방에 원장님 전체 계좌 알림까지 넣으면 다음 문제가 생긴다.

1. 직원에게 원장님 전체 계좌/자산 정보가 노출된다.
2. 미수관리 알림과 개인/자산 흐름이 섞여 CRM 판단이 흐려진다.
3. 향후 자동 파싱이 어느 메시지를 CRM 대상이라고 봐야 하는지 불명확해진다.
4. 알림 과부하로 실무자가 중요한 미수/입금 알림을 놓칠 수 있다.

## 3. 신규 방 이름과 역할

권장 이름:

- `P21 원장님 은행·자산 관제실`
- 또는 `원장님 은행·자산 현황`

역할:

1. 원장님 관련 전체 주요 계좌의 입출금 원문을 계좌별 토픽에 모은다.
2. Telegram의 `ALL` 뷰에서는 전체 계좌 흐름을 시간순으로 볼 수 있게 한다.
3. 별도 `00_전체요약` 토픽에는 원문 중복이 아니라 일/주/月 요약과 이상징후만 올린다.
4. 장기적으로 NocoDB/Sheet/DB 원장에 정규화해 자산 분석, 현금흐름 리포트, 내부이체 제거, 월간 자산 추이를 만든다.

핵심 원칙:

> Telegram은 관제 화면이고, 분석의 정본은 정규화된 거래 원장이다.

## 4. 권장 토픽 구조

사용자의 아이디어처럼 **계좌별 토픽**을 기본 단위로 둔다. 카테고리별 토픽보다 계좌별 토픽이 원문 추적과 은행별 장애 대응에 유리하다.

### 4.1 기본 토픽

| 토픽 | 용도 | 원문/요약 |
|---|---|---|
| `00_전체요약` | 일일/주간/월간 전체 요약, 자산 스냅샷, 큰 변동 알림 | 요약만 |
| `01_NH_1234_사업운영` | 농협 사업운영 계좌 | 원문 + 정규화 요약 |
| `02_NH_5678_매출입금` | 농협/기타 매출 입금 계좌 | 원문 + 정규화 요약 |
| `03_KB_1234_개인` | 국민 개인 또는 별도 목적 계좌 | 원문 + 정규화 요약 |
| `04_SH_1234_예적금` | 예적금/자산 이동 계좌 | 원문 + 정규화 요약 |
| `05_CARD_카드대금` | 카드 결제/자동이체/대출/이자 | 원문 + 정규화 요약 |
| `90_내부이체후보` | 계좌 간 이동으로 보이는 거래 | 검토용 |
| `99_미분류_검토` | 파싱 실패, 계좌 미등록, 고액/이상거래 | 검토용 |

`1234`는 계좌 끝 4자리만 쓰고, 전체 계좌번호는 노출하지 않는다.

### 4.2 ALL 뷰 운영 원칙

Telegram의 `ALL`은 모든 토픽 메시지가 합쳐져 보이는 사용자 화면이다. 이를 활용하면 전체 입출금 타임라인을 자연스럽게 볼 수 있다.

다만 같은 거래를 `계좌 토픽`과 `00_전체요약`에 모두 원문으로 중복 발송하면 알림이 과도해진다. 따라서 원칙은 다음과 같다.

- 계좌별 토픽: 실시간 원문/상세
- Telegram ALL: 계좌별 토픽 메시지가 자연스럽게 합쳐지는 전체 타임라인
- `00_전체요약`: 원문 재전송 금지, 정기 요약/고액 변동/이상징후만 발송

## 6. 데이터 원장 최소 필드

Telegram 메시지만으로는 장기 분석이 어렵다. 처음부터 최소 원장 필드를 정해두는 것이 좋다.

| 필드 | 설명 |
|---|---|
| `transactionId` | 중복 방지용 고유 ID |
| `source` | Gmail, 은행메일, 수동입력, API 등 |
| `occurredAt` | 실제 거래일시 |
| `collectedAt` | 시스템 수집일시 |
| `bankName` | 은행명 |
| `accountAlias` | `NH 1234 사업운영` 같은 별칭 |
| `accountMasked` | 마스킹 계좌번호 |
| `direction` | `deposit` / `withdrawal` |
| `amount` | 금액 |
| `balanceAfter` | 거래 후 잔액 |
| `counterparty` | 입금자/출금처 |
| `memo` | 적요/메모 |
| `topicKey` | 라우팅 토픽 키 |
| `messageThreadId` | Telegram 토픽 ID |
| `category` | 사업수입, 운영비, 개인이체, 세금, 카드, 대출 등 |
| `classificationStatus` | 자동분류 / 검토필요 / 수동확정 |
| `isInternalTransfer` | 내부 계좌 간 이체 여부 |
| `crmRelated` | CRM 관련 후보 여부 |
| `crmMatchStatus` | 자동반영 / 검토 / 미매칭 / 해당없음 |

## 7. CRM 미수관리와의 경계

새 방은 기존 CRM 자동반영의 정본이 아니다. 초기에는 다음 경계를 유지한다.

### 기존 `은행 알림` 방

- CRM 입금반영, 미수관리, 직원 공유가 목적
- 현재 농협 입금/출금 및 CRM 처리 결과를 계속 담당
- 기존 n8n 워크플로우와 방 설정 변경 없음

### 신규 `원장님 은행·자산 관제실`

- 전체 자산 흐름 관찰과 리포팅이 목적
- CRM 관련 거래 여부는 표시할 수 있지만, CRM 상태를 직접 바꾸는 방이 아니다
- CRM과 연결하려면 별도 검증 단계를 거친 뒤 기존 CRM 엔진으로 전달한다

같은 거래가 두 방에 보일 수는 있다. 하지만 의미는 다르다.

- 기존 방: “이 입금이 고객 미수에 반영됐는가?”
- 신규 방: “전체 자금 흐름에서 이 거래가 어떤 의미인가?”

## 8. 보안·권한 가드레일

금융 정보이므로 기본값을 보수적으로 둔다.

1. 사람 멤버는 원장님과 장지호만 둔다.
2. 직원, 외부 협력자, 일반 운영 봇은 초대하지 않는다.
3. OpenClaw/AI 명령 봇은 절대 초대하지 않는다.
4. 은행/재무 전용 봇만 초대하고, 가능한 한 전송 권한만 사용한다.
5. 전체 계좌번호는 메시지에 쓰지 않는다. 은행명 + 끝 4자리 + 용도 별칭만 쓴다.
6. 토큰, chat ID, thread ID는 문서에 원문으로 쓰지 않고 운영 환경변수/비밀 저장소에서 관리한다.
7. 토픽 ID 매핑은 명명된 환경변수로 관리한다.
8. 파싱 실패, 계좌 미등록, 고액 거래, 중복 의심은 `99_미분류_검토`로 보낸다.
9. Telegram `ALL`은 보기용이다. 회계/자산 분석의 정본은 원장 테이블이어야 한다.
10. 새로운 방의 메시지를 기존 CRM 방으로 자동 포워딩하지 않는다.

권장 환경변수 이름 예시:

```text
P21_TG_OWNER_FINANCE_CHAT_ID
P21_TG_OWNER_FINANCE_THREAD_SUMMARY
P21_TG_OWNER_FINANCE_THREAD_REVIEW
P21_TG_OWNER_FINANCE_THREAD_NH_1234_OPERATING
P21_TG_OWNER_FINANCE_THREAD_KB_1234_PERSONAL
```

## 9. 자동화 아키텍처 초안

```text
은행 알림 원천
  ├─ Gmail/보안메일
  ├─ 은행 앱/메일/문자 연동
  └─ 수동 입력 또는 추후 API
        ↓
수집/파서 n8n
        ↓
거래 정규화 + 중복 제거
        ↓
은행 거래 원장 저장
        ↓
계좌 별칭/토픽 라우팅 레지스트리 조회
        ↓
Telegram 신규방 계좌별 Topic 발송
        ↓
검토 필요 건은 99_미분류_검토
        ↓
일/주/月 요약은 00_전체요약
```

중요한 점:

- 현재는 기존 CRM 방을 건드리지 않는다.
- 새 방 파일럿은 별도 n8n 워크플로우 또는 기존 파서의 “읽기 복제” 형태로 시작한다.
- CRM 자동반영과 자산 원장은 transactionId로 연결하되, 첫 단계에서는 느슨하게 연결한다.

## 10. 단계별 도입안

### Phase 0 — 설계/권한 정리

- 이 문서 확정
- 기존 `은행 알림` 방 변경 금지 재확인
- 원장님과 장지호 전용 신규 방 생성 준비
- 수집할 계좌 목록/은행/용도/알림 가능 여부 정리

### Phase 1 — Telegram 방/토픽 생성

- 새 슈퍼그룹 생성
- 토픽 모드 ON
- 계좌별 토픽 생성
- `00_전체요약`, `99_미분류_검토` 생성
- 각 토픽의 `message_thread_id` 확보

### Phase 2 — 파일럿

- 대표 계좌 1~2개만 연결
- 기존 CRM 방은 무변경
- 새 방에 원문/요약이 올바른 토픽으로 가는지 확인
- 1주간 중복, 누락, 잔액 불일치, 알림 과다 여부 검증

### Phase 3 — 원장화

- 거래 원장 테이블 생성
- `transactionId` 중복 제거
- 계좌별 잔액/순증감/내부이체 후보 계산
- 일일 요약 자동 발송

### Phase 4 — 전체 계좌 확대

- 원장님 주요 계좌 전체 연결
- 카드/대출/예적금/세금 자동이체 분류 추가
- 주간/월간 자산 리포트 자동 생성

### Phase 5 — CRM/경영 분석 연결

- CRM 미수 회수액과 실제 입금 흐름 비교
- 사업/개인/내부이체/고정비 분리
- 월간 현금흐름, 잔액 추이, 고정비 변화 리포트

## 11. 우선 결정할 사항

### 11.1 이번 단계에서 확정

1. 새 방 이름은 `P21 원장님 은행·자산 관제실`을 기본안으로 한다.
2. 기존 직원 포함 `은행 알림` 방과 CRM 입금 자동화는 변경하지 않는다.
3. 계좌별 토픽명은 `번호_은행코드_끝4자리_용도` 형식으로 통일한다.
4. 파일럿은 대표 계좌 1개로 시작하고, 원장님이 보안 채널에서 2번째 계좌를 지정하면 2개까지 확장한다.
5. 파일럿 1순위는 기존 CRM 입금 수집에서 이미 검증된 매출입금 계좌의 **읽기 복제**다. 단, 신규 방 발송은 CRM 자동반영과 분리하고 기존 방 라우팅은 건드리지 않는다.
6. 파일럿 2순위는 CRM과 직접 연결되지 않는 운영비/개인/카드대금성 계좌 1개다. 이 계좌는 출금·잔액 흐름 검증용이며 CRM 반영 후보로 쓰지 않는다.
7. `00_전체요약`은 파일럿 기간에는 매일 오전 9시 전후 1회만 발송한다. 실시간 원문 재전송은 하지 않는다.

### 11.2 아직 승인 전까지 보류

1. 실제 계좌 식별값, chat ID, thread ID, bot token은 이 문서에 쓰지 않는다.
2. 신규 Finance bot 생성 여부는 실서비스 변경 승인 시 확정한다. 기본 권장안은 기존 Bank bot 재사용이 아니라 전용 Finance bot 분리다.
3. 거래 원장 저장소는 파일럿 전에는 `NocoDB 비공개 테이블`을 1순위로 두되, 실제 생성은 별도 승인 후 진행한다.
4. 고액 거래 기준은 실제 파일럿 데이터 분포를 본 뒤 확정한다. 파일럿 전에는 모든 미분류/파싱 실패를 검토 대상으로 본다.

## 12. 추천 MVP

가장 안전한 MVP는 다음이다.

1. 기존 `은행 알림` 방은 절대 변경하지 않는다.
2. 새 `P21 원장님 은행·자산 관제실` 방을 만든다.
3. 계좌별 토픽과 `00_전체요약`, `99_미분류_검토`만 만든다.
4. 대표 계좌 1~2개만 1주 파일럿으로 연결한다.
5. 계좌별 토픽에는 원문/상세, `00_전체요약`에는 일일 요약만 보낸다.
6. 거래는 반드시 원장 테이블에도 저장한다.

이 구조는 사용자가 말한 “ALL에서 각 계좌별 입출금 내역을 모아 전체 자산 분석을 한다”는 장점을 살리면서, Telegram을 데이터베이스처럼 과신하는 위험은 줄인다.

## 13. 신규 Telegram 방/토픽 ID 확보 계획

이 단계는 **ID를 확보하는 운영 절차만 정의**한다. 실제 raw 값은 문서, 커밋, 채팅 답변에 남기지 않는다.

### 13.1 생성 전 가드레일

1. 기존 `은행 알림` 방에는 새 봇, 새 토픽, 새 라우팅을 추가하지 않는다.
2. CRM 관련 워크플로우는 읽기/참조만 가능하며, 실서비스 변경 전에는 비활성 초안 또는 별도 파일럿 WF로만 작업한다.
3. 신규 방 멤버는 원장님, 장지호, 전용 Finance bot까지만 허용한다.
4. OpenClaw/AI 명령 봇, 직원, 외부 협력자는 초대하지 않는다.
5. ID 수집 로그는 로컬 임시 파일에도 raw 값을 남기지 않거나, 남기더라도 비밀 저장소 이동 후 즉시 삭제한다.

### 13.2 방/토픽 생성 순서

1. Telegram에서 신규 슈퍼그룹 `P21 원장님 은행·자산 관제실`을 만든다.
2. Topics 기능을 켠다.
3. 필수 토픽을 먼저 만든다.
   - `00_전체요약`
   - `99_미분류_검토`
   - `01_NH_****_매출입금` 또는 원장님이 지정한 파일럿 1 계좌 토픽
   - 원장님이 2번째 계좌를 지정한 경우 `02_은행코드_****_용도`
4. 전용 Finance bot을 추가하되, 가능한 한 메시지 발송에 필요한 최소 권한만 준다.
5. 각 토픽에 아래 형식의 등록 메시지를 1회 보낸다.

```text
REGISTER:P21_OWNER_FINANCE:<topicKey>
```

예:

```text
REGISTER:P21_OWNER_FINANCE:SUMMARY
REGISTER:P21_OWNER_FINANCE:REVIEW
REGISTER:P21_OWNER_FINANCE:PILOT_A
```

### 13.3 ID 수집 방법

권장 순서:

1. 전용 Finance bot 기준으로만 일회성 수집한다.
2. Webhook이 설정된 봇이면 n8n 실행 기록에서 등록 메시지 이벤트를 확인한다.
3. Webhook이 없는 신규 봇이면 일회성 `getUpdates` 또는 Telegram Trigger 수동 실행으로 등록 메시지만 읽는다.
4. 수집 후 raw `chat_id`, `message_thread_id`는 비밀 저장소나 운영 환경변수에만 넣는다.
5. 문서에는 아래처럼 raw 값 없이 확보 상태만 남긴다.

| 항목 | 환경변수명 | 문서 기록값 |
|---|---|---|
| 신규 방 | `P21_TG_OWNER_FINANCE_CHAT_ID` | 확보됨/미확보 |
| 전체요약 토픽 | `P21_TG_OWNER_FINANCE_THREAD_SUMMARY` | 확보됨/미확보 |
| 미분류 검토 토픽 | `P21_TG_OWNER_FINANCE_THREAD_REVIEW` | 확보됨/미확보 |
| 파일럿 A 토픽 | `P21_TG_OWNER_FINANCE_THREAD_PILOT_A` | 확보됨/미확보 |
| 파일럿 B 토픽 | `P21_TG_OWNER_FINANCE_THREAD_PILOT_B` | 필요 시 |

### 13.4 ID 수집 검증

실서비스 연결 전에는 아래 4개만 확인한다.

1. `00_전체요약`에 테스트 요약 1건이 들어간다.
2. `99_미분류_검토`에 테스트 검토 1건이 들어간다.
3. 파일럿 계좌 토픽에 테스트 거래 1건이 들어간다.
4. 기존 `은행 알림` 방에는 아무 메시지도 추가로 가지 않는다.

테스트 메시지는 실제 금융정보가 아닌 더미 값만 사용한다.

## 14. 파일럿 계좌 1~2개 확정 기준

실제 계좌번호와 잔액은 문서화하지 않는다. 여기서는 **역할 기준으로만 확정**한다.

### 14.1 파일럿 A — 매출입금 계좌 읽기 복제

- 상태: 1차 파일럿 확정
- 목적: 이미 수집 안정성이 있는 입금성 계좌를 신규 관제방에 계좌별 토픽으로 보내는지 검증
- 데이터 범위: 입금/출금 원문, 잔액, CRM 후보 여부 표시
- 금지: CRM 자동반영 호출, 기존 `은행 알림` 방 라우팅 변경, 기존 CRM 메시지 포맷 변경
- 토픽 예시: `01_NH_****_매출입금`
- 성공 기준:
  1. 신규 방 파일럿 토픽에만 메시지가 들어간다.
  2. 기존 `은행 알림` 방 메시지 수와 포맷은 변하지 않는다.
  3. 동일 거래의 중복 발송 여부를 `transactionId`로 추적할 수 있다.

### 14.2 파일럿 B — 비CRM 운영비/개인/카드성 계좌

- 상태: 원장님이 보안 채널에서 계좌 별칭을 지정하면 2차 파일럿로 확정
- 목적: 출금·고정비·카드대금성 흐름과 잔액 변화를 신규 관제방에서 볼 수 있는지 검증
- 데이터 범위: CRM과 직접 연결하지 않는 거래 원문과 분류 후보
- 금지: CRM 관련 후보로 자동 분류, 직원 포함 방 공유, 기존 Bank bot 라우팅 변경
- 토픽 예시: `02_은행코드_****_운영비` 또는 `02_은행코드_****_개인`
- 성공 기준:
  1. 출금/자동이체/카드성 거래가 CRM과 분리되어 표시된다.
  2. 내부이체 후보가 있으면 `90_내부이체후보` 또는 `99_미분류_검토`로만 간다.
  3. 알림량이 과하면 즉시 일괄 요약 모드로 낮출 수 있다.

### 14.3 파일럿 기간과 판정

- 기간: 첫 실거래 수집일부터 7일
- 일일 확인: 누락, 중복, 잔액 불일치, 토픽 오배정, 알림 과다
- 중단 조건:
  1. 기존 `은행 알림` 방 또는 CRM 워크플로우에 영향이 감지됨
  2. raw 금융정보가 허용되지 않은 문서/로그/채팅에 남음
  3. 토픽 오배정이 반복됨
  4. 원장님이 알림량 과다 또는 보안 우려로 중단 요청

## 15. 실서비스 변경 전 승인 절차

승인은 “방을 만들었다”와 “운영 자동화를 연결한다”를 분리한다.

### 15.1 Gate A — 방/토픽 생성 승인

승인 전 확인:

- 신규 방 이름
- 멤버 2명 + 전용 bot만 포함
- 토픽 목록
- raw ID 미기재
- 기존 `은행 알림` 방 무변경

승인 결과물:

- raw 값 없는 토픽 매핑 상태표
- 테스트 더미 메시지 스크린샷 또는 확인 메모

### 15.2 Gate B — 파일럿 자동화 연결 승인

승인 전 제출해야 할 변경 패킷:

| 항목 | 내용 |
|---|---|
| 변경 대상 | 신규 Finance 파일럿 WF 또는 비활성 초안 WF |
| 금지 대상 | 기존 `은행 알림` 방, WF-CRM 입금 자동반영 엔진, 기존 CRM 메시지 포맷 |
| 파일럿 계좌 | `PILOT_A`, 필요 시 `PILOT_B` 별칭만 표기 |
| 비밀값 위치 | 환경변수/비밀 저장소 이름만 표기 |
| 테스트 | 더미 거래 dry-run, 신규 방 토픽 라우팅 확인 |
| 롤백 | 파일럿 WF 비활성화, 신규 방 bot 권한 제거 |

승인 문구는 아래처럼 변경 범위와 날짜를 포함해 남긴다.

```text
승인: P21_OWNER_FINANCE_PILOT / YYYY-MM-DD / 신규 관제방 파일럿만 / 기존 은행 알림·CRM 자동화 변경 금지
```

### 15.3 Gate C — 파일럿 종료 후 확대 승인

7일 파일럿 리포트에 아래가 모두 있어야 확대한다.

1. 누락/중복/오배정 건수
2. 잔액 불일치 건수
3. 기존 CRM 영향 없음 확인
4. raw ID/토큰/금융정보 노출 없음 확인
5. 원장님이 실제 알림량을 수용 가능하다고 확인

## 16. 즉시 실행 체크리스트

실서비스 변경 없이 지금 진행 가능한 작업:

- [ ] 원장님이 보안 채널에서 파일럿 A 계좌 별칭 확인
- [ ] 원장님이 파일럿 B 사용 여부 결정
- [ ] 신규 방 생성
- [ ] Topics ON
- [ ] `00_전체요약`, `99_미분류_검토`, 파일럿 계좌 토픽 생성
- [ ] 각 토픽 등록 메시지 발송
- [ ] raw ID는 비밀 저장소에 저장하고 문서에는 `확보됨`만 기록
- [ ] Gate A 승인 메모 작성

아직 하지 말아야 할 작업:

- [ ] 기존 `은행 알림` 방 설정 변경
- [ ] 기존 Bank bot의 기존 라우팅 변경
- [ ] WF-CRM 입금 자동반영 엔진 수정
- [ ] 실제 계좌번호, 잔액, chat ID, thread ID를 문서/채팅/커밋에 기록
- [ ] 승인 전 n8n 운영 WF 활성화
