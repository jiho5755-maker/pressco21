# 원장님 은행·자산 관제방 설계안

- 작성일: 2026-04-25
- 작업 브랜치: `work/workspace/telegram-bank-room-governance`
- 상태: 설계 초안 / 라이브 Telegram·n8n 변경 없음
- 목적: 기존 직원 포함 `은행 알림` 방은 유지하고, 원장님+장지호 전용의 계좌별 토픽 기반 자산 관제방을 새로 설계한다.

## 1. 결론

새 방을 만드는 방향이 맞다.

기존 `은행 알림` 방은 직원과 함께 보는 **입금·미수관리 실무방**으로 유지한다. 새 방은 원장님과 장지호만 보는 **은행·자산 관제방**으로 분리한다. 이 분리는 단순히 방을 하나 더 만드는 것이 아니라, `CRM 처리`와 `경영/자산 분석`의 책임을 분리하는 안전장치다.

권장 최종 구조:

| 구분 | 기존 방 | 신규 방 |
|---|---|---|
| 목적 | 입금 확인, CRM 미수/반영 검토 | 원장님 전체 계좌의 입출금·잔액·자산 흐름 관찰 |
| 멤버 | 원장님 + 장지호 + 직원 + Bank bot | 원장님 + 장지호 + 전용 Bank/Finance bot |
| 데이터 범위 | CRM 관련 입금/출금 알림 중심 | 원장님이 쓰는 모든 주요 계좌 |
| 공개 수준 | 직원 실무 공유 가능 범위 | 매우 민감한 금융/자산 정보 |
| 자동화 책임 | 미수관리/입금반영 판단 | 자산 모니터링/리포팅/이상거래 검토 |
| 변경 방침 | 변경 없음 | 신규 구축 |

## 2. 기존 방을 그대로 두는 이유

현재 문서상 기존 `은행 알림` 그룹은 `원장님 + 장지호 + 이재혁 + Bank_bot` 구성이며, 농협 입금 실시간 및 CRM 검토 용도로 정의되어 있다.

- `docs/ai-native-upgrade/P1-bot-wf-matrix.md`
- `docs/ai-native-upgrade/P1-telegram-bot-reorganization.md`
- `docs/offline-crm-bank-automation-ops-guide-2026-03-15.md`

특히 기존 CRM 은행 자동화 운영 규칙은 다음 원칙을 가진다.

- 입금은 CRM과 매칭해 `자동반영 / 검토 / 미매칭`으로 분기한다.
- 출금은 CRM 자동반영 대상이 아니다.
- 같은 입금 건에 대해 원본 은행 알림과 CRM 처리 결과를 따로 두 번 보내지 않는다.

따라서 기존 방에 원장님 전체 계좌 알림까지 넣으면 다음 문제가 생긴다.

1. 직원에게 원장님 전체 계좌/자산 정보가 노출된다.
2. 미수관리 알림과 개인/자산 흐름이 섞여 CRM 판단이 흐려진다.
3. 향후 자동 파싱이 어느 메시지를 CRM 대상이라고 봐야 하는지 불명확해진다.
4. 알림 과부하로 실무자가 중요한 미수/입금 알림을 놓칠 수 있다.

## 3. 신규 방 이름과 역할

권장 이름:

- `P21 원장님 은행·자산 관제실`
- 또는 `원장님 은행·자산 현황`

역할:

1. 원장님 관련 전체 주요 계좌의 입출금 원문을 계좌별 토픽에 모은다.
2. Telegram의 `ALL` 뷰에서는 전체 계좌 흐름을 시간순으로 볼 수 있게 한다.
3. 별도 `00_전체요약` 토픽에는 원문 중복이 아니라 일/주/月 요약과 이상징후만 올린다.
4. 장기적으로 NocoDB/Sheet/DB 원장에 정규화해 자산 분석, 현금흐름 리포트, 내부이체 제거, 월간 자산 추이를 만든다.

핵심 원칙:

> Telegram은 관제 화면이고, 분석의 정본은 정규화된 거래 원장이다.

## 4. 권장 토픽 구조

사용자의 아이디어처럼 **계좌별 토픽**을 기본 단위로 둔다. 카테고리별 토픽보다 계좌별 토픽이 원문 추적과 은행별 장애 대응에 유리하다.

### 4.1 기본 토픽

| 토픽 | 용도 | 원문/요약 |
|---|---|---|
| `00_전체요약` | 일일/주간/월간 전체 요약, 자산 스냅샷, 큰 변동 알림 | 요약만 |
| `01_NH_1234_사업운영` | 농협 사업운영 계좌 | 원문 + 정규화 요약 |
| `02_NH_5678_매출입금` | 농협/기타 매출 입금 계좌 | 원문 + 정규화 요약 |
| `03_KB_1234_개인` | 국민 개인 또는 별도 목적 계좌 | 원문 + 정규화 요약 |
| `04_SH_1234_예적금` | 예적금/자산 이동 계좌 | 원문 + 정규화 요약 |
| `05_CARD_카드대금` | 카드 결제/자동이체/대출/이자 | 원문 + 정규화 요약 |
| `90_내부이체후보` | 계좌 간 이동으로 보이는 거래 | 검토용 |
| `99_미분류_검토` | 파싱 실패, 계좌 미등록, 고액/이상거래 | 검토용 |

`1234`는 계좌 끝 4자리만 쓰고, 전체 계좌번호는 노출하지 않는다.

### 4.2 ALL 뷰 운영 원칙

Telegram의 `ALL`은 모든 토픽 메시지가 합쳐져 보이는 사용자 화면이다. 이를 활용하면 전체 입출금 타임라인을 자연스럽게 볼 수 있다.

다만 같은 거래를 `계좌 토픽`과 `00_전체요약`에 모두 원문으로 중복 발송하면 알림이 과도해진다. 따라서 원칙은 다음과 같다.

- 계좌별 토픽: 실시간 원문/상세
- Telegram ALL: 계좌별 토픽 메시지가 자연스럽게 합쳐지는 전체 타임라인
- `00_전체요약`: 원문 재전송 금지, 정기 요약/고액 변동/이상징후만 발송

## 6. 데이터 원장 최소 필드

Telegram 메시지만으로는 장기 분석이 어렵다. 처음부터 최소 원장 필드를 정해두는 것이 좋다.

| 필드 | 설명 |
|---|---|
| `transactionId` | 중복 방지용 고유 ID |
| `source` | Gmail, 은행메일, 수동입력, API 등 |
| `occurredAt` | 실제 거래일시 |
| `collectedAt` | 시스템 수집일시 |
| `bankName` | 은행명 |
| `accountAlias` | `NH 1234 사업운영` 같은 별칭 |
| `accountMasked` | 마스킹 계좌번호 |
| `direction` | `deposit` / `withdrawal` |
| `amount` | 금액 |
| `balanceAfter` | 거래 후 잔액 |
| `counterparty` | 입금자/출금처 |
| `memo` | 적요/메모 |
| `topicKey` | 라우팅 토픽 키 |
| `messageThreadId` | Telegram 토픽 ID |
| `category` | 사업수입, 운영비, 개인이체, 세금, 카드, 대출 등 |
| `classificationStatus` | 자동분류 / 검토필요 / 수동확정 |
| `isInternalTransfer` | 내부 계좌 간 이체 여부 |
| `crmRelated` | CRM 관련 후보 여부 |
| `crmMatchStatus` | 자동반영 / 검토 / 미매칭 / 해당없음 |

## 7. CRM 미수관리와의 경계

새 방은 기존 CRM 자동반영의 정본이 아니다. 초기에는 다음 경계를 유지한다.

### 기존 `은행 알림` 방

- CRM 입금반영, 미수관리, 직원 공유가 목적
- 현재 농협 입금/출금 및 CRM 처리 결과를 계속 담당
- 기존 n8n 워크플로우와 방 설정 변경 없음

### 신규 `원장님 은행·자산 관제실`

- 전체 자산 흐름 관찰과 리포팅이 목적
- CRM 관련 거래 여부는 표시할 수 있지만, CRM 상태를 직접 바꾸는 방이 아니다
- CRM과 연결하려면 별도 검증 단계를 거친 뒤 기존 CRM 엔진으로 전달한다

같은 거래가 두 방에 보일 수는 있다. 하지만 의미는 다르다.

- 기존 방: “이 입금이 고객 미수에 반영됐는가?”
- 신규 방: “전체 자금 흐름에서 이 거래가 어떤 의미인가?”

## 8. 보안·권한 가드레일

금융 정보이므로 기본값을 보수적으로 둔다.

1. 사람 멤버는 원장님과 장지호만 둔다.
2. 직원, 외부 협력자, 일반 운영 봇은 초대하지 않는다.
3. OpenClaw/AI 명령 봇은 절대 초대하지 않는다.
4. 은행/재무 전용 봇만 초대하고, 가능한 한 전송 권한만 사용한다.
5. 전체 계좌번호는 메시지에 쓰지 않는다. 은행명 + 끝 4자리 + 용도 별칭만 쓴다.
6. 토큰, chat ID, thread ID는 문서에 원문으로 쓰지 않고 운영 환경변수/비밀 저장소에서 관리한다.
7. 토픽 ID 매핑은 명명된 환경변수로 관리한다.
8. 파싱 실패, 계좌 미등록, 고액 거래, 중복 의심은 `99_미분류_검토`로 보낸다.
9. Telegram `ALL`은 보기용이다. 회계/자산 분석의 정본은 원장 테이블이어야 한다.
10. 새로운 방의 메시지를 기존 CRM 방으로 자동 포워딩하지 않는다.

권장 환경변수 이름 예시:

```text
P21_TG_OWNER_FINANCE_CHAT_ID
P21_TG_OWNER_FINANCE_THREAD_SUMMARY
P21_TG_OWNER_FINANCE_THREAD_REVIEW
P21_TG_OWNER_FINANCE_THREAD_NH_1234_OPERATING
P21_TG_OWNER_FINANCE_THREAD_KB_1234_PERSONAL
```

## 9. 자동화 아키텍처 초안

```text
은행 알림 원천
  ├─ Gmail/보안메일
  ├─ 은행 앱/메일/문자 연동
  └─ 수동 입력 또는 추후 API
        ↓
수집/파서 n8n
        ↓
거래 정규화 + 중복 제거
        ↓
은행 거래 원장 저장
        ↓
계좌 별칭/토픽 라우팅 레지스트리 조회
        ↓
Telegram 신규방 계좌별 Topic 발송
        ↓
검토 필요 건은 99_미분류_검토
        ↓
일/주/月 요약은 00_전체요약
```

중요한 점:

- 현재는 기존 CRM 방을 건드리지 않는다.
- 새 방 파일럿은 별도 n8n 워크플로우 또는 기존 파서의 “읽기 복제” 형태로 시작한다.
- CRM 자동반영과 자산 원장은 transactionId로 연결하되, 첫 단계에서는 느슨하게 연결한다.

## 10. 단계별 도입안

### Phase 0 — 설계/권한 정리

- 이 문서 확정
- 기존 `은행 알림` 방 변경 금지 재확인
- 원장님과 장지호 전용 신규 방 생성 준비
- 수집할 계좌 목록/은행/용도/알림 가능 여부 정리

### Phase 1 — Telegram 방/토픽 생성

- 새 슈퍼그룹 생성
- 토픽 모드 ON
- 계좌별 토픽 생성
- `00_전체요약`, `99_미분류_검토` 생성
- 각 토픽의 `message_thread_id` 확보

### Phase 2 — 파일럿

- 대표 계좌 1~2개만 연결
- 기존 CRM 방은 무변경
- 새 방에 원문/요약이 올바른 토픽으로 가는지 확인
- 1주간 중복, 누락, 잔액 불일치, 알림 과다 여부 검증

### Phase 3 — 원장화

- 거래 원장 테이블 생성
- `transactionId` 중복 제거
- 계좌별 잔액/순증감/내부이체 후보 계산
- 일일 요약 자동 발송

### Phase 4 — 전체 계좌 확대

- 원장님 주요 계좌 전체 연결
- 카드/대출/예적금/세금 자동이체 분류 추가
- 주간/월간 자산 리포트 자동 생성

### Phase 5 — CRM/경영 분석 연결

- CRM 미수 회수액과 실제 입금 흐름 비교
- 사업/개인/내부이체/고정비 분리
- 월간 현금흐름, 잔액 추이, 고정비 변화 리포트

## 11. 우선 결정할 사항

1. 새 방 이름을 무엇으로 할지
2. 1차 파일럿 계좌를 어느 1~2개로 할지
3. 계좌별 토픽명을 `은행+끝4자리+용도` 형식으로 통일할지
4. 기존 Bank bot을 재사용할지, 신규 Finance bot을 만들지
5. 거래 원장을 NocoDB, Google Sheet, SQLite/DB 중 어디에 둘지
6. `00_전체요약`은 매일 몇 시에 발송할지
7. 고액 거래 기준을 얼마로 둘지

## 12. 추천 MVP

가장 안전한 MVP는 다음이다.

1. 기존 `은행 알림` 방은 절대 변경하지 않는다.
2. 새 `P21 원장님 은행·자산 관제실` 방을 만든다.
3. 계좌별 토픽과 `00_전체요약`, `99_미분류_검토`만 만든다.
4. 대표 계좌 1~2개만 1주 파일럿으로 연결한다.
5. 계좌별 토픽에는 원문/상세, `00_전체요약`에는 일일 요약만 보낸다.
6. 거래는 반드시 원장 테이블에도 저장한다.

이 구조는 사용자가 말한 “ALL에서 각 계좌별 입출금 내역을 모아 전체 자산 분석을 한다”는 장점을 살리면서, Telegram을 데이터베이스처럼 과신하는 위험은 줄인다.
