# PRESSCO21 n8n 효율화 검토서

작성일: 2026-03-09  
대상: 대표, Claude Code, Codex, 후속 자동화 구현 담당자

## 목적

이 문서는 PRESSCO21 운영 환경에서 `n8n`을 어디까지 유지하고, 어디서부터 코드 기반 구현으로 전환해야 하는지 판단하기 위한 기준 문서다.

핵심 질문은 두 가지다.

1. 현재 `n8n` 세팅이 실제 운영상 필요한가
2. 어떤 영역에서는 `n8n`이 오히려 토큰과 유지보수 비용을 키우는가

이 문서는 이후 Claude Code가 새 작업을 시작할 때 아래 둘 중 하나를 결정하는 근거로 사용한다.

- `n8n` 유지 또는 확장
- `n8n` 축소 또는 코드 기반 이관

## 결론 요약

### 최종 판단

`n8n`은 현재 PRESSCO21에 필요하다.  
다만 모든 프로젝트의 기본 백엔드로 계속 쓰는 것은 비효율이 커지고 있다.

### 유지해야 하는 영역

- 메이크샵 뒤에서 동작하는 예약, 주문 폴링, 정산, 승인, 이메일, 텔레그램 알림
- 일정 기반 배치 작업
- 외부 시스템 3개 이상을 연결하는 운영 자동화
- 시크릿 보관, 재시도, 실행 로그 확인이 중요한 흐름

### 축소하거나 이관해야 하는 영역

- 단순 CRUD 프록시
- 앱 내부 검색/필터/리포트 계산
- UI 변경에 따라 자주 수정되는 비즈니스 로직
- `n8n`이 사실상 API 게이트웨이 역할만 하는 구조

### 현재 가장 먼저 손봐야 할 대상

1. `offline-crm-v2`의 `n8n` 프록시
2. `WF-05` 같은 대형 단일 워크플로우
3. 자주 바뀌는 규칙이 길게 들어간 Code 노드

## 이 판단의 근거

### 1. 저장소 구조 자체가 이미 `n8n` 중심 운영을 전제로 한다

- `README.md`
  - `파트너클래스/`를 `6페이지 + n8n WF` 구조로 정의
  - 전체 기술 스택을 `n8n + NocoDB` 중심으로 설명
- `docs/파트너클래스/README.md`
  - 현재 기준 문서 인덱스와 파트너클래스 문서 우선순위를 정리
  - `shared-service-identity.md`, `enterprise-elevation-strategy-2026-03-10.md`를 최신 판단 기준으로 연결

이 말은 현재 `n8n`이 부가 기능이 아니라 운영 중간층이라는 뜻이다.

### 2. 파트너클래스는 `n8n`이 실제 백엔드 오케스트레이터 역할을 한다

대표 흐름:

- WF-01: 목록/상세 조회
- WF-04: 예약 저장
- WF-05: 주문 폴링, 정산 기록, 배치 작업
- WF-07/08/APPROVE: 신청/승인/회원그룹 변경
- WF-11/12/13: 리마인더, 후기 요청, 등급 업데이트

특히 `파트너클래스/n8n-workflows/WF-05-order-polling-batch.json`은 다음을 한꺼번에 담당한다.

- 10분 폴링
- 자정 정합성 검증
- 03시 상품 동기화
- 04시 실패 재시도
- 05시 D+3 정산

이 영역은 `n8n`이 적합하다. 이유는 다음과 같다.

- 스케줄 실행이 핵심이다
- 메이크샵 API, NocoDB, 이메일, 텔레그램을 연결한다
- 실패 시 재처리와 운영 로그 확인이 중요하다
- 비전공자 운영자가 GUI 실행 이력을 보는 가치가 크다

### 3. 반대로 CRM은 현재 `n8n`을 너무 얇게 사용한다

`offline-crm-v2/src/lib/api.ts`와  
`파트너클래스/n8n-workflows/WF-CRM-PROXY-nocodb-proxy.json`을 보면 구조가 명확하다.

현재 구조:

`React App -> x-crm-key -> n8n proxy -> xc-token -> NocoDB`

이 프록시는 주로 아래만 한다.

- API key 확인
- 허용 테이블/메서드 검증
- NocoDB URL 변환
- 응답 포맷 정리

즉 `n8n`이 실제 자동화 오케스트레이션이 아니라, 보안 프록시 겸 라우터 역할만 하고 있다.

이 영역은 다음 이유로 코드 기반 API가 더 적합하다.

- 구조가 단순하다
- UI 요구에 따라 자주 바뀐다
- 프론트 변경과 같이 움직이는 경우가 많다
- 워크플로우 JSON을 읽고 수정하는 토큰 비용 대비 이점이 약하다

### 4. 메이크샵 API 한도는 `n8n` 유지의 핵심 이유가 아니다

`docs/파트너클래스/api-budget-caching-strategy.md` 기준:

- 최악의 경우에도 조회 53회/시간
- 처리 20회/시간
- 메이크샵 한도 대비 여유가 큼

즉 `n8n`의 가치 포인트는 쿼터 절약이 아니다.

실제 가치 포인트:

- 시크릿 보호
- 스케줄/배치
- 재시도
- 운영 로그
- 외부 시스템 연동

### 5. 전체 작업 방식 차원에서는 `n8n`이 범용 필수 도구가 아니다

`/Users/jangjiho/workspace` 하위에는 `notion-cms`, `work-management` 같은 일반 앱형 프로젝트가 있고, 이들은 Next.js 중심으로 운영된다.

또한 블로그 자동화 로드맵에서도 `n8n`은 핵심 발행 파이프라인이 아니라 후행 알림/색인 제출 단계에서만 붙는다.

이 말은 곧 다음을 의미한다.

- 사용자님의 전체 작업 방식에 `n8n`이 절대 필수인 것은 아니다
- PRESSCO21 중에서도 특정 운영 자동화 영역에만 강하게 적합하다

## 앞으로의 의사결정 원칙

새 기능을 만들 때 아래 기준으로 먼저 분류한다.

### A. `n8n`에 넣어야 하는 경우

아래 조건 중 3개 이상이면 `n8n` 우선 검토:

- 스케줄 실행이 필요하다
- 외부 시스템이 3개 이상 연결된다
- 이메일/텔레그램/웹훅 알림이 포함된다
- 운영자가 GUI 로그를 직접 보는 가치가 크다
- 재시도/실패 처리 흐름이 중요하다
- 시크릿을 프론트엔드에서 분리해야 한다
- 수동 운영자가 중간 개입할 수 있어야 한다

### B. 코드로 구현해야 하는 경우

아래 조건 중 2개 이상이면 코드 우선:

- UI와 결합된 CRUD다
- 검색/정렬/필터/리포트가 핵심이다
- 요구사항이 자주 바뀐다
- 프론트 타입과 백엔드 타입을 함께 관리해야 한다
- 테스트 자동화가 중요하다
- `n8n` 없이도 단일 API 서버로 충분하다
- 실제 자동화보다 앱 내부 도메인 로직 비중이 높다

### C. `n8n` 사용 금지에 가까운 경우

다음 경우는 특별한 이유가 없으면 `n8n`으로 만들지 않는다.

- 단순 프록시
- 단순 DB CRUD
- 단일 서비스 호출 뒤 응답만 전달
- 프론트의 뷰 로직 변화에 따라 자주 수정되는 엔드포인트

## 워크플로우 설계 가드레일

이 항목은 Claude Code가 실제 구현 여부를 결정할 때 지켜야 한다.

### 1. 기본 원칙

- `1 trigger = 1 책임`
- Webhook형과 Schedule형 책임을 섞지 않는다
- 한 워크플로우 안에서 unrelated 배치를 합치지 않는다
- 자주 바뀌는 규칙을 긴 Code 노드 하나에 몰아넣지 않는다

### 2. 노드 수 기준

강제 규칙은 아니지만, 아래를 넘기면 분할 검토를 먼저 한다.

- Webhook 중심 워크플로우: 15노드 전후
- 배치 중심 워크플로우: 25노드 전후
- 30노드 이상: 분할 설계 검토 필수
- 50노드 이상: 유지 필요성 재검토 필수

현재 `WF-05`는 이 기준을 넘으므로 분할 대상이다.

### 3. Code 노드 기준

아래 중 하나라도 해당하면 Code 노드를 얇게 하거나 외부 코드/문서로 분리한다.

- 비즈니스 규칙이 자주 바뀐다
- 프롬프트/정책/분기 로직이 길다
- 같은 계산 로직이 여러 워크플로우에 복제된다
- AI가 매번 전체 JSON을 읽어야 수정 가능하다

## 토큰 비용 관점에서의 원칙

현재 토큰 낭비의 핵심은 `n8n` 존재 자체보다 `큰 워크플로우 JSON을 자연어로 계속 수정하는 방식`이다.

따라서 다음 원칙을 적용한다.

### 1. 전체 JSON을 매번 읽지 않는다

- 필요한 워크플로우만 읽는다
- 필요한 노드만 발췌한다
- 관련 문서와 책임 범위를 먼저 요약한 뒤 수정한다

### 2. 자주 수정되는 로직은 파일 기반으로 관리한다

예시:

- 수수료 정책
- 분기 규칙
- 프롬프트 텍스트
- 상태 전이 규칙
- 에러 코드 정책

가능하면 문서 또는 코드 파일에 canonical source를 두고, 워크플로우는 이를 반영하는 얇은 계층으로 유지한다.

### 3. 대형 워크플로우는 분할이 곧 토큰 절감이다

특히 다음은 효과가 크다.

- `WF-05` 분할
- 조회 API와 배치 API 분리
- 승인/알림/후처리 분리

## 우선순위 로드맵

### Phase 1. 즉시 적용

목표: 새로운 낭비를 막는다.

- 새 기능 시작 전 이 문서 기준으로 `유지 / 하이브리드 / 코드 이관` 먼저 판정
- `n8n`을 기본 선택지로 쓰지 않는다
- 30노드 이상 워크플로우는 확장 전에 분할 여부를 먼저 검토

### Phase 2. 이관 1순위

목표: 가장 얇은 `n8n` 사용처를 제거한다.

대상:

- `offline-crm-v2`의 `WF-CRM-PROXY`

권장 방향:

- 소형 TypeScript API 또는 Edge Function으로 대체
- 인증, 화이트리스트, NocoDB 호출만 유지
- 프론트와 타입을 더 직접적으로 맞춘다

기대 효과:

- CRM 관련 토큰 비용 절감
- 프론트와 API 동시 수정 난이도 감소
- 테스트와 배포 단순화

### Phase 3. 구조 개선 1순위

목표: 유지해야 하는 `n8n`을 더 싸게 유지한다.

대상:

- `WF-05 Order Polling + Batch Jobs`

권장 분할안:

1. 주문 폴링
2. 정합성 검증
3. 상품 동기화
4. 실패 재시도
5. D+3 정산

기대 효과:

- AI가 수정할 때 읽는 컨텍스트 축소
- 장애 원인 분리
- 실행 로그 해석 용이

### Phase 4. 중기

목표: `n8n`의 역할을 운영 자동화 버스로 고정한다.

- 새 앱 프로젝트는 앱/서버 코드 중심으로 시작
- `n8n`은 알림/배치/연동 계층으로만 붙인다
- 장기적으로는 워크플로우 인벤토리 문서와 책임 맵을 유지한다

## Claude Code용 실행 판단 체크리스트

Claude Code는 새 작업 또는 리팩터링 요청을 받을 때 아래 순서로 판단한다.

### Step 1. 이 작업이 진짜 자동화인가

질문:

- 스케줄이 필요한가
- 외부 시스템이 3개 이상 연결되는가
- 실패 재시도/알림/운영 로그가 중요한가

판정:

- 예가 많으면 `n8n` 후보
- 아니면 코드 후보

### Step 2. 이 작업이 사실상 앱 백엔드인가

질문:

- 프론트와 함께 자주 바뀌는가
- CRUD, 검색, 리포트가 중심인가
- 타입 안정성과 테스트가 중요한가

판정:

- 예면 코드 우선

### Step 3. 기존 `n8n`을 확장해야 하는가, 줄여야 하는가

질문:

- 기존 워크플로우 노드 수가 이미 큰가
- 책임이 여러 개인가
- Code 노드에 규칙이 과도하게 몰려 있는가

판정:

- 예면 새 노드를 더 붙이지 말고 먼저 분할 검토

### Step 4. 즉시 행동 규칙

다음 경우 Claude는 기본적으로 `확장하지 말고 축소/이관 제안`부터 해야 한다.

- `offline-crm-v2` 관련 새 API 요청
- 단순 프록시 추가 요청
- 기존 30노드 이상 워크플로우에 unrelated 기능 추가 요청

다음 경우 Claude는 기본적으로 `n8n 유지`를 전제로 설계해도 된다.

- 파트너클래스 주문/정산/알림/승인
- 정부지원사업 수집/분석/알림
- 업무 자동화 리마인더/허브

## 보류 판단이 필요한 경우

아래는 즉시 구현보다 먼저 운영 판단이 필요한 항목이다.

- 파트너클래스 조회 API를 코드로 옮길지 여부
- 챗봇 백엔드를 `n8n`에 계속 둘지 여부
- 장기적으로 NocoDB 직접 접근 정책을 어떻게 통일할지 여부

이 경우 Claude는 바로 구현하지 말고 다음 두 가지를 먼저 제시해야 한다.

1. 유지 시 장단점
2. 이관 시 장단점과 예상 작업량

## 요약된 의사결정

### 계속 써야 하는 곳

- 파트너클래스 운영 자동화
- 업무/정부지원사업 자동화
- 메인페이지 배치형 콘텐츠 동기화

### 줄여야 하는 곳

- CRM 프록시
- 대형 단일 워크플로우
- 자주 바뀌는 Code 노드 규칙

### 기본 정책

앞으로 PRESSCO21에서 `n8n`은 기본 백엔드가 아니라, 운영 자동화 오케스트레이터로 제한한다.
