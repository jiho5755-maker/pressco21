# Offline CRM 은행 자동화 운영 규칙

작성일: 2026-03-15  
대상 시스템: PRESSCO21 Offline CRM, n8n, Gmail NH 보안메일 수집, Telegram `은행 알림`

## 1. 목적

이 문서는 농협 계좌 입출금 메일을 Gmail로 수집한 뒤:

- 관계자에게 즉시 텔레그램 알림을 보내고
- 입금 건만 CRM과 매칭해 반자동으로 처리하는

현재 운영 기준을 정리한다.

핵심 원칙은 다음 두 가지다.

- `출금`은 알림 전용으로 처리한다.
- `입금`은 CRM 매칭 후 `자동반영 / 검토 / 미매칭`으로 분기한다.

## 2. 현재 운영 플로우

### 2.1 은행 이벤트 수집

수집 경로:

- 농협 입출금 메일
- Gmail `presscobank@gmail.com`
- n8n `WF-CRM-02 Gmail 입금알림 수집`

수집 대상:

- 입금
- 출금

수집 결과:

- `transactions[]`: 입금/출금 전체 거래
- `deposits[]`: 입금만 분리한 서브셋

### 2.2 관계자 알림

`WF-CRM-02`는 거래를 파싱한 뒤 텔레그램 그룹 `은행 알림`에 거래당 메시지 1개만 보낸다.

운영 기준:

- `입금`은 은행 정보와 CRM 처리 결과를 한 메시지에 합쳐 보낸다.
- `출금`은 CRM 후속 처리 없이 은행 거래 알림만 보낸다.
- 같은 입금 건에 대해 `원본 은행 알림`과 `CRM 처리 결과`를 따로 두 번 보내지 않는다.

메시지 예시:

- `[입금 알림]`
  `계좌 / 입금자 / 입금액 / 거래일시 / 통장잔액 / CRM처리 / 안내`
- `[은행 거래 알림]`
  `계좌 / [출금] 상대방 / 금액 / 거래일시 / 잔액`

### 2.3 CRM 반자동반영

입금만 `WF-CRM-01 입금자동반영 엔진`으로 전달한다.

판정 결과는 다음 중 하나다.

- `자동반영`
- `검토`
- `미매칭`

입금 메시지 하단에는 다음 중 하나의 처리 결과를 붙인다.

- `CRM처리: 자동반영 완료`
- `CRM처리: 검토 필요`
- `CRM처리: 미매칭`

## 3. 판정 규칙

### 3.1 자동반영

아래 조건이 모두 맞으면 자동반영한다.

- 고객 후보가 1명으로 특정된다.
- 해당 고객의 열린 명세표 중 `입금금액과 잔액이 정확히 일치하는 명세표`가 1건이다.

주의:

- 같은 고객에게 미수 명세표가 여러 건 있어도 상관없다.
- 그중 정확 일치 명세표가 1건이면 자동반영한다.

### 3.2 검토

아래 경우는 자동반영하지 않고 검토로 보낸다.

- 부분 입금
- 초과 입금
- 같은 고객 안에서 정확 일치 후보가 2건 이상
- 고객 후보가 여러 명
- 이름은 맞지만 명세표 특정이 불안정

검토 예시:

- `장지호 / 1,500원 / 부분 입금`
- `장지호 / 5,500원`인데 동명이인 또는 다중 후보

### 3.3 미매칭

아래 경우는 미매칭으로 처리한다.

- CRM 고객 후보가 없음
- 고객은 있어도 연결할 명세표 또는 장부 후보가 없음

## 4. 출금 운영 규칙

출금은 현재 CRM 자동반영 대상이 아니다.

운영 기준:

- 텔레그램 알림만 보낸다.
- CRM 입금 수집함에는 반영하지 않는다.
- 회계 분류 자동화는 후속 범위로 둔다.

이유:

- 출금은 운영비, 배송비, 환불, 공급처 지급 등 성격이 다양하다.
- 현재는 오분류보다 알림 우선이 안전하다.

## 5. 텔레그램 운영 기준

현재 텔레그램 그룹:

- `은행 알림`

현재 메시지 레이어:

- `출금 알림`
  - 실시간 자금 유출 확인용
- `입금 알림`
  - 은행 정보와 CRM 처리 결과를 한 번에 확인

권장 운영 방식:

- 관계자는 한 메시지에서 입금/출금 핵심 정보만 빠르게 읽는다.
- 입금 건은 같은 메시지의 `CRM처리`와 `안내` 줄로 후속 조치 여부를 판단한다.

## 6. 실무 처리 규칙

### 6.1 정확 일치 입금

- 기대 동작: 자동반영
- 확인 항목:
  - 텔레그램 입금 알림 1건만 도착
  - 메시지 하단에 `CRM처리: 자동반영 완료` 표시
  - 명세표 잔액 0원 처리

### 6.2 부분 입금

- 기대 동작: 검토
- 확인 항목:
  - 텔레그램 입금 알림 1건만 도착
  - 메시지 하단에 `CRM처리: 검토 필요` 표시
  - 명세표 자동 완납 금지
  - 수동 판단 후 반영

### 6.3 초과 입금

- 기대 동작: 검토
- 확인 항목:
  - 초과 입금 또는 예치금 후보로 판단
  - 자동반영 금지

### 6.4 미매칭 입금

- 기대 동작: 미매칭
- 확인 항목:
  - 텔레그램 입금 알림 1건만 도착
  - 메시지 하단에 `CRM처리: 미매칭` 표시
  - 고객 또는 명세표 신규 연결 필요

### 6.5 출금

- 기대 동작: 알림만
- 확인 항목:
  - 관계자가 금액과 상대방을 식별할 수 있어야 함

## 7. 최근 검증 결과

2026-03-15 기준 실동작 검증 완료:

- NH 보안메일 첨부 `Message.html` 파싱 성공
- 텔레그램 `은행 알림` 발송 성공
- 입금 payload를 `WF-CRM-01`로 전달 성공
- 부분 입금은 `검토`로 분기 성공

실검증 예시:

- 입금자: `장지호`
- 금액: `1,500원`
- 결과:
  - 입금 알림 1건 발송
  - 부분 입금으로 검토 분기

## 8. 인증 운영 규칙

CRM 웹은 세션 인증을 기본으로 사용한다.

다만 자동화 경로는 브라우저 세션과 분리된 전용 인증을 유지해야 한다.

운영 기준:

- 브라우저 화면과 일반 정적 자원 접근은 세션 쿠키로 통과
- 자동화가 읽는 `/data/*` 경로는 `X-CRM-Automation-Key` 헤더로 통과
- 전환 기간 장애 완화를 위해 BasicAuth fallback도 유지

원칙:

- 사람 로그인 방식이 바뀌어도 n8n 자동화는 같이 죽지 않도록 분리한다.
- `/data/*`를 공개로 다시 열지 않는다.
- 자동화 키는 서버 파일 `/etc/pressco21-crm/automation-key`에서 관리한다.

## 9. 향후 개선 우선순위

1. 고객별 텔레그램 chat ID 저장 구조 추가
2. 검토 큐에서 수동 승인/예치금 처리 UX 강화
3. 출금 유형 분류 레이어 추가
4. 자동반영/검토/미매칭 운영 리포트 추가
