# 11번가 Open API 쓰기 작업 안전 체크리스트

> 재고, 가격, 즉시할인, 판매중지, 클레임 승인/거부, 발송처리는 운영 사고로 직결된다.

## 1. 공통 실행 순서

1. 대상 식별자 확보: `prdNo`, `prdStckNo`, `sellerPrdCd`, `ordNo`, `ordPrdSeq`, `dlvNo` 등
2. fresh read: 실행 직전 조회 API로 현재 상태 저장
3. dry-run: 변경 payload와 예상 변경값을 JSON/CSV로 저장
4. 승인: 가격/정산/주문/클레임 영향이 있으면 대표 승인 후 진행
5. execute: 승인된 대상만 `--execute`로 제한 실행
6. verify: 실행 후 조회 API로 실제 반영 확인
7. exception report: 실패 대상, 오류 메시지, 재시도 여부를 별도 파일로 저장

## 2. 중단 기준

아래 중 하나라도 발생하면 일괄 실행을 중단한다.

- HTTP 4xx/5xx 또는 실패성 `resultCode`가 반복된다.
- fresh read 대상 수와 execute 대상 수가 다르다.
- verify에서 변경 후 값이 확인되지 않는다.
- 고객 노출가 상승, 옵션가 한도 초과, 정산 손실 가능성이 발견된다.
- API 관리 화면/IP 등록/키 상태가 의심된다.

## 3. 가격/즉시할인

- 기본즉시할인을 없앨 때는 고객 노출가가 올라가지 않도록 할인 후 실판매가를 새 `selPrc`로 넣고 `cuponcheck=N` 처리한다.
- 11번가는 판매가 기준으로 수수료가 계산될 수 있으므로, 높은 판매가 + 큰 기본즉시할인 구조는 정산 손실 위험이 있다.
- 옵션가격 한도 때문에 수정이 실패할 수 있다. 실패 건은 강제 재시도하지 말고 옵션 구조를 별도 검토한다.

확인할 오류 패턴:

- 판매가는 기본즉시할인금액 이하로 설정할 수 없음
- 판매가는 최대 80% 인하까지만 수정 가능
- 옵션가격이 판매가의 허용 범위를 벗어남

## 4. 재고

- `prdNo`와 `prdStckNo`가 같은 상품/옵션에 매칭되는지 fresh read로 확인한다.
- 품절 해제는 재고 수량뿐 아니라 판매중지 상태, 판매기간, 옵션 품절 여부도 함께 확인한다.
- 실행 후 `stock-detail` 또는 상품상세조회로 반영 상태를 검증한다.

## 5. 주문/클레임/배송

- `GET`이어도 API명이 처리/승인/거부/발송/수정/등록이면 write API로 취급한다.
- 발주확인, 발송처리, 취소/반품/교환 승인·거부는 중복 실행 위험이 있으므로 직전 상태값을 확인한다.
- 송장번호, 택배사 코드, 발송일 포맷을 dry-run 보고서에 포함한다.

## 6. 로그/보안

- API KEY, 세션 쿠키, 추가인증 결과는 출력·커밋·handoff에 남기지 않는다.
- 실패 리포트에는 상품/주문 식별자와 오류 요약만 남긴다.
- 개인정보성 응답은 필요한 필드만 마스킹해 저장한다.
