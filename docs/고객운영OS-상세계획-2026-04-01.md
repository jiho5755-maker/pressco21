## 2. 문제 재정의

현재 PRESSCO21의 고객 운영은 기능별로는 이미 돌아가고 있다. 문제는 `흩어짐`이다.

### 2.1 현재 운영 축

- 자사몰 주문/회원/상품: 메이크샵 관리자
- 오픈마켓 주문: 스마트스토어/쿠팡/11번가 관리자
- 통합 주문/송장/재고 운영: 사방넷
- 전화주문/명세표/입금/미수: `offline-crm-v2`
- 상담: 채널톡 무료 플랜 중심
- 메시지/자동화: 메이크샵 기본 기능 + 일부 n8n + 수동 운영

### 2.2 실제 문제

1. 고객 1명을 여러 시스템에서 따로 보고 있다.
2. 주문과 입금이 연결되지 않는 경우가 많다.
3. 전화주문이 실무상 중요하지만 시스템상 일급 객체가 아니다.
4. 사방넷은 운영 허브지만 고객관계/분석 허브는 아니다.
5. CRM v2는 잘 쓰고 있지만 지금 상태에서 모든 걸 밀어 넣으면 망가질 위험이 있다.
6. 총매출, 실결제액, 순매출, 실수금이 시스템마다 다르게 보일 수 있다.

### 2.3 진짜 목표

`상담툴 교체`가 목표가 아니다.

진짜 목표는 다음이다.

- 고객, 주문, 상담, 입금, 출고, 재구매를 한 기준으로 묶는다.
- 기존 실무 앱은 유지하면서 상위에 통합 허브를 만든다.
- 점진적으로 자동화와 분석을 고도화한다.

## 3. 핵심 결론

### 3.1 가장 중요한 구조 판단

`offline-crm-v2를 중심 허브로 키우지 않는다.`

대신 아래 구조를 사용한다.

- `Customer OS Core`
  - 통합 고객, 통합 주문, 통합 이벤트, 통합 분석, 자동화 기준 허브
- `Execution Apps`
  - `offline-crm-v2`, 사방넷, 메이크샵 관리자, 상담툴, 메시징 툴

즉 `허브는 데이터와 기준을 소유`하고, `실행 앱은 실무를 수행`한다.

### 3.2 왜 이렇게 가는가

이유는 세 가지다.

1. `CRM v2 보호`
   - 현재 잘 쓰는 명세표/입금/미수 흐름을 망가뜨리면 안 된다.
2. `확장성`
   - 향후 상담툴이나 메시징 툴이 바뀌어도 허브는 유지되어야 한다.
3. `데이터 자산화`
   - 고객 기준과 운영 지식은 외부 툴이 아니라 내부에 남아야 한다.

## 4. 시스템 경계

### 4.1 Customer OS Core

Customer OS Core는 새로 만드는 상위 중심 허브다.

책임:

- 고객 기준 ID 소유
- 주문 기준 ID 소유
- 주문/결제/입금/출고/상담 이벤트 정규화
- 통합 고객 타임라인 생성
- 통합 매출 지표 계산
- 자동화 트리거 판단
- 운영 대시보드 제공

### 4.2 offline-crm-v2

`offline-crm-v2`는 유지한다.

책임:

- 명세표 발행
- 입금 처리
- 미수 관리
- 예치금/환불대기
- 전화주문 생성/수정/처리
- 회계성 실무 화면

중요:

- 초기에는 Core가 CRM v2를 `대체`하지 않는다.
- Core가 CRM v2 데이터를 `읽고 요약`하는 구조로 시작한다.

### 4.3 사방넷

책임:

- 통합 주문 수집
- 송장 전송
- 재고 운영
- 출고 상태 운영

중요:

- 사방넷은 계속 운영 허브다.
- 다만 고객관계/분석 기준 허브는 아니다.

### 4.4 메이크샵

책임:

- 자사몰 회원, 주문, 상품, 적립금, 고객 관련 보강 원천

중요:

- 메이크샵은 고객 운영 관점에서 사방넷보다 더 깊은 자사몰 데이터를 가진다.
- 따라서 사방넷만으로 부족한 고객 필드는 메이크샵이 보강한다.

### 4.5 상담툴

책임:

- 실시간 상담 UI

중요:

- 상담툴은 이벤트를 생성하는 도구이지 기준 DB가 아니다.
- 대화 이벤트와 메타데이터는 내부로 복제 적재해야 한다.

## 5. 아키텍처 개요

```text
메이크샵 / 스마트스토어 / 쿠팡 / 11번가 / 사방넷 / 상담툴 / CRM v2
                         ↓
                수집기 (n8n + 브라우저 자동화 + 파일 파서)
                         ↓
             Customer OS Core (Raw + Unified DB + Rules)
                         ↓
        Customer OS Portal / 대시보드 / 메시지 자동화 / 리포트
                         ↓
             CRM v2 / Deep Link / 후속 실행 앱 / 업무 큐
```

### 5.1 왜 Portal이 따로 필요한가

Core는 DB와 룰 엔진이고, 직원이 직접 만지는 UI는 `Portal`이다.

Portal 역할:

- 통합 고객 조회
- 통합 주문 조회
- 통합 타임라인
- 채널별 매출
- 리텐션/운영 큐
- CRM v2/사방넷으로 deep link 이동

즉 Portal은 `읽기 + 판단`, 실행은 기존 앱이 담당한다.

## 6. 데이터 소유권 원칙

### 6.1 잠정 확정

| 영역 | 기준 시스템 |
|------|-------------|
| 고객 기준 ID | Customer OS Core |
| 통합 주문 | Customer OS Core |
| 자사몰 회원 보강 | 메이크샵 |
| 통합 주문 운영 상태 | 사방넷 |
| 명세표/입금/미수/예치금 | CRM v2 |
| 상담 대화 UI | 상담툴 |
| 상담 이벤트 저장 | Customer OS Core |
| 분석/대시보드 | Customer OS Core |

### 6.2 중요한 해석

예를 들어 `입금`은 Core에서 새로 계산하지 않는다.

- 소유: `CRM v2`
- Core 역할: CRM v2에서 결과를 복제해 통합 타임라인과 분석에 활용

이렇게 해야 회계성 로직 충돌을 줄일 수 있다.

## 8. 고객 매칭 규칙 초안

### 8.1 잠정 규칙

1. 휴대폰 번호 정확 일치
2. 유선전화 정확 일치
3. 메이크샵 회원 ID 일치
4. 수취인 이름 + 전화번호 일치
5. 입금자명 별칭 + 금액 후보 일치
6. 이름 + 주소 일부 일치

### 8.2 자동 병합 원칙

- `정확 일치`만 자동 병합
- `추정 일치`는 review queue
- 동명이인/가족 주문/학원 단체 주문은 자동 병합 금지

### 8.3 왜 보수적으로 가는가

고객 중복보다 더 위험한 건 `잘못된 병합`이다.

특히 PRESSCO21처럼 전화주문, 입금자명 별칭, 단체/대표 주문이 섞이면 이름만으로 병합하면 사고가 난다.

## 9. 수집 전략 상세

### 9.1 사방넷

현재 가장 현실적인 기본 원천이다.

이유:

- 쿠팡 API 제약을 피할 수 있다.
- 이미 주문/송장/재고 운영 허브 역할을 하고 있다.
- 여러 오픈마켓을 한 번에 수집한다.

초기 전략:

- 브라우저 자동화로 주문 export 다운로드
- 원본 파일 보관
- 매일 1회 전체 export
- 필요 시 낮 시간 1회 추가

검증 필요:

- 사방넷 export 컬럼으로 Customer OS 요구사항이 얼마나 충족되는지
- 상태 이력과 금액 정보가 충분한지
- 누락되는 고객관계 필드가 무엇인지

### 9.2 메이크샵

사방넷을 보완하는 핵심 원천이다.

왜 필요한가:

- 자사몰 회원 식별
- 회원성 데이터
- 적립금/회원상태
- 자사몰 상품/브랜드 기준
- 사방넷 export에 없는 자사몰 고객 맥락

초기 전략:

- Open API 우선
- 필요 시 관리자 export 보조

### 9.3 스마트스토어 / 11번가 / 쿠팡

초기 전략:

- 직접 API를 다 붙이지 않는다.
- 사방넷을 우선 원천으로 삼는다.
- 사방넷에서 부족한 필드만 선택적으로 추가 보강한다.

### 9.4 CRM v2

CRM v2는 `수집 대상`이기도 하다.

왜냐하면:

- 입금/미수/예치금은 CRM v2가 진실 원본이기 때문
- 전화주문도 CRM v2에서 직접 생성할 예정이기 때문

따라서 Core는 CRM v2에서 다음을 복제해야 한다.

- 고객별 미수 요약
- 예치금
- 환불대기
- 전화주문/명세표 주요 이벤트

### 9.5 상담툴

초기 전략:

- 상담 UI는 유지
- 대화 전문 전체 저장은 나중 단계
- 우선은 `이벤트 + 요약 + 고객 연결 키`만 적재

## 10. 왜 `CRM v2 중심 확장`을 폐기했는가

이건 Claude Code가 특히 검증해야 할 부분이다.

### 10.1 폐기 이유

1. `리스크`
   - 현재 잘 쓰는 회계성 운영 플로우가 깨질 수 있다.
2. `경계 혼탁`
   - CRM v2가 고객 허브, 회계 앱, 주문 허브, 상담 허브를 다 떠안게 된다.
3. `배포 부담`
   - 작은 수정이 전체 업무에 영향 준다.
4. `장기 교체 불가`
   - 나중에 허브와 실무 앱을 분리하기 더 어려워진다.

### 10.2 대안 구조의 장점

`Customer OS Core + CRM v2 연동`은 다음 장점이 있다.

- 현재 실무 흐름 보호
- 허브를 읽기 중심으로 먼저 구축 가능
- 나중에 일부 실행 기능만 점진 이동 가능
- 분석/고객관리/자동화와 회계 실무를 분리 가능

### 10.3 Claude Code가 봐야 할 질문

- CRM v2를 보호하면서도 deep link와 요약 카드만으로 충분한가
- Core가 먼저 읽기 포털로 시작하는 접근이 현실적인가
- CRM v2와 Core 사이 복제 주기/동기화 방식은 무엇이 적절한가

## 11. 단계별 실행안 상세

### Phase 0. 기준선 고정

목표:

- 시스템 경계 확정
- 기준 데이터 정의
- 핵심 지표 정의

산출물:

- PRD
- 로드맵
- 이 상세계획 문서
- 의사결정 리스트

### Phase 1. 사방넷 섀도우 런

목표:

- 사방넷 export 자동화가 가능한지
- 사방넷 데이터가 충분한지

해야 할 것:

- 다운로드 스크립트
- 원본 archive
- 메이크샵/사방넷 비교표
- 누락 컬럼 목록

### Phase 2. Customer OS Core 데이터층

목표:

- 기준 고객/주문/이벤트 테이블 생성

해야 할 것:

- raw + unified 테이블 생성
- 고객 매칭 규칙 엔진
- review queue
- CRM v2 요약 복제

### Phase 3. Customer OS Portal 1차

목표:

- 기존 시스템을 안 건드리고 통합 읽기 포털 제공

화면 후보:

- Customer 360
- Unified Orders
- Customer Timeline
- Channel Sales Dashboard

### Phase 4. CRM v2 브리지

목표:

- Portal에서 CRM v2로 자연스럽게 이어지게 한다.

예:

- 고객 화면에서 `명세표 보기`
- 고객 화면에서 `입금 처리`
- 고객 화면에서 `전화주문 생성`

### Phase 5. 전화주문 Desk

목표:

- 전화주문을 정식 시스템 흐름으로 만든다.

해야 할 것:

- 발신번호 조회
- 최근 주문 복사
- 상담/주문/입금 연결
- 후속 메시지 자동 발송

### Phase 6. 자동화/리텐션

목표:

- 고객 운영 서비스를 실제로 고도화한다.

후보:

- 결제 미완료 리마인드
- 리뷰 요청
- 재구매 주기 리마인드
- 휴면 고객 깨우기
- 품절/입고 알림

## 12. KPI 정의 초안

한 숫자로 다 보지 않는다.

### 12.1 최소 4개 공식 지표

- `주문총액`
  - 주문 생성 기준
- `실결제액`
  - 결제 완료 기준
- `순매출`
  - 취소/반품 차감 후 기준
- `실수금액`
  - 실제 입금/수금 반영 기준

### 12.2 이유

메이크샵, 사방넷, CRM v2가 서로 다른 타이밍과 정의를 갖기 때문에, 숫자 하나만 두면 오히려 계속 싸운다.

## 13. 주요 리스크

### 13.1 구조 리스크

- 허브와 CRM v2 경계가 흐려질 수 있다.
- Portal과 CRM v2의 역할이 중복될 수 있다.

### 13.2 데이터 리스크

- 사방넷 export 필드 부족
- 고객 잘못 병합
- 상태 이력 유실
- 주문/입금 타이밍 불일치

### 13.3 운영 리스크

- 브라우저 자동화 실패
- 로그인/캡차/화면변경 이슈
- 직원이 새 Portal을 안 쓰고 예전 화면만 쓸 가능성

### 13.4 범위 리스크

- 분석, 상담, 전화주문, 재고, 자동화를 한 번에 밀어넣을 위험

대응:

- Phase 게이트 유지
- 섀도우 런 선행
- CRM v2 직접 개조 최소화

## 14. 현재 잠정 확정 / 검증 필요 / 미결정

### 14.1 잠정 확정

- 구조는 `Customer OS Core + CRM v2 연동`
- 사방넷은 주문 운영 허브로 유지
- CRM v2는 실무 실행 앱으로 유지
- 메이크샵은 자사몰 보강 원천으로 유지
- 수집은 `사방넷 우선 + 메이크샵 보강`

### 14.2 검증 필요

- 사방넷 export의 실제 데이터 충분성
- 메이크샵 보강 필드 범위
- CRM v2에서 Core로 어떤 수준까지 복제할지
- Portal에서 직원이 실제로 판단이 빨라지는지

### 14.3 미결정

- Core의 실제 구현 스택
  - 순수 Postgres + 자체 API
  - Postgres + NocoDB 관리뷰
  - 별도 앱 프레임워크 선택
- Portal을 기존 CRM 리포와 같은 앱으로 할지, 별도 리포로 할지
- 상담 이벤트 적재 깊이
  - 요약만
  - 전문 일부
  - 전문 전체

## 15. Claude Code 진단 요청 포인트

다음 항목을 중점 진단해주면 좋다.

1. `시스템 경계`
   - Customer OS Core와 CRM v2 분리가 적절한가
2. `데이터 소유권`
   - 어떤 데이터는 절대 CRM v2 밖으로 빼면 안 되는가
3. `사방넷-first 전략`
   - 운영 현실과 데이터 손실 위험을 함께 봤을 때 타당한가
4. `Portal-first 접근`
   - 읽기 포털을 먼저 만드는 순서가 적절한가
5. `전화주문 Desk 위치`
   - CRM v2 안에 둘지, Core Portal 안에 둘지
6. `실행 단위`
   - 지금 계획이 너무 큰지, 더 쪼개야 하는지
7. `리포지토리 구조`
   - Core/Portal을 어디에 둘지

## 17. 한 줄 정리

PRESSCO21 고객운영 OS는 `CRM v2를 갈아엎는 프로젝트`가 아니라, `CRM v2를 포함한 여러 실행 앱 위에 Customer OS Core와 Portal을 얹어 고객 기준과 운영 지식을 내부 자산으로 만드는 프로젝트`다.
