## 현재 구조에서 잘 된 점

- 파트너 신청 -> 관리자 승인 -> 파트너 인증 -> 대시보드 진입의 핵심 골격이 live에서 돌아간다.
- MakeShop 페이지와 n8n/NocoDB 연동이 실제 운영 계정 기준으로 붙어 있다.
- 고객용 목록/상세 UX는 기본 정보 탐색과 FAQ, 지역 탐색까지는 usable 상태다.
- 파트너는 최소한 `승인 후 로그인 -> 대시보드 -> 강의 등록`까지 실제로 운영 가능한 상태다.
- `2610`을 공용 가이드로 열어두면서 이탈 가능성이 큰 필수 교육 gate 를 제거했다.

## 현재 문제점과 아쉬운 점

### 1. 고객 UX

- `2606 목록`은 검색/필터가 생겼지만, 여전히 "왜 이 클래스를 골라야 하는지"를 빠르게 비교하는 장치가 약하다.
- `2607 상세`는 정보 구조는 좋아졌지만, 일정이 없을 때 lead capture, 재입고/오픈 알림, 상담 전환 같은 적극적 회수 장치가 없다.
- 예약/선물하기/주문 이후의 full E2E 는 아직 충분히 닫지 못했다.

### 2. 파트너 UX

- `2608 대시보드`는 기본 뼈대는 됐지만, 초보 파트너 기준으로 "지금 뭘 해야 예약이 생기는지"를 더 명확히 안내할 필요가 있다.
- `8009 강의등록`은 helper 가 들어갔어도 입력 항목이 길고, 일정/키트/승인 후 운영까지의 mental model 이 아직 무겁다.
- `2610`은 공용 가이드가 되었지만, 회원/예비 파트너/승인 파트너 각각에게 같은 메시지를 보여주기 때문에 개인화가 약하다.

### 3. 관리자/운영 UX

- `8011 어드민`은 승인/조회는 되지만, 운영자가 대량 처리하기 좋은 정보 밀도와 감사 추적성은 아직 약하다.
- 테스트 데이터 정리 정책, 승인 후 follow-up, 예외 처리 규칙이 문서보다 UI에서 더 분명해야 한다.
- 관리자 운영을 MakeShop 안에 둘지, 별도 어드민으로 분리할지는 아직 전략적으로 닫히지 않았다.

### 4. 기술/운영 품질

- 등급체계는 한 번 이미 실제 장애를 냈다. 리뉴얼 시 저장 enum, 표시명, 수수료, 교육/가이드 문구를 동시에 바꾸는 운영 discipline 이 필요하다.
- 외부 스크립트 이슈가 남아 있다.
  - Kakao SDK integrity mismatch
  - Channel.io boot `401`
- `2608` 일부 보조 API(`예약 현황`, `후기 관리`)는 데이터가 없는 상태에서 콘솔 에러가 남아 있어 운영 신뢰감이 떨어진다.
- 일부 문서는 예전 등급/수수료 표현이 남아 있어, 전략 재수립 시 기준 문서를 다시 선별해야 한다.

## Claude Code가 이번에 해야 할 일

### 목표

- 현재 구현을 다시 뒤엎는 것이 아니라, `live에서 이미 되는 것`을 기준선으로 두고 다음 고도화 전략을 재정의한다.

### 기대 산출물

1. `As-Is 진단`
   - 고객, 파트너, 관리자, 운영 관점으로 현재 흐름의 강점/약점 정리
2. `고도화 전략`
   - Quick wins (1~2주)
   - Mid-term (3~6주)
   - Structural bets (분기 단위)
3. `우선순위 원칙`
   - 무엇을 먼저 고치고 무엇은 나중으로 미룰지
4. `페이지/플로우별 권고안`
   - `2606`, `2607`, `2608`, `8009`, `8011`, `2610`
5. `운영 구조 판단`
   - MakeShop 내부 운영 유지 vs 별도 어드민 분리

### 반드시 고려할 질문

- 고객은 목록/상세에서 "선택"을 더 쉽게 하려면 무엇이 부족한가
- 파트너는 승인 직후 첫 예약까지 가는 데 어디서 가장 많이 막히는가
- `2610` 공용 가이드를 유지할 때, 회원/예비 파트너/승인 파트너에게 어떤 분기 메시지가 필요할까
- 관리자에게 필요한 것은 더 많은 기능인가, 아니면 더 명확한 처리 흐름과 대량 작업 UX 인가
- 고도화의 1순위는 신규 기능인가, 현재 흐름의 마찰 제거인가

## 전략 수립 시 건드리면 안 되는 전제

- MakeShop D4 제약은 유지
- 기존 page id 는 유지
- `.secrets.env` 수정 금지
- live에서 이미 복구된 `WF-02/WF-08/WF-10/WF-13/WF-ADMIN` 을 근거 없이 되돌리지 말 것
- 등급체계가 바뀌면 `partner-grade-change-playbook.md` 와 `scripts/partnerclass-grade-change-audit.js` 를 먼저 볼 것
- `2610`은 별도 정책 변경이 있기 전까지 `로그인 회원 공용` 으로 본다

## Claude Code 시작 포인트 제안

1. `shared-service-identity.md` 와 `enterprise-elevation-strategy-2026-03-10.md` 로 방향 기준 재확인
2. 이 문서와 `AI_SYNC.md` 로 현재 live 기준선 파악
3. 현재 문제를 `고객 선택`, `파트너 첫 운영`, `관리자 운영 효율`, `운영 신뢰성` 4축으로 재분류
4. 그다음에만 새 전략 문서를 쓴다
