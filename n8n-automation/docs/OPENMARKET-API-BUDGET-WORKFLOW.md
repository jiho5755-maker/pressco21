# 오픈마켓 API 예산·스케줄 워크플로우

> 작성: 2026-04-17  
> 목적: 매출 자동화뿐 아니라 상품등록/수정, 품절·재고관리, 문의 수집·답변, 출고/클레임까지 포함해 API 호출 한도와 운영 충돌을 관리한다.  
> 전제: 정확한 공식 rate limit은 채널/계정/승인 범위마다 달라질 수 있으므로, 현재 문서는 **관측 기반 보수 예산**이다. 실제 한도 확인 후 숫자는 조정한다.

---

## 1. 핵심 결론

월마감 전수조사, 재고 동기화, 문의 수집, 상품등록/수정은 같은 채널 API 자원을 공유한다.  
따라서 매출 감사는 “남는 시간에 무제한으로 긁는 작업”이 아니라, **API 예산을 예약하고 실행하는 작업**이어야 한다.

운영 원칙:

```text
1. 실시간 운영성 API: 문의/주문/출고/재고를 우선
2. 매출 일일 증분: 가벼운 전일 수집으로 유지
3. 재고/품절 동기화: 주기적이되 짧고 제한적으로 실행
4. 상품등록/대량수정: 예약 시간대에 배치 처리
5. 월마감 전수조사: D+3/D+7에 저속·청크·오류보존 방식
6. 연마감: 월마감 확정값 합산, 의심월만 스팟체크
```

---

## 2. API 호출 우선순위

| 우선순위 | 작업 | 이유 | 예산 성격 |
|---|---|---|---|
| P0 | 출고/주문 상태, 긴급 재고/품절, 고객 문의 답변 | 운영 중단 방지 | 항상 예약 |
| P1 | 문의 수집, 재고 모니터링, 전일 매출 수집 | 일상 운영 | 매일 고정 |
| P2 | 주간 백필, 클레임/정산 보정 | 정확도 향상 | 저빈도 |
| P3 | 상품등록/상품수정/가격수정 대량 작업 | 비실시간 배치 | 예약 실행 |
| P4 | 월마감 전수조사 | 회계/분석 확정 | D+3/D+7 예약 |
| P5 | 연마감 재전수조사 | 매우 무거움 | 원칙상 금지, 스팟체크만 |

## 4. 채널별 예산안

### 4.1 MakeShop

확인된 기준:

- 조회 API: 약 500회/시간 기준으로 운영 메모 존재
- 처리 API: 약 500회/시간 기준으로 운영 메모 존재

예산안:

| 작업 | 권장 예산 |
|---|---:|
| 전일 매출 수집 | 1~3 calls/day |
| 상품 검색/상품조회 | 운영 중 100 calls/hour 이하 |
| 품절/재고 수정 | 처리 API 100 calls/hour 이하 |
| 문의/리뷰 수집/답변 | 운영 중 100 calls/hour 이하 |
| 월마감 감사 | 일자별 1 call, 총 31 calls 수준 |
| 대량 상품수정 | 별도 예약, 300 calls/hour 이하 |

운영 규칙:

- 처리 API는 조회 API보다 보수적으로 사용
- 대량 상품등록/수정은 업무시간 피크를 피한다
- F23/F26과 상품대량수정이 같은 시간에 돌지 않게 한다

### 4.2 Naver Commerce

관측:

- 로컬 IP: `GW.IP_NOT_ALLOWED`
- Oracle/n8n IP 사용 필요
- 일별 주문 조회 수십 회 연속 후 429 가능

예산안:

| 작업 | 권장 예산 |
|---|---:|
| 전일 매출 수집 | 1 call/day |
| 문의 수집 | 10~30분 주기, 소량 |
| 상품 문의 답변 | 승인형 실시간, 건별 |
| 재고/상품 수정 | 예약 배치 |
| 월마감 감사 | 14~16일 청크, 일자별 1.4초 이상 지연 |

운영 규칙:

- 전수조사와 문의 수집을 같은 시간대에 겹치지 않는다
- 429 발생 시 즉시 중단 후 실패일만 재시도
- 네이버 API는 Oracle/n8n에서만 실행

### 4.3 11번가

관측:

- `/complete` 단독 조회 금지
- 정상 매출은 복수 endpoint 필요
- 클레임 별도 조회

예산안:

| 작업 | 권장 예산 |
|---|---:|
| 전일 매출 수집 | 6개 order endpoint + claim 필요 시만 |
| 월마감 감사 | 7일 청크 × 6 order endpoint + claim endpoint |
| 상품/재고 조회 | 문서 기반 별도 예약 |
| 클레임 확인 | 주간 또는 월마감 시 |

운영 규칙:

- 7일 청크 고정
- `ordNo + ordPrdSeq` dedupe 필수
- 매출과 정산 API를 혼합하지 않는다
- 정산은 별도 회계 분석으로 분리

### 4.4 Coupang WING

관측:

- 로컬 IP: 403 `IP_NOT_ALLOWED`
- Oracle/n8n IP 사용 필요
- `status=ALL` 불가
- 상태별 호출 필요
- multi-status 전수조회는 429 다수
- n8n Code에서 `URLSearchParams` 사용 금지

예산안:

| 작업 | 권장 예산 |
|---|---:|
| 전일 매출 수집 | 가능하면 전일 `FINAL_DELIVERY` + 최근 미완료 상태 제한 조회 |
| 월마감 감사 | 전월 전체 `FINAL_DELIVERY` 저속 조회, 최근 7~10일만 다중 상태 |
| 출고/사방넷 연동 | 14:30~19:00 우선권 |
| 문의 수집/답변 | 매출 감사와 분리된 시간대 |
| 상품/재고/품절 | 예약 배치, 출고 시간 회피 |

운영 규칙:

- 14:30~19:00 KST 쿠팡 매출 API 금지
- 429 발생 시 기존값 보존
- 실패일/실패 상태만 retry queue
- 월마감 전수조사는 19시 이후 또는 새벽 시간대 권장

### 4.5 CRM/NocoDB

예산안:

- API 한도보다 DB 부하/응답시간을 더 주의
- 월마감은 단일 범위 조회로 처리
- 불필요한 반복 polling 금지

## 5. 워크플로우 분리안

### 5.1 유지할 일일 자동화

| WF | 역할 | 정책 |
|---|---|---|
| F23 통합 일일 매출 | 전일 API 수집 | 빠르고 가벼워야 함. 실패값 0 덮기 금지 |
| F24 일일 리치 리포트 | NocoDB/Sheets 기반 리포트 | API 직접 호출 금지 |
| F26 주간 정산 조정 | 최근 14일 백필 | 저빈도, diff 중심 |

### 5.2 추가해야 할 월마감 감사 WF

이름 제안:

```text
[F27] 월마감 매출 감사 Runner
```

역할:

1. 월 선택
2. 기존 Sheets/NocoDB snapshot 저장
3. 채널별 collector 실행
4. 오류일 기록
5. diff 생성
6. 반영 후보 생성
7. 월마감 감사 탭 업데이트
8. 승인 후 반영

### 5.3 추가해야 할 API 예산/큐 WF

이름 제안:

```text
[F28] 오픈마켓 API 예산 가드
```

역할:

- 채널별 호출량 기록
- 시간대별 실행 가능 여부 판단
- 쿠팡 휴지시간 차단
- 429 발생 시 retry queue 등록
- 월마감 감사 runner가 실행 전 예산 확인

## 7. 월마감 API 예산 계산 방식

각 월마감 실행 전 아래 계산을 한다.

```text
예상 호출량 = 채널별 날짜수 × endpoint 수 × 상태 수
가용 예산 = 관측 안전 호출량 - 일상 운영 예약량 - 버퍼
실행 가능 여부 = 예상 호출량 <= 가용 예산
```

예시: 31일 월 기준

| 채널 | 단순 전수 예상 | 권장 최적화 |
|---|---:|---:|
| MakeShop | 31 calls | 31 calls |
| Naver | 31 calls | 31 calls + throttle |
| 11번가 | 31×6=186 calls + claims | 7일 청크×6 + claims |
| Coupang | 31×6=186 calls | 31×FINAL_DELIVERY + 최근 7~10일×추가상태 |
| CRM | 1~2 calls | 1~2 calls |

쿠팡/네이버는 한도와 429 위험이 커서 반드시 버퍼를 둔다.

권장 버퍼:

```text
운영성 API 50%
월마감 감사 30%
재시도/비상 20%
```

## 8. 실행 금지 조건

아래 조건에서는 월마감 전수조사를 실행하지 않는다.

- 쿠팡 14:30~19:00 KST
- 네이버/쿠팡 인증 오류 발생 중
- F23/F26 등 기존 매출 WF가 실행 중
- 상품 대량 등록/수정 배치가 실행 중
- 출고/사방넷 연동 작업 중
- API retry queue가 과도하게 쌓인 상태

## 11. 메이크샵 개인결제건 처리 정책

메이크샵 개인결제 링크는 CRM/전화·방문 주문의 카드 결제 수단이다. 매출 분석에서는 CRM/직접거래로만 집계하고 메이크샵 매출에서는 제외한다.

식별 기준:

```text
product_cate1 = 097
brandcode starts with 097001
```

보조 패턴:

```text
개인결제 / 개별결제 / 개인결재 / 개별결재
```

F23/F26은 위 기준으로 개인결제건을 메이크샵 매출에서 제외한다. 월마감 감사 탭의 `개인결제 제외` 컬럼에서 월별 검증 상태를 관리한다.
