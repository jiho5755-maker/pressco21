# P1 상세 기능명세: 예치금/환불대기/초과 입금 처리

> Date: 2026-03-11  
> Scope: offline-crm-v2  
> Based on: `accounting-upgrade-prd-v2-2026-03-11.md`

## 1. 목적

P1은 아래 5개를 구현하는 단계다.

1. 초과 입금 처리
2. 예치금
3. 환불대기
4. 명세표 예치금 사용
5. 거래원장 유형 확장

## 2. 이번 단계에서 바꾸는 것

### 유지

- 기존 고객/명세표/거래원장/수금 관리/지급 관리 기능 유지
- 기존 장부 원본 복원/조회 기능 유지
- 기존 수금/지급 이력 로그 유지

### 추가

- 입금 시 초과 금액 분기
- 고객 단위 예치금 잔액
- 고객 단위 환불대기 잔액
- 명세표 작성 시 예치금 사용
- 거래원장 유형 추가

## 3. 기능 1: 초과 입금 처리

### 트리거

- 입금 확인 시 입력 금액이 현재 받을 돈보다 클 때

### 동작

시스템은 저장 전에 선택 팝업을 띄운다.

표시값:

- 현재 받을 돈
- 실제 입금액
- 초과 입금액

선택지:

1. `받을 돈까지만 처리하고 나머지는 예치금으로 보관`
2. `받을 돈까지만 처리하고 나머지는 환불대기로 등록`
3. `전액 예치금으로 처리`
4. `취소`

### 결과

- 선택 1:
  - 받을 돈 0 처리
  - 초과 금액은 예치금 증가
  - 거래원장에 `입금`, `예치금 적립` 기록
- 선택 2:
  - 받을 돈 0 처리
  - 초과 금액은 환불대기 증가
  - 거래원장에 `입금`, `환불대기 생성` 기록
- 선택 3:
  - 받을 돈 차감 없음
  - 입금액 전액을 예치금으로 적립
  - 거래원장에 `예치금 적립` 기록

## 4. 기능 2: 예치금

### 정의

- 고객이 미리 더 낸 돈
- 다음 거래에서 차감 가능

### 저장 항목

- 고객별 현재 예치금 잔액
- 예치금 이력
  - 적립
  - 사용
  - 취소

### 표시 위치

- 고객 상세 카드
- 수금 관리 탭
- 거래원장

### 예치금 유형

- 일반 예치금
- 정부지원 예치금

P1에서는 유형 선택까지는 선택 옵션으로 두고, 최소한 메모에 유형을 남길 수 있어야 한다.

## 5. 기능 3: 환불대기

### 정의

- 고객에게 돌려주기로 확정했지만 아직 송금하지 않은 금액

### 생성 경로

- 초과 입금 후 환불 선택
- 수기 환불대기 등록

### 상태

- 대기
- 환불 완료
- 취소

### 표시 위치

- 고객 상세 카드
- 지급 관리 탭
- 거래원장

## 6. 기능 4: 명세표 예치금 사용

### 트리거

- 명세표 작성 시 연결 고객에게 예치금 잔액이 있을 때

### 동작

명세표 화면에 안내 표시:

- `사용 가능한 예치금 000,000원`

선택지:

- 사용 안 함
- 전액 사용
- 일부 사용

### 결과

- 명세표 총액에서 차감
- 예치금 잔액 감소
- 거래원장에 `예치금 사용` 기록

### 유효성

- 예치금 잔액보다 많이 사용할 수 없음
- 명세표 총액보다 많이 사용할 수 없음

## 7. 기능 5: 거래원장 유형 확장

P1에서 아래 유형을 공식 추가한다.

- 출고
- 입금
- 지급
- 예치금 적립
- 예치금 사용
- 환불
- 반입
- 취소

### 필수 표시값

- 날짜
- 거래처
- 유형
- 금액
- 관련 문서번호 또는 기준 id
- 메모
- 처리 계정

## 9. 화면별 변경 사항

### 수금 관리

- `예치금` 탭 추가
- 초과 입금 처리 팝업 추가
- 예치금 적립 이력 보기

### 지급 관리

- `환불대기` 탭 추가
- 환불 완료 처리 추가

### 고객 상세

- 정산 카드에 `예치금`, `환불대기` 추가
- 기존 장부 원본 탭과 별도로 현재 운영 정산 상태 표시 강화

### 명세표 작성

- 고객 예치금 안내 배지
- 예치금 사용 선택 UI

### 거래원장

- 새 유형 필터 추가
- 예치금/환불 행 상세 보기 추가

## 10. 유효성 규칙

- 받을 돈보다 많은 입금은 초과 입금 분기 없이 저장 불가
- 줄 돈보다 많은 지급 저장 불가
- 예치금 사용은 잔액 범위 내에서만 가능
- 환불 완료는 환불대기 범위 내에서만 가능
- 취소는 원래 이벤트를 지우지 않고 취소 이벤트 또는 취소 상태로 남김

## 11. UX 상세 원칙

- 팝업에서 계산 결과를 즉시 보여준다
- 저장 전 `처리 후 결과`를 보여준다
- 직원이 회계 공식을 몰라도 결과를 이해할 수 있어야 한다
- 모든 화면에서 현재 작업 계정을 본문 안에서 한 번 더 보여준다

## 12. 검증 기준

### 시나리오 1. 정상 입금

- 받을 돈 100,000원
- 입금 100,000원
- 결과: 받을 돈 0

### 시나리오 2. 초과 입금 → 예치금

- 받을 돈 100,000원
- 입금 150,000원
- 결과: 받을 돈 0, 예치금 50,000원

### 시나리오 3. 초과 입금 → 환불대기

- 받을 돈 100,000원
- 입금 150,000원
- 결과: 받을 돈 0, 환불대기 50,000원

### 시나리오 4. 예치금 사용

- 예치금 80,000원
- 명세표 50,000원
- 결과: 예치금 30,000원, 거래원장에 예치금 사용 기록

### 시나리오 5. 환불 완료

- 환불대기 50,000원
- 환불 완료 50,000원
- 결과: 환불대기 0, 거래원장에 환불 기록

## 13. 개발 순서

1. 데이터 구조 추가
2. 초과 입금 처리 팝업
3. 예치금/환불대기 잔액 계산
4. 고객 상세 카드 확장
5. 거래원장 유형 확장
6. 명세표 예치금 사용 연결
7. 탭/보고서 추가

## 14. 완료 기준

- 초과 입금이 예치금/환불대기로 분기 가능
- 고객 상세에서 예치금과 환불대기 확인 가능
- 명세표 작성 시 예치금 사용 가능
- 거래원장에 관련 이력이 남음
- 기존 기능이 깨지지 않음
