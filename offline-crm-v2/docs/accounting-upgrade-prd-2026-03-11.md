# PRD: PRESSCO21 회계 운영 프로그램 고도화

> Version 1.0  
> Date: 2026-03-11  
> Status: Draft for implementation  
> Scope: offline-crm-v2

## 1. 배경

현재 CRM은 `기존 장부 백업 복원`, `고객/거래 조회`, `미수금 관리`, `기존 장부 수금 이력`까지는 실무 사용 가능한 수준으로 올라왔다.  
하지만 실제 현장에서는 아래 3가지가 동시에 필요하다.

1. `제대로 된 회계 운영 프로그램`
2. `직원이 쉽게 배우는 사용 가이드`
3. `원장님 농협 계좌 입금 시 자동 수금 처리`

지금 상태는 1번의 초입에 있고, 2번과 3번은 아직 구조 설계가 없다.  
따라서 이번 고도화의 목표는 CRM을 `조회 중심 프로그램`에서 `수금/지급/원장/자동화 중심 회계 운영 프로그램`으로 올리는 것이다.

## 2. 제품 목표

### Goal 1. 회계 운영 프로그램으로 고도화
- `받을 돈`과 `줄 돈`을 별도 업무 흐름으로 분리한다.
- 고객/공급처별 원장 흐름과 현재 잔액을 신뢰 가능하게 만든다.
- 수금/지급 처리 이력과 취소 이력을 모두 남긴다.

### Goal 2. 직원 사용 가이드 투어
- 비전공자 직원이 처음 들어와도 `무엇을 눌러야 하는지` 이해할 수 있게 한다.
- 화면 첫 진입 시 기능별 설명, 단계별 안내, 용어 설명을 제공한다.
- 실수하기 쉬운 수금/지급/명세표 처리 흐름을 실제 업무 순서대로 안내한다.

### Goal 3. 농협 계좌 입금 자동 반영
- 원장님 농협 계좌 입금 내역을 읽어 고객/거래처와 매칭한다.
- 후보 매칭 결과를 검토 후 승인하거나 자동 승인 규칙을 적용한다.
- 승인된 입금은 회계프로그램 내 수금 처리, 미수금 차감, 거래내역 기록까지 연결한다.

## 3. 핵심 사용자

### 3.1 실무 직원
- 고객 응대, 명세표 작성, 수금 확인, 지급 확인을 담당
- 회계 용어보다 `받을 돈`, `줄 돈`, `입금`, `송금`처럼 쉬운 표현을 선호
- 실수 후 복구 가능성과 로그 확인을 중요하게 생각함

### 3.2 원장/관리자
- 전체 미수/미지급 현황, 회기별 잔액, 담당자 처리 이력을 보고 싶어함
- 은행 입금 자동화와 예외 승인 흐름을 원함
- 특정 고객/공급처의 돈 흐름이 왜 그렇게 계산됐는지 바로 이해하고 싶어함

## 4. 문제 정의

### 4.1 회계 처리 구조가 절반만 완성됨
- 수금 관리는 가능하지만 지급 처리는 조회 위주다.
- 미수금/미지급금은 화면에 보이지만 실제 처리 흐름은 비대칭이다.
- 거래내역에 `수금`, `지급`, `조정`, `취소`가 회계 관점으로 충분히 구조화되어 있지 않다.

### 4.2 비전공자 기준 학습 비용이 아직 높음
- 화면마다 기준 용어를 해석해야 한다.
- 신규 직원은 어떤 순서로 써야 하는지 바로 알기 어렵다.
- 실수하기 쉬운 포인트를 미리 막는 안내가 부족하다.

### 4.3 수금 자동화가 수동 업무에 묶여 있음
- 입금 확인을 사람이 직접 보고 눌러야 한다.
- 계좌 입금과 고객 미수금이 자동 연결되지 않는다.
- 반복 입금/부분 입금/동명이인 입금 같은 예외 처리 규칙이 없다.

## 5. 용어 원칙

직원 혼선을 줄이기 위해 아래 용어를 기본으로 사용한다.

- `미수금` 대신: `받을 돈`
- `미지급금` 대신: `줄 돈`
- `레거시` 대신: `기존 장부`
- `CRM` 대신: `새 입력`
- `baseline` 대신: `이월 잔액`
- `split customer` 대신: `별도 거래처`

관리/보고서에서는 회계 용어를 병기할 수 있으나, 기본 UI는 쉬운 표현을 우선한다.

## 6. 제품 원칙

1. `숫자보다 흐름`
- 현재 잔액만 보여주지 말고, 왜 그 금액이 되었는지 함께 보여준다.

2. `삭제보다 이력`
- 수금/지급/취소는 지워지지 않고 기록으로 남아야 한다.

3. `업무 단위 분리`
- 고객 쪽 받을 돈과 공급처 쪽 줄 돈은 서로 다른 화면과 액션으로 관리한다.

4. `자동화는 승인 가능한 구조`
- 자동 입금 반영은 바로 확정하지 않고, 검토 가능성과 예외 처리를 남긴다.

5. `비전공자 우선`
- 회계 공식보다 사용 흐름이 먼저 이해되어야 한다.

## 7. 상세 요구사항

### 7.1 회계 운영 고도화

#### A. 수금 관리
- 고객별 현재 받을 돈 조회
- 부분 수금, 완납 처리, 취소 처리
- 수금 방법, 처리자, 처리 시각, 메모 기록
- 고객 상세와 거래원장에 수금 이력 노출
- 기준일별 받을 돈 조회

#### B. 지급 관리
- 공급처별 현재 줄 돈 조회
- 부분 지급, 완납 처리, 취소 처리
- 지급 방법, 처리자, 처리 시각, 메모 기록
- 공급처 상세와 거래원장에 지급 이력 노출
- 기준일별 줄 돈 조회

#### C. 거래원장
- 출고
- 입금
- 지급
- 반입/환불
- 수기 조정
- 취소

를 시간순으로 모두 볼 수 있어야 한다.

#### D. 정산 카드
고객/공급처 상세에는 최소 아래 항목이 보여야 한다.

- 이월 잔액
- 이번 회기 매출 또는 매입
- 이번 회기 수금 또는 지급
- 현재 잔액
- 마지막 처리일
- 최근 처리자

#### E. 보고서
- 고객별 받을 돈 현황표
- 공급처별 줄 돈 현황표
- 회기별 정산표
- 기간별 거래원장 엑셀/인쇄
- 담당자별 처리 이력

### 7.2 직원 사용 가이드 투어

#### A. 첫 진입 온보딩
- 주요 메뉴 소개
- 수금/지급/거래원장 차이 설명
- 실제 업무 순서 안내

#### B. 화면별 가이드
- 고객관리: 고객 찾기, 거래원장 보기, 받을 돈 확인
- 명세표 작성: 거래처 선택, 품목 입력, 명세표 발행
- 수금 관리: 입금 확인, 부분 수금, 취소
- 지급 관리: 지급 확인, 부분 지급, 취소

#### C. 실수 방지
- 잘못 누르기 쉬운 버튼은 설명 툴팁 제공
- 초과 금액 입력 방지
- 취소 시 영향 범위 안내
- 현재 작업 계정 표시

#### D. 도움말 자산
- 첫 사용 투어
- 다시 보기 투어
- 화면 우측 도움말 패널
- 자주 묻는 질문

### 7.3 농협 계좌 입금 자동화

#### A. 입금 수집
- 원장님 농협 계좌 입금 내역을 정기적으로 수집
- 최소 수집 항목:
  - 입금일시
  - 입금자명
  - 입금액
  - 적요/메모
  - 계좌번호

#### B. 매칭 엔진
- 고객명 유사도
- 입금액 일치
- 미수 잔액 일치
- 최근 명세표 존재 여부
- 과거 반복 입금 패턴

기준으로 자동 후보를 제안한다.

#### C. 처리 정책
- 자동 확정
  - 1:1 명확 매칭
  - 금액 정확 일치
  - 중복 후보 없음
- 검토 필요
  - 동명이인
  - 부분 입금
  - 초과 입금
  - 여러 명세표에 나눠 적용 필요

#### D. 반영 결과
- 수금 이력 생성
- 고객 받을 돈 차감
- 거래원장에 입금 행 추가
- 처리 계정: `자동화`
- 검토 승인자는 별도 기록

## 8. 비기능 요구사항

### 8.1 감사 추적
- 모든 수금/지급/취소/자동반영은 로그를 남긴다.
- 로그 필드:
  - 유형
  - 금액
  - 전/후 잔액
  - 처리자 계정
  - 처리 시각
  - 취소 여부
  - 원천(`기존 장부`, `새 입력`, `자동 입금`)

### 8.2 안정성
- 월마감/회기마감 후에는 일반 수정 제한
- 자동화 실패 시 수동 검토 대기열로 이동
- 잘못 반영된 자동 수금은 되돌릴 수 있어야 함

### 8.3 성능
- 수금/지급 메인 화면 첫 로드 3초 이내
- 거래원장 검색 응답 2초 이내
- 자동 입금 매칭 배치 처리 1분 이내

## 9. 범위 제외

이번 단계에서 바로 하지 않는 것:

- 정식 복식부기 분개장 전체 구현
- 국세청/전자세금계산서 직접 연동
- 은행 다계좌 통합
- 모바일 앱 별도 개발
- 급여/재고/발주 전체 회계 모듈

## 10. 성공 기준

### Goal 1 성공 기준
- 받을 돈/줄 돈을 따로 처리할 수 있음
- 고객/공급처별 현재 잔액과 거래 흐름이 맞음
- 수금/지급 취소 이력이 남음
- 회기 기준 보고서가 수기 검산과 일치

### Goal 2 성공 기준
- 신규 직원이 15분 안에 핵심 업무 흐름을 이해함
- 단순 문의성 질문이 감소함
- 잘못된 수금/지급 입력이 줄어듦

### Goal 3 성공 기준
- 농협 입금의 상당수를 자동 후보 매칭 가능
- 명확 매칭 건은 자동 반영 가능
- 애매한 건은 검토 큐에서 처리 가능
- 자동 반영 후 받을 돈 잔액과 거래원장이 일치

## 11. 출시 전략

### Phase A. 기반 정리
- 받을 돈/줄 돈 구조 분리
- 지급 처리 기능 추가
- 원장/로그 구조 통일

### Phase B. 사용성 고도화
- 쉬운 용어로 전환
- 투어/도움말 도입
- 실수 방지 UX 보강

### Phase C. 자동화
- 농협 입금 수집
- 자동 매칭
- 자동 반영/검토 승인

## 12. 의사결정 메모

- 현재 복원 기준 데이터는 `기존 장부 최신본`을 진실 원본으로 사용한다.
- 기존 `레거시/CRM` 구분은 내부 계산에는 유지하되, 사용자 화면에서는 쉬운 용어로 점진 교체한다.
- 자동 수금은 반드시 `검토 가능한 구조`로 시작하고, 100% 완전 자동보다 `실수 없는 반자동`을 우선한다.
