# PRESSCO21 CRM AI 표준 개발 워크플로우

이 문서는 "AI가 빨리 고친다"보다 한 단계 더 나아간 운영 기준입니다.

목표는 3가지입니다.

1. 잘 되던 기능을 실수로 망가뜨리지 않기
2. 비전공자도 지금 무슨 변경이 들어가는지 이해할 수 있게 하기
3. 문제가 생겨도 빠르게 원인 파악과 복구가 가능하게 하기

## 한 줄 요약

작업은 항상 이 순서로 진행합니다.

`작업 브리프 작성 -> 기존 정상 동작 고정 -> 작은 범위만 수정 -> 자동 검증 -> 안전 배포 -> 운영 스모크 확인`

## 왜 이렇게 하나요?

사람 개발자는 머릿속에 기존 구조를 오래 유지할 수 있습니다.
AI는 속도는 빠르지만, 여러 파일에 흩어진 규칙을 한 번에 다 기억한 채 수정하는 데 약합니다.

그래서 AI에게는 아래 3개를 더 강하게 적용해야 합니다.

- 변경 범위 제한
- 기존 정상 동작을 테스트로 먼저 고정
- 배포를 "덮어쓰기"가 아니라 "릴리스 승격"으로 처리

## 실제 작업 순서

### 1. 작업 브리프 작성

개발 시작 전에 짧은 브리프를 먼저 남깁니다.

브리프에는 아래 6개만 씁니다.

- 목표
- 영향 화면
- 수정 파일
- 건드리면 안 되는 기존 동작
- 검증 방법
- 배포 여부

템플릿은 [ai-task-brief.md](/Users/jangjiho/workspace/pressco21/offline-crm-v2/docs/templates/ai-task-brief.md)를 씁니다.

### 2. 기존 정상 동작 먼저 고정

"지금 잘 되는 것"을 먼저 테스트나 체크리스트로 고정합니다.

예시:

- 고객 상세 -> 명세표 작성 시 현재 고객이 기본 선택되어야 함
- 고객 주소 라벨이 상세/명세표에서 동일하게 보여야 함
- 품목 자동완성 선택 시 고객 단가는 입력되고, 과세 상태는 유지되어야 함
- 명세표 목록 우측에 `명세표`, `견적서` 버튼이 보여야 함

즉, 먼저 "지켜야 할 것"을 명확히 만들고 나서 수정합니다.

### 3. 수정은 작은 단위로만

한 번에 여러 층을 같이 바꾸지 않습니다.

좋은 예:

- 인쇄 레이아웃만 수정
- 명세표 기본값만 수정
- 고객 상세 프리필만 수정

나쁜 예:

- 인쇄 + 저장 + 목록 + 필터 + 가이드까지 한 번에 수정

### 4. 공통 규칙은 한 군데만

기본값, 라벨, 문서 구분 같은 공통 규칙은 여러 파일에 나누지 않습니다.

예:

- `src/lib/invoiceDefaults.ts`

이렇게 해야 AI가 한 군데만 고치고 다른 곳은 놓치는 사고를 줄일 수 있습니다.

### 5. 자동 검증

최소 기준은 아래입니다.

- `npm run test:regression`
- `npm run build`

회귀 검증은 "배포 전에 반드시 통과해야 하는 문"입니다.

### 6. 릴리스형 배포

운영 서버에 바로 덮어쓰지 않습니다.

배포는 이렇게 합니다.

1. 새 릴리스 폴더 생성
2. 새 빌드 업로드
3. `current` 링크를 새 릴리스로 변경
4. nginx 확인
5. 운영 스모크 체크

이 방식의 장점:

- 이전 버전이 남아 있음
- 문제가 생기면 링크만 이전 릴리스로 되돌리면 됨
- 어떤 버전이 언제 올라갔는지 추적 가능

실행 명령:

- 배포: `npm run deploy:release`
- 롤백: `npm run deploy:rollback -- <release-id>`

## 이 저장소의 기본 체크 세트

### 회귀 테스트

- `npm run test:regression`

포함 범위:

- 고객 검색
- 고객 상세 이동
- 고객 상세 -> 명세표 프리필
- 명세표 기본 날짜/기본 구분
- 목록 우측 `명세표` / `견적서`
- 품목 자동완성 단가/과세 유지

### 빌드 확인

- `npm run build`

### 운영 스모크 체크

배포 후 최소 5개 확인:

1. `/invoices` 진입
2. 오늘 날짜 기본값 확인
3. 고객 상세 -> 명세표 작성 프리필 확인
4. 품목 자동완성 단가/과세 확인
5. 목록 우측 `명세표` / `견적서` 버튼 확인

## AI 전용 추가 규칙

사람 개발자보다 AI에게 더 강하게 적용하는 규칙입니다.

### 변경 예산

한 작업에서 수정 파일 수를 제한합니다.

권장:

- 화면 수정: 1~3개 파일
- 로직 수정: 2~4개 파일
- 공통 규칙 정리: 3~5개 파일

### 금지 범위

작업 브리프에 없는 영역은 건드리지 않습니다.

예:

- 견적서 레이아웃 수정 시 저장 로직 수정 금지
- 고객 상세 UX 수정 시 인쇄 로직 수정 금지

### 증거 의무

작업이 끝나면 아래 중 최소 2개를 남깁니다.

- 테스트 통과 결과
- 스크린샷
- 운영 확인 결과
- 배포 릴리스 ID

### 테스트 데이터 자동 정리

운영/개발 데이터 오염을 막기 위해, E2E가 만든 테스트 명세표는 자동 삭제합니다.

현재 prefix:

- `TEST-E2E-PLAYWRIGHT-`

## 비전공자용 해석

쉽게 말하면 이 문서는 "AI가 마음대로 고치지 못하게 하는 안전장치"입니다.

당신이 봐야 할 핵심은 4개뿐입니다.

- 이번 작업 목표가 무엇인지
- 기존에 지켜야 하는 동작이 무엇인지
- 테스트가 통과했는지
- 운영 배포 후 최소 확인이 끝났는지

즉, 코드를 읽지 않아도 "이번 변경이 안전한지"를 판단할 수 있게 만드는 구조입니다.

## 권장 운영 규칙

### 작은 수정

오늘 바꿀 것만 바꿉니다.

### 검증 후 배포

작업 -> 테스트 -> 빌드 -> 배포 순서를 반드시 지킵니다.

### 배포는 릴리스 단위

문제 생기면 바로 이전 릴리스로 돌아갈 수 있어야 합니다.

### 기준 문서 유지

작업 브리프, 회귀 테스트, 배포 로그가 남아야 다음 AI도 안정적으로 이어받을 수 있습니다.
