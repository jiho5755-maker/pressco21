# 자동입금 자동화 개발 플랜

> Date: 2026-03-12  
> Scope: offline-crm-v2  
> Goal: `엑셀은 배치 보조`, `평소 운영은 자동 인식`, `입금자명 별칭까지 반영`

## 1. 목표

자동입금 기능의 최종 목표는 아래 3개다.

1. 평소 운영에서는 농협 입금 데이터가 들어오면 프로그램이 자동으로 읽는다.
2. 입금자명과 금액을 기준으로 고객을 자동 인식하고, 받을 돈을 자동 차감한다.
3. 고객명과 실제 입금자명이 다를 때도 `입금자명 별칭` 설정으로 자동 처리한다.

엑셀/CSV 업로드는 이 자동화를 대체하는 기능이 아니라, 아래 용도로 유지한다.

- 은행 원본을 한 번에 반영해야 할 때
- 누락분을 재수집할 때
- 초기 정리나 대량 정산을 할 때

## 2. 현재 구현 상태

현재 코드 기준으로 이미 들어간 것:

- `입금 수집함` 페이지
- CSV/XLSX 업로드 파싱 (`.xls`는 보안상 `.xlsx`로 변환 후 업로드)
- 입금자명 + 금액 기준 후보 매칭
- 수동 반영
- 설정 기반 `정확 일치 자동반영` 준비
- 고객별 `입금자명 별칭` 저장 구조

즉 현재는 `배치 업로드 + 자동 후보 매칭 + 수동/조건부 반영` 단계까지 와 있다.

아직 없는 것:

- 농협 계좌에서 실시간 또는 주기적으로 입금 데이터 자동 수집
- 백그라운드 자동 실행
- 자동반영 전용 로그/감사 화면
- 자동반영 실패 건 전용 운영 큐

## 3. 이 플랜이 실현 가능한가

결론: **실현 가능하다.**

다만 `자동반영 엔진`과 `은행 데이터 수집원`은 분리해서 봐야 한다.

### 3.1 이미 실현 가능한 영역

현재 코드 구조만으로도 가능한 것:

- 입금 파일 자동 파싱
- 고객명/장부명/입금자명 별칭 기반 매칭
- 정확 일치 건 자동반영
- 초과 입금 시 예치금/환불대기 분기
- 반영 결과를 수금 관리, 고객 상세, 거래원장에 동시 반영

### 3.2 외부 입력원이 있어야 가능한 영역

`평소 운영 자동화`는 결국 입금 데이터가 자동으로 들어와야 한다.
이 부분은 아래 중 하나가 필요하다.

1. 농협 API 또는 오픈뱅킹 연동
2. 농협 내역 파일이 특정 위치에 자동 저장되는 방식
3. 이메일 첨부파일 자동 수집
4. 계좌내역 CSV/XLSX를 정기 업로드하는 반자동 운영

즉 자동반영 로직 자체는 개발 가능하지만, `완전 자동 수집`은 은행 데이터 유입 방식이 하나 정해져야 끝까지 닫힌다.

## 4. 핵심 운영 원칙

자동입금은 편하면 안 되고, **틀리지 않게** 돌아가야 한다.

원칙은 아래로 고정한다.

1. `정확 일치`만 자동반영한다.
2. 애매한 건은 무조건 `검토 필요`로 보낸다.
3. 자동반영 후에도 사람이 거래원장에서 이유를 추적할 수 있어야 한다.
4. 자동반영으로 잘못 들어간 건 취소 가능해야 한다.
5. 엑셀 업로드는 `.xlsx`와 `.csv`만 지원하고, 평소 운영의 주 흐름은 자동 수집으로 옮긴다.

## 5. 매칭 기준

자동입금 매칭은 아래 기준으로만 확정한다.

### 5.1 자동반영 조건

아래가 모두 맞으면 자동반영 가능:

1. 입금액이 현재 받을 돈 또는 미수 명세표 잔액과 정확히 같을 것
2. 입금자명이 아래 중 하나와 맞을 것
   - 고객명
   - 장부명
   - 고객별 입금자명 별칭
3. 같은 금액/같은 이름 후보가 1건만 나올 것
4. 비활성/거래종료 고객이 아닐 것

### 5.2 검토 필요 조건

아래는 자동반영 금지:

- 같은 금액 후보가 여러 건
- 입금자명은 맞지만 금액이 다름
- 부분 입금
- 초과 입금
- 입금자명이 전혀 다르지만 note만 비슷함
- 거래종료 고객

### 5.3 별칭 규칙

고객별로 여러 개의 입금자명 별칭을 저장할 수 있다.

예:

- `김순자`
- `정부지원금`
- `서상견(단양)`
- `권금희 전주`

매칭 시 아래 값들을 모두 비교 대상으로 본다.

- `customer.name`
- `customer.book_name`
- `customer memo` 안의 `depositorAliases`
- 기존 장부 별칭

## 6. 자동입금 UX

### 6.1 직원이 보는 기본 흐름

1. 자동입금 수집함이 새 내역을 읽는다.
2. 정확한 건은 `자동반영 완료`
3. 애매한 건은 `검토 필요`
4. 직원은 `검토 필요`만 확인해서 처리

즉 직원은 모든 입금을 손으로 처리하는 게 아니라, **예외만 보는 구조**로 바뀌어야 한다.

### 6.2 검토 필요 화면에서 필요한 액션

- 고객 지정
- 명세표 지정
- 기존 장부 받을 돈에 반영
- 예치금으로 처리
- 환불대기로 처리
- 무시

### 6.3 고객 상세에서 필요한 설정

고객 상세 기본정보에 아래 필드가 필요하다.

- `입금자명 별칭`
- `자동입금 우선 고객` 여부
- `자동입금 제외` 여부

이유:

- 이름이 달라도 자동 매칭 가능
- 같은 이름이 많은 고객은 우선순위 제어 가능
- 거래종료/예외 고객은 자동반영 제외 가능

## 7. 데이터 구조

### 7.1 고객 메타

현재 `memo` 안의 숨김 메타 구조에 이미 들어가거나 들어갈 값:

- `depositBalance`
- `refundPendingBalance`
- `depositorAliases`
- 정산 이벤트들

향후 추가 권장 필드:

- `autoDepositDisabled: boolean`
- `autoDepositPriority: number`
- `lastAutoDepositMatchedAt: string`

### 7.2 입금 수집 레코드

수집함 한 건은 최소 아래 필드를 가진다.

- `date`
- `sender`
- `amount`
- `note`
- `sourceFile` 또는 `sourceType`
- `status`
  - `pending`
  - `applied`
  - `review`
  - `ignored`
- `appliedTargetKey`
- `appliedAt`

### 7.3 자동반영 감사 로그

자동반영이 실제 운영으로 들어가면 별도 로그가 필요하다.

- `source`
  - `bank_feed`
  - `csv_upload`
- `sender`
- `amount`
- `matchedCustomerId`
- `matchedInvoiceId`
- `matchingReason`
- `appliedBy`
  - `automation`
  - `manual`
- `appliedAt`
- `revertedAt`

## 8. 구현 단계

### Phase 1. 별칭 기반 자동반영 안정화

목표:

- 고객별 입금자명 별칭 저장
- 업로드 시 별칭 포함 자동매칭
- `정확 일치 자동반영` 설정이 켜져 있으면 즉시 자동반영

산출물:

- 고객 상세 `입금자명 별칭`
- 수집함 자동반영
- 자동반영 결과가 수금 관리/거래원장/고객 상세에 동기화

판정:

- 이 단계는 지금 코드 기반으로 바로 실현 가능

### Phase 2. 검토 필요 운영 큐

목표:

- 자동반영 못 한 건을 직원이 빠르게 정리

필수 기능:

- 고객 직접 지정
- 명세표 직접 지정
- 부분 입금 반영
- 초과 입금 분기
- 예치금/환불대기 선택
- 반영 이력 다운로드

판정:

- 이것도 현재 프론트 + 기존 API로 충분히 가능

### Phase 3. 주기 수집 자동화

목표:

- 파일 업로드 없이 입금 데이터를 주기적으로 끌어오기

가능한 구현 방식:

1. `은행 API/오픈뱅킹`
   - 가장 이상적
   - 완전 자동화 가능
2. `이메일 첨부 수집`
   - 현실적 대안
   - 농협 내역 파일이 메일로 오면 n8n이 저장 후 CRM 반영
3. `공유 폴더/서버 업로드 감시`
   - 사람이 파일만 넣으면 자동 처리
4. `정기 수동 업로드`
   - 가장 약하지만 즉시 가능

권장 시작:

- 3 또는 4로 먼저 운영 안정화
- 이후 1 또는 2로 고도화

### Phase 4. 완전 자동반영

목표:

- 정확 매칭 건은 사람이 개입하지 않아도 자동 수금 처리

전제:

- Phase 1, 2가 안정화되어야 함
- 자동반영 오류율이 매우 낮아야 함

규칙:

- `정확 일치 + 단일 후보 + 자동반영 허용 고객`만 자동 처리
- 나머지는 무조건 검토 큐

## 9. 운영상 중요한 예외

### 9.1 입금자명과 고객명이 다른 경우

해결:

- 고객별 별칭 설정

예:

- 실제 고객명: `채명례 이사장님`
- 입금자명: `김순자`
- 고객 상세에 `김순자`를 별칭으로 넣어 자동 처리

### 9.2 정부지원금/기관 예산

특징:

- 입금자는 기관명
- 실제 거래 고객과 다를 수 있음

해결:

- 해당 고객 또는 기관에 별칭 등록
- 자동반영이 아니라 예치금 경로로 연결 가능

### 9.3 초과 입금

해결:

- 자동반영 금지
- 검토 큐에서 아래 중 선택
  - 예치금
  - 환불대기
  - 수기 차감

### 9.4 동명이인

해결:

- 자동반영 금지
- `고객 우선순위`, `자동입금 제외`, `별칭`으로 좁힘

## 10. 성공 기준

### 운영 성공 기준

- 평소 입금 건의 다수가 자동으로 `정확 후보`까지 올라온다.
- 반복 입금 고객은 별칭 설정 후 거의 손대지 않고 처리된다.
- 직원은 예외 건만 검토하면 된다.
- 수금 관리, 고객 상세, 거래원장이 항상 같이 맞는다.

### 기술 성공 기준

- 업로드 기준 자동반영 오류 0건
- 동일 샘플 재업로드 시 중복 반영 없음
- 자동반영 후 취소 가능
- 별칭 추가 후 다음 입금부터 바로 매칭

## 11. 권장 실행 순서

1. Phase 1 완성
   - 별칭 저장
   - 별칭 자동매칭
   - 정확 일치 자동반영
2. Phase 2 구현
   - 검토 필요 운영 큐 완성
3. 수집 방식 확정
   - 은행 API / 이메일 / 공유폴더 / 수동 업로드 중 선택
4. Phase 3 구현
   - 주기 수집
5. Phase 4 적용
   - 완전 자동반영

## 12. 최종 판단

이 자동입금 플랜은 **실현 가능하다.**

다만 아래처럼 이해해야 한다.

- `자동 매칭/자동 반영 엔진`은 지금 구조로 개발 가능
- `실시간 자동 수집`은 은행 데이터가 들어오는 경로를 하나 정해야 닫힌다

따라서 가장 현실적인 순서는 이렇다.

1. 지금 CRM 안에서 별칭 기반 자동매칭/자동반영 완성
2. 검토 필요 큐 완성
3. 은행 데이터 유입 경로 연결

이 순서면 엑셀은 보조 기능으로 남고, 평소 운영은 자동입금 중심으로 옮길 수 있다.
