# 구례군 농업기술센터 입금 차액/예치금 수정보완 팀미팅 및 개발계획

## 1. 최종 확정 결론

구례군 농업기술센터 케이스는 “입금 1,000,000원” 전체를 해당 주문 매출이나 해당 거래명세표 수금액으로 처리하지 않는다.

- 해당 주문/명세표 금액: 743,500원
- 실제 입금액: 1,000,000원
- 해당 명세표 수금 처리액: 743,500원
- 차액: 256,500원
- 차액 처리: 고객 예치금/선입금으로 별도 보관
- 이후 주문: 예치금에서 차감 가능해야 함
- 세금계산서 발급 금액: 입금액 1,000,000원이 아니라 해당 명세표 공급가액/세액/합계 743,500원 기준

즉, 이번 수정보완의 핵심은 “현금 흐름”과 “매출/세금계산서 대상 금액”을 분리해 CRM에 안전하게 남기는 것이다.

## 2. 팀미팅 결정

### 박서연님(CFO)

- 초과입금은 해당 주문 매출이 아니므로 바로 매출/세금계산서 금액에 합산하지 않는다.
- 차액은 고객별 예치금으로 남기고, 이후 주문에서 사용될 때 “예치금 사용” 거래로 차감한다.
- 고객 상세 거래내역에서는 다음 흐름이 읽혀야 한다.
  - 입금 확인: 1,000,000원 수령
  - 해당 명세표 완납: 743,500원
  - 예치금 적립: 256,500원
  - 이후 명세표에서 예치금 사용: 사용액만큼 차감

### 김도현님(COO)

- 운영자는 실수 없이 처리해야 하므로, 초과입금 감지 시 “예치금 보관”이 기본 선택이어도 된다.
- 다만 이후 주문에서 예치금 자동 차감은 1차에서는 강제하지 않는다.
- 새 명세표 작성 시 사용 가능 예치금이 보이고, “전액” 버튼으로 한 번에 차감할 수 있으면 운영상 충분하다.
- 자동 차감은 잘못된 고객 선택 시 사고가 커질 수 있으므로 2차 옵션으로 분리한다.

### 최민석님(CTO)

- 기존 코드에 이미 다음 기능 일부가 구현되어 있다.
  - `Receivables.tsx`: 초과입금 감지 후 예치금/환불대기 선택
  - `InvoiceDialog.tsx`: 고객 예치금 사용 입력 및 invoice meta의 `depositUsedAmount` 저장
  - `accountingMeta.ts`: 고객/명세표 회계 메타 직렬화
  - `DepositInbox.tsx`: 자동입금 수집함에서 입금액을 미수 명세표에 배부하고 남은 금액을 예치금으로 적립
- 따라서 신규 대형 기능 추가가 아니라, 실제 구례군 시나리오 기준으로 기존 구현의 누락/오류를 검증하고 보완한다.
- 특히 다음 지점을 중점 확인한다.
  - 명세표 `current_balance`가 “이번 명세표 잔액”인지 “전잔액 포함 현재 잔액”인지 화면마다 다르게 해석되지 않는지
  - 예치금 사용액이 고객 예치금 잔액에서 정확히 차감되는지
  - 명세표 수정 시 예치금 사용액 변경분이 고객 예치금에 원복/재차감되는지
  - 고객 상세 거래내역/거래명세표 출력/미수금 화면에서 같은 숫자로 보이는지

### 조현우님(법무고문)

- 예치금은 아직 새 주문의 과세 공급가액이 아니다.
- 세금계산서 자동화와 연결할 때도 입금액 기준 발급을 금지하고, 명세표 품목/공급가액/세액 기준만 사용해야 한다.
- 수정세금계산서나 발행취소 판단은 별도 업무 흐름으로 분리한다.

## 3. 수정보완 목표 동작

### 3.1 미수금 화면 수동 입금 확인

조건:

- 미수 명세표 잔액: 743,500원
- 입금액 입력: 1,000,000원

기대 동작:

1. 초과입금 256,500원 감지
2. 기본 처리 방식: 예치금 보관
3. 저장 후 해당 명세표:
   - `paid_amount`: 기존 입금액 + 743,500원
   - `current_balance`: 0원
   - `payment_status`: `paid`
   - payment history에 743,500원 기록
4. 고객 회계 메타:
   - `depositBalance`: 기존 예치금 + 256,500원
   - events에 `deposit_added` 기록
5. 고객 통계 재계산 후 고객 상세/미수금/거래내역에 반영

### 3.2 이후 새 명세표 작성

조건:

- 고객 예치금: 256,500원
- 새 주문 금액: 예: 300,000원

기대 동작:

1. 고객 선택 시 사용 가능 예치금 256,500원 표시
2. 운영자가 “전액” 또는 직접 금액 입력으로 예치금 사용
3. 저장 후 새 명세표:
   - invoice meta `depositUsedAmount`: 사용액
   - 잔액: 새 주문금액 - 입금액 - 예치금 사용액
4. 고객 회계 메타:
   - `depositBalance`: 기존 예치금 - 사용액
   - events에 `deposit_used` 기록

### 3.3 명세표 수정/취소성 수정

조건:

- 기존 명세표에 예치금 사용액이 있음
- 수정 후 예치금 사용액이 감소/증가함

기대 동작:

- 증가분은 고객 예치금에서 추가 차감
- 감소분은 고객 예치금으로 원복
- 음수 예치금이 되지 않도록 방어
- 수정 이력은 events로 남김

## 4. 개발 작업 순서

### Step 1. 현행 구현 재현

1. 로컬 개발 서버 실행
2. 운영/테스트 데이터에서 구례군 고객 또는 테스트 고객으로 다음 흐름을 수동 재현
   - 743,500원 미수 명세표 생성
   - 1,000,000원 입금 확인
   - 고객 상세에서 예치금 256,500원 확인
   - 이후 새 명세표에서 예치금 차감
3. 화면/콘솔/API 오류 기록

### Step 2. 회계 계산 함수 분리 또는 보강

기존 로직이 화면 컴포넌트에 흩어져 있으면, 테스트 가능한 순수 함수로 분리한다.

후보 파일:

- `offline-crm-v2/src/lib/accountingMeta.ts`
- `offline-crm-v2/src/lib/receivables.ts`

검증할 계산:

- 초과입금 분리
- 예치금 사용 가능액
- 명세표 잔액
- payment status
- 예치금 변경 delta

### Step 3. UI 문구 보강

운영자가 헷갈리지 않도록 문구를 명확히 한다.

- “초과 입금”
- “예치금으로 보관”
- “환불대기로 등록”
- “사용 가능 예치금”
- “예치금 반영 후 잔액”
- “세금계산서 금액은 명세표 금액 기준” 안내 문구 필요 여부 검토

### Step 4. 테스트 추가

권장 테스트:

- 단위 테스트가 가능한 구조면 계산 함수 테스트 우선
- E2E는 최소 smoke만 추가

필수 시나리오:

1. 743,500원 미수에 1,000,000원 입금 시 예치금 256,500원 생성
2. 이후 300,000원 명세표에서 256,500원 예치금 사용 시 잔액 43,500원
3. 예치금 사용액을 수정하면 고객 예치금이 delta 기준으로 원복/재차감
4. 세금계산서 발급 계획과 충돌하지 않도록 명세표 합계는 입금액이 아니라 품목 합계 유지
