# 오픈뱅킹 기반 공급처 자동지급 전략

## 목적

- PRESSCO21 CRM에서 승인 완료된 공급처 지급건을 실제 송금까지 연결한다.
- 비전공자 직원은 `예정 등록 → 승인 → 송금 실행 결과 확인`만 이해하면 되도록 단순화한다.
- 금융결제원 오픈뱅킹 승인 전에는 개발 착수하지 않고, 승인 후 바로 구현 가능한 수준으로 전략을 고정한다.

## 1차 목표

- 공급처 지급건을 CRM에서 관리하다가, `승인 완료` 상태인 건만 자동 송금 실행
- 송금 결과를 다시 CRM에 반영
- 실패/중복/한도 초과를 자동 차단

핵심 UX 문구:

- 보낼 수 있음
- 승인 필요
- 계좌 확인 필요
- 송금 실패
- 송금 완료

## 전제 조건

다음 조건이 충족되기 전까지는 개발하지 않는다.

1. 금융결제원 오픈뱅킹 테스트/운영 승인 확보
2. PRESSCO21 출금계좌 등록 가능 여부 확인
3. Client ID / Client Secret / 이용기관 권한 확보
4. Callback URL 등록 가능 상태 확인
5. 테스트베드에서 계좌실명조회, 입금이체 호출 가능 범위 확인

## 권장 Callback URL

- 운영 기준: `https://crm.pressco21.com/openbanking/callback`

이유:

- 나중에 CRM 내부에서 인증/연동 상태를 보여주기 쉽다.
- n8n webhook로 바로 받는 구조보다 장기 유지보수에 유리하다.

## 구현 범위

### Phase A. 연동 기반

1. 오픈뱅킹 연동 서버 모듈
- 토큰 발급/갱신
- 계좌실명조회
- 입금이체 실행
- 이체 결과 조회

2. CRM 메타 필드
- 자동지급 사용 여부
- 자동지급 1회 한도
- 자동지급 1일 한도
- 승인 필요 여부
- 송금 가능 상태
- 실명조회 확인 상태
- 외부거래 고유키

### Phase B. 지급 실행

1. 공급처 지급건 상태
- 초안
- 승인 대기
- 승인 완료
- 송금 요청
- 송금 완료
- 실패
- 재확인 필요

2. 자동 차단 규칙
- 계좌 미등록
- 실명조회 실패
- 승인 미완료
- 한도 초과
- 동일 지급건 중복 실행
- 영업시간 외 실행 금지 여부

### Phase C. 결과 반영

1. 송금 성공
- 지급 완료 이력 기록
- 공급처 지급 예정 잔액 차감
- 활동 로그 적재

2. 송금 실패
- 실패 상태 기록
- 실패 사유 저장
- 재시도 가능 여부 저장

## UX 원칙

### 대표 관점

- 오늘 실제 송금될 총액
- 승인 대기 금액
- 실패 건수
- 위험 건수

### 비전공자 직원 관점

- 무엇부터 처리할지
- 지금 송금 가능한지
- 왜 막혔는지
- 송금 후 결과가 무엇인지

### 화면 구성 원칙

1. 복잡한 API 용어를 직접 노출하지 않는다.
2. 기술 상태는 관리자 상세 영역에만 둔다.
3. 기본 화면에는 행동과 결과만 보여준다.

## 권장 실행 흐름

1. 공급처 계좌 등록
2. 지급 예정 등록
3. 승인 완료
4. 계좌 검증
5. 송금 실행
6. 결과 반영
7. 실패 시 재검토

## 감사 로그 원칙

반드시 남겨야 할 항목:

- 누가 승인했는지
- 언제 송금 요청했는지
- 어떤 공급처에 얼마를 보냈는지
- 응답 코드/거래 고유키
- 실패 사유

## 보안 원칙

- Secret은 `.secrets.env`나 서버 credential에만 둔다.
- 프론트엔드에 Secret을 노출하지 않는다.
- 송금 실행은 반드시 서버 측 모듈에서만 호출한다.
- 테스트/운영 키를 분리한다.

## 개발 착수 조건

아래 세 가지가 모두 확인되면 구현에 들어간다.

1. 금융결제원 승인 완료
2. 테스트베드 계좌 연동 성공
3. 출금계좌 사용 가능 범위 확인 완료
