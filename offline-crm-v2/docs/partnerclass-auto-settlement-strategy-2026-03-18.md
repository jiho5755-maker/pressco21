# 파트너클래스 자동 정산 전략 메모

## 목적

- 향후 파트너클래스의 파트너사/협회/강사 정산을 규칙 기반으로 자동화한다.
- 현재 CRM 공급처 자동지급 구조를 재사용 가능한 정산 엔진 형태로 확장한다.

## 왜 지금 미리 설계하는가

- 나중에 파트너클래스 정산이 붙을 때 CRM 공급처 지급 구조를 그대로 활용하려면,
  지금부터 데이터 구조와 상태 체계를 맞춰두는 것이 유리하다.
- 단, 실제 개발은 공급처 자동지급 1차가 안정화된 뒤 시작한다.

## 목표 구조

1. 거래 발생
- 주문 확정
- 수강 완료
- 취소/환불 반영

2. 정산 규칙 엔진
- 정산 대상 선별
- 정산 금액 계산
- 최소 지급 금액 확인
- 보류 조건 확인

3. 자동 지급 엔진
- 승인 필요 여부 판단
- 조건 충족 시 송금 실행
- 결과 원장 반영

## 정산 대상

- 파트너사
- 협회
- 강사

## 정산 기준 예시

- 주문 확정 후 N일
- 수강 완료 후 N일
- 환불 가능 기간 종료 후 지급
- 수수료율 차감 후 순지급
- 최소 지급 금액 이상일 때만 지급
- 정산 주기: 주간 / 월간 / 수시

## 필요한 공통 필드

- 정산 대상 유형
- 정산 주기
- 수수료율 또는 정산율
- 자동 승인 여부
- 최소 지급 금액
- 보류 상태
- 계좌 등록 상태
- 최근 지급 상태

## 자동화 정책 예시

### 예시 1. 월간 자동 정산

- 매월 말 집계
- 다음 달 5일 자동 송금
- 최소 50,000원 이상만 지급
- 계좌 등록 및 승인 완료 필수

### 예시 2. 신규 파트너 보호 정책

- 첫 1회는 수동 승인
- 두 번째부터 자동 승인 허용

### 예시 3. 환불 보류 정책

- 최근 14일 내 환불 가능 거래가 있으면 자동 지급 보류

## UX 원칙

비전공자 관점에서는 아래만 보이면 된다.

- 이번 주 지급 예정
- 승인 필요
- 자동 지급 가능
- 보류
- 지급 완료

대표 관점에서는 아래가 중요하다.

- 이번 달 정산 총액
- 자동 지급 예정 금액
- 승인 대기 금액
- 보류 금액

## CRM과의 연결 원칙

- 공급처 자동지급 구조를 공통 지급 엔진으로 발전시킨다.
- 파트너클래스 정산도 최종적으로는 `지급 예정 → 승인 → 지급 완료` 흐름을 공유한다.
- 전용 화면이 필요하더라도 지급 상태 체계와 감사 로그 체계는 공통으로 간다.

## 나중에 구현할 때 우선순위

1. 정산 예정 생성
2. 승인 정책 엔진
3. 자동 지급
4. 정산 리포트

## 지금 단계 결론

현재는 공급처 자동지급 1차가 우선이다.

파트너클래스 자동 정산은:

- 설계 메모만 유지
- 실제 개발은 오픈뱅킹 승인 및 CRM 자동지급 안정화 후 시작
