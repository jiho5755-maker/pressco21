# RPA 자동송금 파일럿 PRD

## 문서 목적

- PRESSCO21 CRM에서 공급처 지급과 향후 파트너 정산 지급을 `확인 버튼 1회`에 가깝게 처리하기 위한 1차 파일럿 구조를 정의한다.
- 공식 송금 API가 없는 상황에서 `은행 웹 + RPA` 방식이 실제 운영 가능한지 최소 범위로 검증한다.
- 본 문서는 `파일럿 설계 문서`이며, 전면 자동송금 도입 문서가 아니다.

## 배경

- 오픈뱅킹/공식 송금 API는 개인사업자, 비용, 승인 절차 문제로 당장 도입이 어렵다.
- 일일이 공급처/파트너 계좌로 수동 송금하는 업무 비용이 크다.
- PRESSCO21 CRM에는 이미 `지급 예정`, `승인`, `지급 완료`, `활동 로그` 흐름이 구축되어 있다.
- 따라서 `송금 실행`만 자동화하면 업무 효율이 크게 오른다.

## 파일럿 목표

1. CRM에서 승인 완료된 지급건을 `전용 Windows PC`에서 실행 가능한 형태로 전달한다.
2. 담당자는 `지급 실행` 버튼만 누르고, RPA가 은행 웹에서 예약이체 또는 다건이체를 수행한다.
3. 성공/실패 결과를 CRM에 다시 반영한다.
4. 실패 시 즉시 중단하고, 동일 지급건 중복 송금을 막는다.

## 성공 기준

- 파일럿 대상 지급건 10건 이상을 오류 없이 처리
- 중복 송금 0건
- 실행 후 CRM 결과 반영 누락 0건
- 실무 담당자 기준 체감 소요 시간 50% 이상 단축

## 파일럿 범위

### 포함

- 공급처 지급건
- 승인 완료 상태의 지급건만
- 전용 Windows PC 1대
- 은행 1곳만 먼저 선정
- 지급 실행 로그, 캡처, 결과 반영

### 제외

- 고객 환불 자동송금
- 파트너클래스 자동 정산 전면 연결
- 무인(Unattended) 24시간 자동 실행
- 복수 은행 동시 지원

## 권장 파일럿 방식

### 1순위

- `예약이체 자동등록 RPA`

이유:

- 즉시이체보다 사고 위험이 낮다.
- 은행이 실제 출금을 담당하므로 운영 안정성이 높다.
- 비전공자 직원이 이해하기 쉽다.

### 2순위

- `다건이체 자동입력/업로드`

### 3순위

- `즉시이체 RPA`

## 전제 조건

1. 전용 Windows PC 또는 미니PC 1대 확보
2. 해당 PC는 송금 업무 외 다른 용도로 사용하지 않음
3. 인터넷뱅킹/기업뱅킹 로그인 방식과 OTP 절차 확인
4. 은행 메뉴에 예약이체, 다건이체, 파일이체 중 최소 1개가 존재
5. 대표 또는 담당자 승인 절차를 CRM에서 먼저 끝냄

## 사용자 유형

### 대표

- 오늘 송금 총액만 확인
- 위험 건수만 확인
- 필요 시 승인 여부만 결정

### 담당 직원

- 지급 예정 확인
- 승인 완료 확인
- `지급 실행` 클릭
- 실패/완료 결과 확인

## 사용자 흐름

1. 공급처 지급 예정 등록
2. 승인 완료
3. `지급 실행 대기함` 확인
4. 직원이 `지급 실행` 클릭
5. RPA 에이전트가 실행 요청 수신
6. 은행 사이트 접속
7. 예약이체 또는 다건이체 등록
8. 실행 결과 캡처
9. CRM에 `지급 완료` 또는 `실패` 반영
10. 활동 로그 및 텔레그램 알림 기록

- 초안
- 승인 대기
- 승인 완료
- 지급 요청
- 지급 완료

## 필수 안전장치

1. 동일 지급건 중복 실행 금지
2. 1회 최대 금액 제한
3. 1일 총액 제한
4. 허용된 계좌만 송금
5. 실행 전 공급처명/계좌/금액 재검증
6. 실행 후 스크린샷 저장
7. 실패 시 즉시 중단
8. 실패 사유 저장
9. 대표/관리자 알림

예시 문구:

- 지급 완료
- 실패, 다시 확인 필요

## 권장 기술 방향

### 1차 권장

- `Power Automate Desktop` 기반 Attended RPA

이유:

- 초기 비용이 상대적으로 낮다.
- Windows 기반 운용이 쉽다.
- 웹 자동화 확장과 UI 클릭 흐름을 빠르게 구축할 수 있다.

### 차선

- Playwright 또는 Selenium 기반 커스텀 자동화

주의:

- 보안프로그램, 은행 UI 변경 대응 난이도가 더 높다.
- 유지보수 부담이 커질 수 있다.

## 운영 방식

### 권장

- Attended RPA

의미:

- 직원이 최종 실행 버튼은 누른다.
- 이후 이체 입력/실행/결과 기록은 RPA가 수행한다.

### 비권장 1차

- Unattended RPA

이유:

- 비용과 운영 제약이 커진다.
- 파일럿 단계에서는 과하다.

## 은행 선정 원칙

1. 예약이체 메뉴가 명확한지
2. 다건이체 또는 파일이체가 가능한지
3. 로그인/OTP 흐름이 일관적인지
4. 브라우저/보안모듈 안정성이 높은지
5. 실무자가 이미 익숙한 은행인지

## 권장 은행 파일럿 우선순위

1. 신한
2. KB
3. 우리

이유:

- 주거래은행 연속성
- 개인사업자 기업뱅킹 가능성
- 대량/예약 이체 운용 가능성

## 파일럿 완료 후 판단 기준

아래 조건을 만족하면 확대 검토:

- 10건 이상 무사고
- 오류 복구 절차 검증 완료
- 담당자 학습 부담이 크지 않음
- 은행 UI 변경 대응 가능성 수용 가능

반대로 아래면 중단:

- OTP/보안프로그램 때문에 실행 안정성이 매우 낮음
- 은행 UI 변동이 너무 큼
- 결과 반영 누락이 반복됨
- 담당자에게 오히려 부담이 큼

## 파일럿 후 확장 방향

1. 공급처 지급 전면 확대
2. 고객 환불 일부 연결
3. 파트너클래스 정산 지급 연결
4. 장기적으로 공식 송금 API로 교체 가능성 검토
