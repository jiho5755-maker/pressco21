# RPA 자동송금 도입 순서

## 목적

- PRESSCO21 CRM에 직접 송금 보조 기능을 붙일 때, 어디까지 먼저 만들고 어떤 순서로 검증할지 정리한다.

## 원칙

1. 전면 자동화보다 파일럿이 먼저다.
2. 은행 1곳만 먼저 붙인다.
3. 공급처 지급부터 시작한다.
4. 파트너클래스 정산은 공급처 지급 파일럿 성공 후 확장한다.

## Phase 1. 은행 선정

### 해야 할 일

1. 신한, KB, 우리 중 1곳 실사
2. `예약이체 / 다건이체 / 파일이체` 가능 여부 확인
3. 파일럿 적합 은행 1곳 확정

### 완료 기준

- 파일럿 은행 1곳 결정
- 추천 실행 메뉴 결정

## Phase 2. 전용 PC 준비

### 해야 할 일

1. Windows 전용 PC 또는 미니PC 준비
2. 은행 전용 브라우저 설정
3. OTP/인증서/보안모듈 정상 동작 확인
4. Power Automate Desktop 설치

### 완료 기준

- 수동으로 은행 이체 메뉴 진입까지 문제 없음
- 동일 PC에서 반복 사용 가능

## Phase 3. CRM 실행 대기함 보강

### 해야 할 일

1. 지급 실행 대기함
2. 계좌/금액 복사 버튼
3. 다건 송금용 내보내기
4. 송금 완료 일괄 처리

### 완료 기준

- 자동송금이 없어도 수동 송금 속도가 충분히 빨라짐

## Phase 4. RPA 파일럿 구현

### 해야 할 일

1. CRM `지급 실행` 버튼 추가
2. 로컬 RPA 에이전트 호출 방식 결정
3. 예약이체 또는 다건이체 자동 입력 구현
4. 결과 캡처 저장
5. 성공/실패를 CRM에 반영

### 완료 기준

- 테스트 지급건 3건 이상 성공
- 실패 시 복구 경로 확인

## Phase 5. 소규모 운영 파일럿

### 대상

- 소액
- 소수 공급처
- 반복 지급 거래처 위주

### 해야 할 일

1. 실제 운영 공급처 몇 곳만 연결
2. 10건 이상 실행
3. 문제 유형 기록

### 완료 기준

- 무사고 10건 이상
- 중복 송금 0건
- 결과 반영 누락 0건

## Phase 6. 확대 여부 결정

### 확대 조건

- 담당자 만족도 높음
- 안정성 수용 가능
- 은행 UI 변경 대응 가능

### 보류 조건

- 인증 절차가 너무 불안정
- 유지보수 부담이 큼
- 사고 위험이 높음

## 파트너클래스 확장 조건

공급처 지급 파일럿이 안정화되면, 동일 구조를 파트너클래스 정산으로 확장한다.

### 확장 시 필요한 것

1. 정산 규칙 엔진
2. 파트너별 지급 예정 생성
3. 수수료 차감 계산
4. 파트너 지급 원장
5. 동일한 `승인 -> 지급 실행 -> 결과 반영` 구조 재사용

## 최종 권장안

지금 당장 구현 순서는 다음이 맞다.

1. 은행 1곳 선택
2. 전용 PC 준비
3. CRM 수동 송금 보조 UX 강화
4. 예약이체 자동등록 RPA 파일럿
5. 성공 시 확대

즉, 자동송금은 바로 전면 도입이 아니라 `CRM 보조 -> 파일럿 -> 확대` 순으로 간다.
