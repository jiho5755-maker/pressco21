# 공급처/파트너 자동송금 후보군 및 실패 시 대안 정리

## 문서 목적

- PRESSCO21 CRM에서 `확인 버튼만 누르면 송금`에 가까운 자동지급 구조를 만들기 위한 후보군을 정리한다.
- 개인사업자 기준으로 실제 계약/도입 가능성이 있는지 먼저 구분한다.
- 외부 서비스 도입이 실패했을 때, 직접 구현 가능한 대안을 별도로 정리한다.
- 이 문서는 `후보 분석 문서`이며, 아직 특정 방식을 확정하지 않는다.

## 현재 전제

- 금융결제원 오픈뱅킹은 비용과 승인 절차 부담이 커서 당장 추진하지 않는다.
- 농협 오픈플랫폼은 현재 확인 기준 `법인 사업자` 중심으로 보이며, 개인사업자 직접 송금 API 경로는 불명확하다.
- 우리가 원하는 최종 UX는 다음과 같다.
  - CRM에서 지급 예정 확인
  - 담당자 또는 대표가 확인
  - `지급 실행` 버튼 1회
  - 공급처/파트너 계좌로 실제 송금
  - 결과가 CRM에 자동 반영

## 판단 기준

각 후보는 아래 기준으로 평가한다.

1. 개인사업자 가능성
2. 실제 국내 계좌로 직접 송금 가능한지
3. CRM에서 자동화하기 쉬운지
4. 비용 부담
5. 운영 안정성

평가 표기:

- 높음
- 중간
- 낮음
- 불명확

## 외부 후보 상세

### 1. 포트원 플랫폼 정산

가능한 역할:

- 파트너클래스 정산 규칙 엔진
- 파트너 정산 원장
- 수수료, 할인, 정산 상태 관리
- 일괄 지급 상태 처리

주의:

- 공개 문서만으로는 `PRESSCO21 계좌에서 국내 파트너 계좌로 직접 송금`이 확정되지 않았다.
- 즉, 송금 엔진이라기보다 `정산 구조와 프로세스` 쪽이 더 강해 보인다.

우리에게 맞는 활용:

- 장기적으로 파트너클래스 정산 구조를 정리하는 플랫폼 후보
- 단기적으로는 공급처 송금 실행 엔진으로 확정할 수 없음

현재 판단:

- `정산 자동화` 후보
- `직접 송금` 후보는 추가 확인 필요

### 2. KG이니시스 지급대행

가능한 역할:

- 하위 판매자/파트너 정산
- 지급 요청
- 송금 결과 확인
- 잔액 대사

장점:

- 공개 특약과 매뉴얼에 지급대행 구조가 비교적 명확하다.
- 개인사업자 가능성이 문서상 배제되지 않는다.
- `지급 요청 -> 결과 조회` 형태로 CRM 연동 그림을 그리기 쉽다.

주의:

- 최소 계약 조건, 월 이용료, 송금 수수료는 별도 확인이 필요하다.
- PRESSCO21처럼 `자사몰 대금 수취 후 파트너 재정산` 구조가 가능한지 계약 범위 확인이 필요하다.

현재 판단:

- 가장 현실적인 지급대행 후보 1순위

### 3. 토스페이먼츠 지급대행

가능한 역할:

- 지급대행 API 기반 송금/정산

장점:

- API 문서가 잘 정리되어 있다.
- 운영 안정성은 높게 기대할 수 있다.

주의:

- 공개 문서상 라이브 테스트에도 비용이 붙는 구조다.
- 개인사업자 가능 여부와 최소 계약 조건을 따로 확인해야 한다.

현재 판단:

- 기술 구조는 적합할 수 있으나 비용 검증이 먼저 필요

### 4. 우리은행 웹펌뱅킹

가능한 역할:

- 지급서비스
- 수납서비스
- 계좌간 자동이체

장점:

- 공식 안내상 `개인사업자 및 중소기업` 대상 문구가 확인된다.
- 은행 직접 계약 방식이라 `우리 계좌에서 직접 송금` 니즈에 가장 가깝다.

주의:

- API, 파일 업로드, 웹 모듈 중 어떤 방식인지 추가 확인이 필요하다.
- 구축비/월비용/건당 비용과 심사 절차를 확인해야 한다.

현재 판단:

- 개인사업자 기준 은행 직접 자동송금 후보 1순위

### 5. KB 기업 대량이체 / StarCMS

가능한 역할:

- 대량이체
- 다수 사업장/정산 자금관리

장점:

- 공식 자료에서 개인사업자 대상이 확인된다.
- 대량이체 기능이 있어 공급처 지급과 파트너 정산 모두에 응용 가능하다.

주의:

- 실제 자동 연동이 API인지, 파일 업로드인지, 웹 기반인지 확인이 필요하다.

현재 판단:

- 은행 직접 자동송금 후보 2순위

### 6. 신한 기업CMS / 펌뱅킹

가능한 역할:

- 자동이체/CMS/자금관리

장점:

- 주거래은행이므로 계좌, 한도, 실사용 측면에서 가장 자연스럽다.

주의:

- 공개 자료만으로는 우리/KB보다 구조가 덜 명확하다.
- 신청서 기반 추가 확인이 필요하다.

현재 판단:

- 실제 계약 가능 시 가장 운영 친화적일 수 있는 후보

## 실패 시 직접 구현 대안

외부 지급대행/은행 계약형 서비스가 모두 실패할 경우, 아래 직접 구현 대안으로 전환한다.

### A. 예약이체 자동등록 RPA

개념:

- CRM에서 지급 승인 완료
- 전용 PC가 은행 인터넷뱅킹에 접속
- 예약이체를 자동 등록
- 실제 출금은 은행이 수행

장점:

- 즉시 송금보다 안전하다.
- 직원은 `승인 -> 예약 등록`만 이해하면 된다.
- 비용이 매우 낮다.

주의:

- 전용 PC 운영이 필요하다.
- 인증서, OTP, 보안모듈, 브라우저 환경 통제가 필요하다.

현재 판단:

- 직접 구현 대안 중 최우선

### B. 즉시이체 RPA

개념:

- CRM에서 승인 완료된 건을 전용 PC가 은행 화면에서 바로 송금

장점:

- 구조가 가장 단순하다.
- CRM에서 `지급 실행` 버튼과 연결하기 쉽다.

주의:

- 예약이체보다 사고 위험이 높다.
- 중복 송금 방지, 실패 중단, 실행 로그가 필수다.

현재 판단:

- 예약이체 자동등록이 안 될 경우 차선책

### C. 파일이체 생성 + 업로드 자동화

개념:

- CRM이 지급 파일 생성
- 전용 PC가 은행 이체 업로드 메뉴에 자동 업로드
- 대량 지급 실행

장점:

- 다건 지급에 유리하다.
- 공급처와 파트너 정산 규모가 커질수록 효율이 높다.

주의:

- 은행별 파일 형식과 업로드 메뉴 지원 여부를 확인해야 한다.

현재 판단:

- 은행이 지원하면 가장 안정적인 직접 구현 대안

## PRESSCO21 기준 추천 순서

### 1차 확인 후보

1. 우리은행 웹펌뱅킹
2. KG이니시스 지급대행
3. 신한 기업CMS/펌뱅킹
4. KB 대량이체/StarCMS
5. 토스페이먼츠 지급대행
6. 포트원 정산 구조

### 2차 전환 대안

위 후보가 모두 비용/계약/심사 조건에서 맞지 않으면:

1. 예약이체 자동등록 RPA
2. 파일이체 자동화
3. 즉시이체 RPA

## CRM 설계 관점 권장 방향

외부 수단이 무엇이든 CRM 쪽 상태 구조는 공통으로 가져간다.

권장 상태:

- 초안
- 검토 완료
- 승인 대기
- 승인 완료
- 지급 요청
- 지급 완료
- 실패
- 재확인 필요

즉, 송금 엔진이 무엇으로 결정되든 CRM은 동일한 상태 구조를 유지한다.

## 지금 당장 할 일

1. 후보별 문의 메일 발송
2. 각 후보의 아래 항목 회신 확보
   - 개인사업자 가능 여부
   - 최소 계약 조건
   - 초기비용 / 월비용 / 건당 수수료
   - 실제 국내 계좌 송금 가능 여부
   - API / 파일 / 웹모듈 방식
3. 회신 결과를 이 문서에 추가해 후보군을 줄이기

## 보류 판단 기준

아래 중 하나라도 해당하면 우선순위를 내린다.

- 개인사업자 불가
- 월 고정비 과다
- 송금 API/API 대체 수단 불명확
- 국내 실계좌 송금이 아닌 내부 정산 기능만 제공
- CRM에서 결과 수집이 어렵거나 수동 작업이 많이 남음

## 최종 정리

현재 시점 결론:

- `외부 계약형 후보`는 존재한다.
- 가장 유력한 축은 `우리은행 웹펌뱅킹`, `KG이니시스 지급대행`, `신한 기업CMS`, `KB 대량이체`다.
- `포트원`은 정산 엔진 측면에서 유력하지만 직접 송금은 별도 확인이 필요하다.
- 전부 실패해도 `예약이체 자동등록 RPA` 또는 `파일이체 자동화`로 직접 구현은 가능하다.

즉, 자동송금은 아직 끝난 이슈가 아니고 `후보 압축 -> 실패 시 직접 구현`의 두 갈래 전략으로 계속 추진한다.
