# PRESSCO21 승인 정책 가이드 v0.1 (초안)

_상태: 초안 / 현실 검증 필요 / 실무 적용 전_

_대상: 지호님, 대표님, 플로라, Claude Code, Codex, 향후 개발 에이전트_

## 1. 문서 목적

이 문서는 PRESSCO21 운영에서 **누가 무엇을 확인하고, 누가 최종 승인하며, 어디까지 자동화가 가능하고 어디서 사람 판단이 반드시 필요한지**를 정리하기 위한 승인 정책 초안이다.

이 문서의 목적은 아래와 같다.

1. 승인 책임자를 명확히 한다.
2. 지호님 / 대표님 / 실무자 권한 경계를 구분한다.
3. 플로라와 자동화가 어디까지 개입 가능한지 정한다.
4. 향후 CRM, 파트너클래스, n8n, OpenClaw 자동화 설계 기준으로 삼는다.

---

## 2. 이 문서는 아직 초안이다

이 문서는 현재까지의 대화와 운영 맥락을 기준으로 정리한 구조적 초안이다.

아직 아래 이유로 최종 확정본은 아니다.
- 대표님 실제 의사결정 패턴을 더 반영해야 할 수 있음
- 직원별 실무 처리 범위를 더 세밀하게 봐야 함
- 금액 기준/예외 규칙이 구체적으로 확정되지 않음
- 시스템 구현 시 role/permission 구조와 다시 맞춰야 함

즉, 이 문서는 **승인 정책의 첫 기준점**으로 보고 지호님 현실 체크 후 보정해야 한다.

---

## 3. 현재 반영된 핵심 현실 전제

### 3.1 실질 운영 결정권
- 법적/직함상 대표는 이진선 대표님
- 운영/시스템/자동화의 실질 결정권자는 지호님
- 따라서 승인 체계는 단순히 "대표 승인" 하나로 묶지 않고 아래 3층으로 본다.

1. **지호님 확인 필수**
2. **대표 최종 승인 필수**
3. **실무자 단독 처리 가능**

### 3.2 자동화와 승인 관계
- 자동화 범위는 넓게 갈 수 있다
- 다만 수정/확정 권한은 좁게 가져가야 한다
- 저숙련 직원에게 넓은 수정 권한을 열어두면 운영이 흔들릴 수 있다

### 3.3 외부 커뮤니케이션 승인 기준
외부 발송은 단순 민감정보 여부보다,
**대표 없이 보냈을 때 상대가 불만/오해를 가질 수 있는가**를 핵심 기준으로 본다.

---

## 4. 승인 레벨 정의 (초안)

## Level 1. 실무자 단독 처리 가능
### 의미
반복적이고 저위험이며, 실무 범위 안에서 처리 가능한 업무

### 예시
- 정해진 양식에 따른 내부 기록 입력
- 이미 승인된 작업의 후속 처리
- 단순 상태 업데이트(권한 허용 범위 안)
- 내부 자료 정리
- 단순 문의 응답 초안 검토 후 발송(사전 승인된 템플릿 기반)

### 주의
- 금액 변경
- 외부 관계 악화 가능성
- 정책 해석이 필요한 사안
은 제외

---

## Level 2. 지호님 확인 필수
### 의미
운영, 시스템, 자동화, 내부 우선순위, 상태 변경, 문서 구조와 관련된 주요 판단은 지호님 확인을 거친다.

### 예시
- 자동화 신규 적용/수정
- 상태 변경 규칙 수정
- 내부 업무 흐름 변경
- 정산 초안 검토
- 입금 대사 후보 확정
- 디자인/기획 요청 체계 변경
- 파트너/클래스 운영 정책 보정
- 내부용 중요 문서 확정

### 현재 해석
실질적으로 PRESSCO21 운영 시스템의 허브 승인자 역할

---

## Level 3. 대표 최종 승인 필수
### 의미
대표 명의, 대외 관계, 금액/거래 책임, 브랜드 책임이 걸리는 판단은 대표 최종 승인 필요

### 예시
- 가격 정책 최종 변경
- 환불/보상 최종 판단
- 고액 정산/지급 확정
- 외부 공식 메시지/제안 발송
- 중요한 파트너/제휴 수용 여부
- 대표 명의 문서/입장 정리

### 비고
지호님이 사전 검토/초안 정리까지 할 수 있으나,
최종 책임이 대표에게 있는 사안은 대표 승인을 남긴다.

---

## 5. 영역별 승인 정책 초안

## 5.1 브리핑 / 요약 / 정리
### 기본 승인 레벨
- **자동 가능** 또는 **지호님 요청 시 실행**

### 이유
위험도가 낮고 체감 가치가 높다.

### 예시
- 회의 요약
- 오늘 할 일 브리핑
- 미입금/미처리 건 요약
- 디자인 요청 정리

---

## 5.2 내부 문서 초안 생성
### 기본 승인 레벨
- **자동 초안 생성 가능**
- **중요 문서는 지호님 확인**

### 예시
- 기획 초안
- 내부 운영 문서 초안
- 디자인 요청문 초안
- 회의록 초안

---

## 5.3 상태 변경
### 기본 승인 레벨
- **권한 허용 범위 내 실무자 또는 자동화 가능**
- 다만 규칙 설계/범위 확정은 **지호님 확인 필수**

### 예시
- 입금확인 후 상태 변경
- 모집중 → 마감
- Task todo → doing → done
- 내부 검토 상태 전환

### 주의
상태 자동화는 넓게 가능하지만,
아무나 상태를 바꾸게 하면 안 된다.

---

## 5.4 입금 / 대사 / 정산
### 기본 승인 레벨
- **후보 제안 / 초안 계산은 자동 가능**
- **확정은 지호님 확인 필수**
- **대외 책임이 큰 지급은 대표 최종 승인 가능성 검토**

### 예시
- 입금 대사 후보
- 미수금 리포트
- 정산 계산 초안
- 지급 예정 목록

### 주의
정산은 자동화 가능 영역이지만,
최종 확정/송금은 별도 승인 레벨이 필요하다.

---

## 5.5 외부 발송 / 외부 메시지
### 기본 승인 레벨
- **초안은 자동 가능**
- **발송은 지호님 확인 또는 대표 승인 필요**

### 예시
- 파트너 안내
- 거래처 설명 메시지
- 공식 이메일
- 고객 중요 응대 메시지

### 판단 기준
상대방이 오해하거나 불만을 가질 수 있는가?
그렇다면 자동 발송 금지에 가깝게 본다.

---

## 5.6 가격 / 환불 / 보상 / 정책 변경
### 기본 승인 레벨
- **대표 최종 승인 필수**

### 이유
브랜드/금액/대외 책임이 걸리는 영역

### 예시
- 가격 조정
- 예외 할인
- 환불 금액 확정
- 보상 기준 변경
- 외부 정책 변경

---

## 5.7 파트너 / 제휴 / 클래스 승인
### 기본 승인 레벨
- **실무 검토 + 지호님 확인**
- 중요도에 따라 **대표 최종 승인**

### 예시
- 파트너 신청 승인
- 제휴 진행 여부
- 중요한 클래스 공개 승인
- 정산 정책이 걸린 파트너 수용 여부

---

## 6. 플로라 승인 정책 초안

## 플로라가 승인 없이 할 수 있는 것
- 요약
- 브리핑
- 분류
- 초안 작성
- 내부 문서 구조화
- 업무(Task) 초안 생성

## 플로라가 제안까지만 해야 하는 것
- 상태 변경 제안
- 정산 초안
- 우선순위 추천
- 외부 발송 문안
- 파트너 승인 추천

## 플로라가 하면 안 되는 것
- 지호님 확인 없이 운영 규칙 바꾸기
- 대표 승인 대체
- 외부 민감 메시지 독단 발송
- 가격/환불/정산 최종 확정

---

## 7. 직원 UX 기준 반영

승인 정책은 단순 통제 문서가 아니라,
직원들이 실제로 사용할 수 있어야 한다.

### 현재 반영 원칙
- 입력 규칙이 너무 많으면 안 된다
- 승인 단계가 지나치게 복잡하면 안 된다
- AI가 먼저 과하게 끼어들면 안 된다
- 실무자가 자기 업무 흐름 안에서 자연스럽게 처리할 수 있어야 한다

즉,
**권한은 강하게 통제하되 사용 흐름은 단순해야 한다**
가 핵심이다.

### 자동 가능

- 내부 업무 생성

### 지호님 확인 필수
- 자동화 규칙 변경
- 내부 중요 문서 확정
- 상태 변경 범위 조정
- 정산 초안 검토
- 입금 대사 확정
- 운영 구조 변경

### 대표 최종 승인 필수
- 가격 / 환불 / 보상
- 외부 공식 메시지
- 대외 제휴 핵심 판단
- 대표 책임이 걸리는 문서/결정

## 9. 지호님 현실 체크 포인트

다음은 추가로 지호님이 체크하면 좋은 항목이다.

1. 지호님 확인 필수 항목이 너무 넓지는 않은지
2. 대표 최종 승인 범위를 금액 기준으로도 나눌지
3. 실무자 단독 처리 가능 범위를 더 넓혀도 되는 부서가 있는지
4. 파트너 승인에서 대표 승인까지 필요한 케이스가 실제 얼마나 많은지
5. 정산 확정에서 대표 승인과 지호님 확인을 어떻게 나눌지

## 11. 최종 운영 문장 (초안)

> PRESSCO21의 승인 정책은 실무자 단독 처리, 지호님 확인, 대표 최종 승인 3층 구조를 기본으로 한다.  
> 자동화 범위는 넓게 가져갈 수 있지만, 수정/확정 권한은 좁게 가져간다.  
> 플로라는 브리핑과 초안, 구조화에는 적극 개입하되, 민감 실행의 최종 책임을 대체하지 않는다.
