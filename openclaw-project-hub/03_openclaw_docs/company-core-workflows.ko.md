# PRESSCO21 핵심 업무 흐름 가이드 v1

_대상: 지호님, 플로라, Claude Code, Codex, 향후 개발 에이전트_

## 1. 문서 목적

이 문서는 PRESSCO21의 핵심 운영 흐름을 **원천 엔터티와 상태체계 기준으로 연결**하기 위한 업무 흐름 정의서다.

목적은 아래와 같다.

1. 회사 핵심 업무가 어떤 순서로 흘러가는지 공통 기준을 만든다.
2. CRM, 파트너클래스, NocoDB, n8n, 플로라가 같은 흐름 기준으로 동작하게 한다.
3. 자동화 가능한 구간과 사람이 확인해야 하는 구간을 구분한다.
4. 향후 화면 설계, DB 설계, 승인 설계, 브리핑 설계의 기준으로 삼는다.

---

## 2. 설계 원칙

### 원칙 1. 흐름은 엔터티와 상태를 기준으로 적는다
막연한 설명이 아니라, 어떤 엔터티가 어떤 상태를 거쳐 이동하는지 표현한다.

### 원칙 2. 사람 역할과 자동화 역할을 분리한다
자동화 후보는 자동화 후보로, 최종 판단은 승인 단계로 나눈다.

### 원칙 3. 예외 상황을 별도로 본다
정상 흐름과 예외 흐름을 섞지 않는다.

### 원칙 4. 플로라 개입 지점을 명시한다
어디서 플로라가 요약/경고/업무 생성/브리핑을 해야 하는지 표시한다.

## 3. PRESSCO21 핵심 업무 흐름 목록

1. B2B 견적-문서-입금-정산 흐름
2. 상품 주문-결제-출고-사후처리 흐름
3. 파트너 신청-승인-온보딩 흐름
4. 클래스 등록-승인-모집-운영-정산 흐름
5. 재고 감시-품절-보충 흐름
6. 플로라 운영 브리핑 흐름

---

## 4. 흐름 A. B2B 견적 → 문서 → 입금 → 정산

### 목적
기관/협회/기업/공방 등 B2B 거래를 체계적으로 관리한다.

### 핵심 엔터티
- Organization
- Customer
- Quote
- Invoice / TransactionDocument
- Payment
- Settlement
- Task
- Approval
- EventLog

### 정상 흐름
1. 거래처 또는 담당자와 상담
2. `Quote` 생성 (`draft` → `issued`)
3. 내부 검토 또는 대표 확인 필요 시 `Approval.requested`
4. 견적 확정 후 `Invoice/TransactionDocument` 발행
5. 입금 대기 (`Payment.unpaid` / `pending_confirmation`)
6. 입금 확인 (`Payment.paid`)
7. 대사 완료 (`Payment.reconciled`)
8. 정산 필요 시 `Settlement.collecting → calculated → approved → paid_out`
9. 거래 종료 후 `EventLog`와 후속 `Task` 정리

### 승인 포인트
- 고액 견적
- 특이 거래조건
- 환불/재발행
- 정산 금액 승인

### 예외 흐름
- 견적 수정 / 재발행
- 부분입금
- 문서 수정발행
- 입금 지연 / 미수금
- 정산 보류

### 자동화 후보
- 견적 발행 후 입금 대기 리마인드
- 미입금 건 경보
- 재발행 이력 정리
- 정산 대상 자동 수집

### 플로라 개입 지점
- 미입금/미정산 건 브리핑
- 고액 견적/예외 거래 알림
- 대표 승인 필요 건 요약
- 거래 상태 한눈에 정리

---

## 5. 흐름 B. 상품 주문 → 결제 → 출고 → 사후처리

### 목적
자사몰/채널 상품 판매 흐름을 안정적으로 관리한다.

### 핵심 엔터티
- Customer
- Order
- Product
- Inventory
- Payment
- Task
- EventLog

### 정상 흐름
1. 고객 주문 생성 (`Order.placed`)
2. 결제 대기 또는 즉시 결제 (`Payment.unpaid` → `paid`)
3. 주문 준비 (`Order.preparing`)
4. 재고 확인 / 차감
5. 출고 (`Order.shipped`)
6. 배송 완료 (`Order.delivered`)
7. 필요 시 CS 후속처리

### 승인 포인트
- 특수포장 / 예외출고
- 대량주문 특이 처리
- 환불/교환 승인

### 예외 흐름
- 주문 취소
- 부분 배송
- 품절 대체 안내
- 반품 / 교환
- 고객 문의 반복 대응

### 자동화 후보
- 재고 임계치 알림
- 품절/품절해제 후보 알림
- 출고 전 체크리스트 생성
- 반복 문의 답변 초안 생성

### 플로라 개입 지점
- 오늘 출고 우선순위 요약
- 재고 부족 경고
- 반복 CS 유형 요약
- 미처리 주문/예외 주문 알림

---

## 6. 흐름 C. 파트너 신청 → 승인 → 온보딩

### 목적
파트너 생태계를 체계적으로 모집하고 승인/활성화한다.

### 핵심 엔터티
- Partner
- Application
- Organization
- Customer
- Approval
- Task
- EventLog

### 정상 흐름
1. 파트너 신청 접수 (`Application.submitted`)
2. 검토 (`Application.under_review`)
3. 승인 판단 (`Approval.requested → approved/rejected`)
4. 승인 시 `Partner.approved`
5. 온보딩 진행 (`Partner.onboarding`)
6. 활성 전환 (`Partner.active`)

### 승인 포인트
- 파트너 수용 여부
- 등급 결정
- 정산 정책 적용
- 지역/역할 적합성 판단

### 예외 흐름
- 신청 정보 부족
- 보류 (`on_hold`)
- 반려 (`rejected`)
- 승인 후 미활성 / 휴면

### 자동화 후보
- 신청 접수 알림
- 검토 대기 건 리마인드
- 온보딩 체크리스트 생성
- 휴면 파트너 감시

### 플로라 개입 지점
- 신규 신청 요약
- 승인 대기 건 브리핑
- 온보딩 누락 항목 알림
- 휴면/이탈 위험 파트너 경고

---

## 7. 흐름 D. 클래스 등록 → 승인 → 모집 → 운영 → 정산

### 목적
파트너클래스 핵심 운영 흐름을 관리한다.

### 핵심 엔터티
- Class
- Instructor
- Partner
- Application
- Payment
- Settlement
- Review
- Task
- EventLog

### 정상 흐름
1. 클래스 초안 생성 (`Class.draft`)
2. 등록 요청 (`submitted`)
3. 검토/승인 (`reviewing → approved`)
4. 공개/모집 (`published → recruiting`)
5. 신청 접수 (`Application.submitted`)
6. 결제/입금 처리 (`Payment.unpaid → paid`)
7. 운영 진행 (`Class.running`)
8. 종료 (`Class.completed`)
9. 리뷰/후속 관리
10. 정산 (`Settlement.collecting → paid_out`)

### 승인 포인트
- 클래스 공개 승인
- 가격/정산 조건 승인
- 운영 주체 확인

### 예외 흐름
- 반려 클래스
- 모집 중 취소
- 신청 취소 / 환불
- 정산 보류
- 강사/파트너 변경

### 자동화 후보
- 신청 접수 안내
- 모집 마감 리마인드
- 리뷰 요청 자동화
- 정산 대상 자동 수집

### 플로라 개입 지점
- 승인 대기 클래스 브리핑
- 모집중이지만 신청 저조한 클래스 경고
- 정산 대기 건 요약
- 리뷰/재참여 유도 포인트 정리

---

## 8. 흐름 E. 재고 감시 → 품절 → 보충

### 목적
재고 부족/품절/보충 흐름을 체계화한다.

### 핵심 엔터티
- Product
- Inventory
- Task
- EventLog

### 정상 흐름
1. 재고 수량 변동
2. 안전재고/임계치 비교
3. 경고 발생
4. 보충 여부 판단
5. 품절/품절해제 반영
6. 후속 발주 또는 생산 준비

### 승인 포인트
- 품절 처리 확정
- 대체 상품 판단
- 재입고 정책

### 자동화 후보
- 임계치 도달 알림
- 품절 후보 목록 자동 생성
- 반복 보충 품목 우선순위 정리

### 플로라 개입 지점
- 재고 부족 경고
- 품절 위험 상품 브리핑
- 보충 우선순위 추천

---

## 9. 흐름 F. 플로라 운영 브리핑 흐름

### 목적
원천데이터와 상태를 기반으로 대표/실무자에게 actionable 브리핑을 제공한다.

### 핵심 엔터티
- StatusCode
- Task
- Approval
- EventLog
- BriefingSnapshot (파생)

### 정상 흐름
1. 원천데이터와 상태 변화 수집
2. 중요 이슈 추출
3. 대기/지연/위험 항목 분류
4. 담당자/우선순위 기준 정리
5. 브리핑 생성
6. 필요 시 후속 Task/Approval 생성

### 대표 브리핑 우선 항목
- 미입금 / 미정산
- 승인 대기
- 파트너 위험
- 진행 막힘
- 오늘 우선 처리 3건

### 실무 브리핑 우선 항목
- 오늘 해야 할 일
- 대기중 업무
- 실패 자동화
- 재고/출고/문서 예외

### 자동화 후보
- 아침 브리핑
- 저녁 미완료 건 요약
- 주간 병목 정리

---

## 10. 흐름 간 연결 포인트

### 견적 흐름 ↔ 정산 흐름
B2B 거래는 견적/문서/입금 이후 정산으로 이어질 수 있다.

### 클래스 흐름 ↔ 상품 흐름
클래스는 상품(재료/키트) 판매와 연결된다.

### 파트너 흐름 ↔ 클래스 흐름
파트너 승인 이후 클래스 운영 주체가 된다.

### 주문 흐름 ↔ 재고 흐름
모든 판매는 재고 흐름과 연결된다.

### 모든 흐름 ↔ 플로라 브리핑 흐름
플로라는 각 흐름의 상태 이상 징후를 요약하고 알린다.

---

## 11. 자동화 우선 후보

현재 기준 우선순위:

1. 미입금/미정산 브리핑
2. 파트너 신청/승인 대기 알림
3. 클래스 승인/모집 상태 브리핑
4. 재고 부족 경고
5. 업무(Task) 대기/막힘 요약

---

## 12. 미확정/확인 필요 사항

1. B2B 문서 흐름에서 Quote와 Invoice 사이 실제 예외 규칙
2. 파트너 승인 후 온보딩 단계 상세 항목
3. 클래스 운영 종료 후 리뷰/재참여 흐름 여부
4. 재고 흐름에서 생산/발주와 연결되는 엔터티 추가 필요성
5. 플로라 브리핑 주기와 대상별 포맷 차이
