# PRESSCO21 DB 의사결정 지침 v1

_대상: 지호님, 플로라, Claude Code, Codex, 향후 개발 에이전트_

## 1. 문서 목적

이 문서는 PRESSCO21의 향후 개발에서 **어떤 데이터베이스/데이터 플랫폼을 어떤 상황에 선택할지**에 대한 공통 기준이다.

목표는 아래 4가지다.

1. 데이터 구조를 일관되게 유지한다.
2. 개발 에이전트가 프로젝트마다 DB를 제각각 고르지 않게 한다.
3. 운영 장부, 제품 개발, 자동화, 플로라 OS를 서로 충돌 없이 연결한다.
4. 무료/저비용으로 시작하되 장기 확장성을 해치지 않는다.

---

## 2. 최상위 결론

PRESSCO21의 기본 방향은 다음과 같다.

- **핵심 원장 DB 표준은 PostgreSQL로 간다.**
- **NocoDB는 내부 운영 장부/관리 UI 레이어로 유지한다.**
- **Supabase는 신규 제품형 서비스/포털/앱 개발 시 선택적으로 사용한다.**
- **SQLite 계열(SQLite, Turso, PocketBase)은 보조툴/실험용/로컬 단독 앱으로 제한한다.**

즉, 단일 제품 선택이 아니라 아래의 역할 분리 구조를 기본으로 한다.

- **원장 엔진:** PostgreSQL
- **운영 UI:** NocoDB
- **제품형 백엔드:** Supabase 또는 별도 앱 백엔드
- **실험/로컬 툴:** SQLite 계열

---

## 3. 핵심 원칙

### 원칙 1. DB 엔진, 운영 UI, 백엔드 플랫폼을 구분해서 판단한다

개발 시 아래를 혼동하지 않는다.

- **PostgreSQL** = 데이터 엔진 / 원장
- **NocoDB** = 운영 장부 UI / 협업 테이블 레이어
- **Supabase** = PostgreSQL 기반 BaaS / 제품 개발 플랫폼
- **SQLite/PocketBase/Turso** = 경량 앱/실험/보조 저장소

에이전트는 NocoDB를 DB로 갈아탄다, Supabase로 DB를 바꾼다 같은 모호한 표현을 쓰지 말고,
반드시 **엔진 / 운영 UI / 백엔드 플랫폼** 중 어느 계층의 선택인지 분리해서 설명한다.

### 원칙 2. 회사 핵심 데이터는 PostgreSQL 우선

아래 데이터는 장기적으로 PostgreSQL을 기준 원장으로 둔다.

- 고객
- 거래처
- 주문/신청 핵심 엔터티
- 견적/명세/입금/정산
- 파트너/강사/클래스 핵심 엔터티
- 플로라가 읽고 판단할 운영 기준 데이터
- 자동화가 공통으로 참조하는 마스터 데이터

### 원칙 3. 운영 실무 장부는 NocoDB를 적극 활용

아래 유형은 NocoDB가 잘 맞는다.

- 실무자가 직접 보고 수정하는 목록형 데이터
- 운영 장부
- CRM성 관리 테이블
- 내부 검수/체크리스트 데이터
- 상태 변경이 잦은 운영용 레코드

NocoDB는 버릴 대상이 아니라 **운영 생산성 레이어**로 본다.

### 원칙 4. Supabase는 제품형 개발에 선택 도입

아래 조건이 있으면 Supabase를 우선 검토한다.

- 외부 사용자 로그인/권한이 필요함
- 회원/강사/파트너 포털을 빠르게 만들어야 함
- 파일 업로드/스토리지가 필요함
- 실시간 상태 반영이 필요함
- 빠른 MVP가 중요함
- 프론트엔드와 직접 붙는 백엔드가 필요함

단, 회사 전체 원장을 무조건 Supabase에 일원화하는 접근은 기본 전략이 아니다.

### 원칙 5. SQLite 계열은 보조/실험 범위로 제한

아래 목적에는 허용 가능하다.

- 로컬 단독 실행 도구
- 빠른 프로토타입
- 캐시/임시 저장
- 에이전트 보조 메모리
- 소규모 유틸리티 앱

단, 아래에는 원칙적으로 사용하지 않는다.

- 회사 핵심 원장
- 다중 사용자 핵심 운영 데이터
- 정산/문서/거래 중심 시스템
- 장기 통합 대상 마스터 데이터

## 4. 플랫폼별 판단 기준

## 4.1 PostgreSQL

### 기본 평가
- 가장 범용적이고 안정적인 관계형 원장
- 장기 표준으로 적합
- NocoDB, Supabase, n8n, 앱 백엔드와 모두 잘 연결됨

### 장점
- 안정성 높음
- 확장성 높음
- SQL 생태계 풍부
- 복잡한 관계형 데이터 모델에 적합
- 향후 통합/이전 시 유리

### 단점
- 자체 UI 없음
- 운영자가 직접 쓰기엔 별도 레이어 필요
- 인증/API/권한은 별도 설계가 필요할 수 있음

### PRESSCO21에서의 기본 역할
- 회사 핵심 데이터 원장
- 통합 마스터 데이터 저장소
- CRM/파트너클래스/자동화의 공통 데이터 기반

### 기본 판정
- **장기 표준 채택**

---

## 4.2 NocoDB

### 기본 평가
- SQL DB 위에 올리는 운영/협업 UI
- 운영 장부와 실무 편집에 강함

### 장점
- 실무자가 보기 쉬움
- 빠른 테이블 운영 가능
- SQL 미숙련자도 활용 가능
- self-host에 적합
- 운영 데이터 수정/조회에 생산성 높음

### 단점
- 제품형 백엔드 전체를 대체하진 못함
- 인증/실시간/스토리지/복잡 권한 정책은 별도 고려 필요
- 앱 중심 서비스의 핵심 플랫폼으로는 한계가 있음

### PRESSCO21에서의 기본 역할
- 운영 장부 UI
- 관리성 데이터 편집 레이어
- 내부 장부/검수/상태관리 테이블
- 플로라 및 자동화가 참조하는 실무 관리 화면

### 기본 판정
- **유지 및 적극 활용**
- 단, **원장 엔진 자체로 혼동하지 말 것**

---

## 4.3 Supabase

### 기본 평가
- PostgreSQL 기반 BaaS
- 신규 서비스, 포털, 인증 기반 앱 개발에 적합

### 장점
- 인증/권한/스토리지/실시간 기능이 빠르게 붙음
- 프론트엔드와 연동이 쉬움
- MVP 개발 속도가 빠름
- Postgres 기반이라 데이터 모델이 비교적 튼튼함

### 단점
- SaaS 종속성 가능성
- 무료 플랜 한계 존재
- 운영 장부 UI로는 NocoDB보다 덜 직관적일 수 있음
- 회사 전체 핵심 원장을 여기에 전부 몰아넣는 전략은 신중해야 함

### PRESSCO21에서의 기본 역할
- 파트너클래스 같은 제품형 서비스의 후보 백엔드
- 외부 사용자 포털
- 로그인 기반 앱
- 빠른 실험용 서비스 백엔드

### 기본 판정
- **선택 도입**
- 전면 전환 대상이 아니라 **제품형 신규 개발 옵션**

---

## 4.4 SQLite / Turso / PocketBase

### 기본 평가
- 가볍고 빠른 보조 선택지
- 단일 앱/소규모 툴/실험용에는 매우 유용

### 장점
- 셋업이 간단함
- 저비용/무료로 시작 쉬움
- 로컬 실행/휴대성 좋음
- 작은 도구 제작 속도 빠름

### 단점
- 핵심 원장으로 커질수록 한계
- 다중 사용자/복잡 관계형 운영 시스템에는 불리
- 회사 통합 원장 표준으로는 부적합

### PRESSCO21에서의 기본 역할
- 실험용 도구
- 로컬 단독 유틸
- 캐시/임시 저장소
- 에이전트 부가 기능의 경량 저장층

### 기본 판정
- **보조용 허용**
- 핵심 원장에는 기본 비권장

---

## 5. 상황별 선택 규칙

## 상황 A. 내부 운영 장부/실무 관리 시스템
예시:
- 고객 목록
- 파트너 관리
- 신청 현황
- 운영 체크리스트
- 상태 추적 테이블

### 권장 구조
- **PostgreSQL + NocoDB**

### 이유
- 관계형 원장은 PostgreSQL이 담당
- 실무자 운영 UI는 NocoDB가 담당
- 자동화와 연결도 쉬움

### 에이전트 지침
Claude Code / Codex는 이런 요청에서 Supabase를 기본 제안으로 삼지 말고,
먼저 **PostgreSQL 원장 + NocoDB 운영 UI** 구조를 검토한다.

---

## 상황 B. 신규 외부 사용자용 서비스/MVP
예시:
- 파트너 포털
- 강사/회원용 웹앱
- 로그인/권한/파일 업로드가 필요한 서비스
- 실시간 기능이 있는 앱

### 권장 구조
- **Supabase 우선 검토**
- 또는 **PostgreSQL + 별도 앱 백엔드**

### 이유
- 제품 기능을 빠르게 붙이기 좋음
- 인증/스토리지/실시간 지원이 빠름

### 에이전트 지침
Claude Code / Codex는 제품 MVP 요청에서 아래를 먼저 판단한다.

1. 외부 사용자 로그인 필요한가?
2. 파일 업로드 필요한가?
3. 실시간 기능 필요한가?
4. 개발 속도가 우선인가?

셋 이상이 Yes면 Supabase를 우선 제안한다.

---

## 상황 C. 로컬 단독 실행 툴 / 소규모 자동화 보조 앱
예시:
- 개인 작업용 유틸
- 임시 배치 도구
- 로컬 분석 앱
- 에이전트 부가 기능 프로토타입

### 권장 구조
- **SQLite / PocketBase / Turso**

### 이유
- 단순하고 빠름
- 개발 비용 낮음
- 운영 부담 적음

### 에이전트 지침
핵심 원장과 분리된 보조툴일 때만 허용한다.
핵심 원장과 나중에 합쳐질 가능성이 크면 처음부터 PostgreSQL 쪽으로 설계한다.

---

## 상황 D. 회사 공통 마스터 / 장기 통합 데이터
예시:
- 고객 마스터
- 거래처 마스터
- 정산 기준 데이터
- 파트너/강사 마스터
- 플로라 참조 기준 데이터

### 권장 구조
- **PostgreSQL 고정**

### 에이전트 지침
이 범주에는 SQLite, PocketBase, 단독 Supabase 전용 스키마 고립 구조를 우선 제안하지 않는다.
반드시 장기 통합 가능성을 먼저 본다.

---

## 6. Claude Code / Codex용 실무 지침

## 6.1 기본 행동 규칙

Claude Code와 Codex는 새 기능/시스템/프로토타입을 설계할 때 아래 순서로 판단한다.

1. 이 기능이 **운영 장부형**인지, **제품형 서비스**인지, **보조툴**인지 분류한다.
2. 데이터가 회사 핵심 원장에 들어갈지 판단한다.
3. 핵심 원장이면 PostgreSQL 기준으로 설계한다.
4. 운영자 편집 UI가 필요하면 NocoDB 연결 가능성을 검토한다.
5. 외부 사용자 기능이 강하면 Supabase를 검토한다.
6. 로컬/실험용일 때만 SQLite 계열을 검토한다.

## 6.2 금지 규칙

에이전트는 아래를 기본적으로 피한다.

- 이유 없이 새 DB 종류를 추가하는 것
- 핵심 데이터를 로컬 SQLite 파일에 먼저 넣고 나중에 옮기자는 접근
- 운영 장부 문제를 무조건 Supabase로 해결하려는 접근
- 제품형 요구사항을 NocoDB만으로 해결하려는 접근
- DB 선택 이유 없이 요즘 많이 쓰니까로 결정하는 것

## 6.3 반드시 답해야 할 질문

새 시스템 설계 시 Claude Code / Codex는 아래 질문에 답한 뒤 DB를 제안한다.

1. 이 데이터는 회사 핵심 원장인가?
2. 다중 사용자가 동시에 다루는가?
3. 운영자가 표처럼 직접 수정해야 하는가?
4. 외부 사용자 로그인/권한이 필요한가?
5. 파일 업로드/실시간 기능이 필요한가?
6. n8n/플로라가 이 데이터를 자주 참조하는가?
7. 6개월 후 다른 시스템과 통합될 가능성이 큰가?

## 6.4 추천 출력 형식

Claude Code / Codex는 DB 제안 시 아래 형식으로 답한다.

- **분류:** 운영 장부 / 제품형 서비스 / 보조툴
- **핵심 데이터 여부:** 핵심 원장 / 비핵심
- **권장 구조:** 예) PostgreSQL + NocoDB
- **이유:** 3줄 이내
- **대안:** 예) Supabase 가능하나 현재는 과함
- **리스크:** 향후 마이그레이션, 벤더 종속, 운영 부담 등

## 7. PRESSCO21 표준 권장안

현재 기준 표준 권장안은 다음과 같다.

### 7.1 회사 전체 데이터 계층
- **L1 원장층:** PostgreSQL
- **L2 운영 관리층:** NocoDB
- **L3 제품/포털층:** Supabase 또는 앱 백엔드
- **L4 자동화층:** n8n / OpenClaw / webhook
- **L5 판단층:** 플로라 OS / 리포트 / 대시보드 / 브리핑

### 7.2 시스템별 기본 방향
- **CRM:** PostgreSQL 중심, 운영 UI는 NocoDB 연계 검토
- **파트너클래스:** 제품 성격 강해지면 Supabase 검토 가능, 단 핵심 마스터는 PostgreSQL 기준 유지
- **플로라 OS:** 통합 참조 데이터는 PostgreSQL 기준으로 정리
- **자동화:** n8n이 읽고 쓰기 쉬운 구조 유지
- **회사 지식/정책:** 구조화가 필요하면 PostgreSQL 또는 문서 기반 병행

---

## 8. 의사결정 매트릭스

| 상황 | 1순위 | 2순위 | 비고 |
|---|---|---|---|
| 내부 운영 장부 | PostgreSQL + NocoDB | PostgreSQL 단독 + 별도 UI | NocoDB 적극 권장 |
| 외부 사용자용 포털 | Supabase | PostgreSQL + 별도 백엔드 | 빠른 MVP면 Supabase |
| 핵심 마스터 데이터 | PostgreSQL | PostgreSQL + NocoDB | 고정 원칙 |
| 로컬 보조 툴 | SQLite / PocketBase | Turso | 핵심 원장 분리 필수 |
| 에이전트 실험 기능 | SQLite / PocketBase | PostgreSQL | 운영 편의에 따라 선택 |
| 장기 통합 예정 서비스 | PostgreSQL 우선 | Supabase + 추후 분리 고려 | 처음부터 통합성 고려 |

## 9. 최종 운영 문장

PRESSCO21는 앞으로 데이터 구조를 아래처럼 판단한다.

> 핵심 원장은 PostgreSQL에 둔다.  
> 운영 실무 UI는 NocoDB를 적극 활용한다.  
> 제품형 신규개발은 Supabase를 선택적으로 사용한다.  
> SQLite 계열은 보조툴과 실험용으로 제한한다.

이 원칙은 플로라 종합 OS를 위한 공통 기반이며,
Claude Code와 Codex는 새 시스템을 설계할 때 이 문서를 기본 기준으로 따른다.
