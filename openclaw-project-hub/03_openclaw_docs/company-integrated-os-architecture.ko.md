# PRESSCO21 원천데이터 중심 통합관리시스템 아키텍처 v1

_대상: 지호님, 플로라, Claude Code, Codex, 향후 개발 에이전트_

## 1. 문서 목적

이 문서는 PRESSCO21의 장기 목표인 **플로라 기반 회사 종합 OS**를 만들기 위한 상위 아키텍처 기준 문서다.

이 문서의 목적은 아래와 같다.

1. 회사의 핵심 데이터를 하나의 원천 체계로 정리한다.
2. 운영, 자동화, 서비스, AI 판단이 같은 데이터 축 위에서 작동하도록 한다.
3. CRM, 파트너클래스, 자동화, 리포트, 플로라 OS가 서로 다른 방향으로 발전하지 않게 한다.
4. Claude Code와 Codex가 신규 시스템을 설계할 때 흔들리지 않도록 상위 기준을 제공한다.

---

## 2. 최상위 비전

PRESSCO21가 지향하는 구조는 다음과 같다.

> 하나의 원천 데이터 위에  
> 운영 관리, 자동화, 서비스, 보고, AI 판단이 모두 연결되는  
> 회사 종합 운영 시스템을 만든다.

즉, 앞으로의 목표는 단순한 툴 추가가 아니라 아래와 같은 구조다.

- **원천데이터(원장)**
- **운영 시스템(CRM/NocoDB/관리 화면)**
- **자동화 시스템(n8n/OpenClaw/webhook)**
- **외부 서비스/포털(파트너클래스 등)**
- **판단/브리핑/오케스트레이션(플로라 OS)**

이 구조의 핵심은 **도구를 통합하는 것**이 아니라,
**데이터 원천을 통일하고 각 도구를 그 위에 연결하는 것**이다.

---

## 3. 최상위 원칙

### 원칙 1. 원천데이터는 가능하면 하나의 중심 원장으로 유지한다

핵심 운영 데이터는 가능한 한 하나의 중심 원장에 둔다.
기본 원장은 **PostgreSQL**을 표준으로 한다.

이 원칙의 목적은 아래와 같다.

- 중복 입력 감소
- 데이터 불일치 방지
- 자동화 기준 통일
- 플로라 판단 정확도 향상
- 향후 통합 비용 절감

### 원칙 2. 운영 UI와 원장은 분리하되, 데이터는 분리하지 않는다

NocoDB, CRM, 대시보드, 관리화면은 **원장을 다루는 인터페이스**로 본다.
운영 화면이 데이터를 소유하는 구조를 지양한다.

즉,
- UI는 여러 개일 수 있다.
- 원장은 최대한 하나를 유지한다.

### 원칙 3. 자동화도 별도 진실을 만들지 않는다

n8n, OpenClaw, webhook 기반 자동화는 원장을 읽고,
필요한 결과를 다시 원장에 기록하는 구조를 기본으로 한다.

자동화가 별도의 상태 체계를 갖고 핵심 상태를 따로 관리하는 구조는 지양한다.

### 원칙 4. 외부 서비스도 원장과 연결 가능한 구조로 설계한다

파트너클래스 같은 외부 서비스/포털은 별도 서비스 계층일 수 있으나,
회사 핵심 마스터 데이터와 장기적으로 연결 가능한 구조로 설계한다.

### 원칙 5. 플로라는 상단 AI 인터페이스이며, 원장의 대체물이 아니다

플로라는 DB가 아니다.
플로라는 회사 데이터를 읽고,
의사결정/우선순위/브리핑/자동화 트리거를 담당하는 상단 운영 인터페이스다.

## 4. PRESSCO21 통합 아키텍처 계층

## L1. 원천데이터 계층 (Source of Truth)

### 역할
- 회사의 핵심 사실 데이터를 저장
- 공통 마스터 데이터 유지
- 상태값과 관계 구조의 기준 제공

### 기본 기술 방향
- **PostgreSQL 표준**

### 포함 데이터 예시
- 고객
- 거래처
- 파트너
- 강사
- 클래스
- 신청
- 결제/입금
- 견적
- 거래명세
- 정산
- 담당자
- 상태코드
- 공통 분류체계
- 운영 이벤트 로그

### 절대 원칙
- 핵심 마스터는 중복 원장을 만들지 않는다.
- 로컬 파일DB, 임시 툴, 서비스별 DB에 동일 마스터를 복제해 주 원장처럼 운영하지 않는다.

---

## L2. 운영 관리 계층

### 역할
- 실무자가 데이터를 입력/수정/조회
- 운영 상태를 관리
- 검수/체크리스트/실행 업무를 수행

### 대표 시스템
- NocoDB
- 내부 CRM
- 운영용 대시보드
- 관리용 리스트/폼 화면

### 원칙
- 운영 계층은 원장을 보는 창구다.
- 운영 편의 때문에 별도 원장을 만들지 않는다.
- 상태값, 코드체계, 기준 엔터티는 원장을 따라야 한다.

---

## L3. 자동화 계층

### 역할
- 상태 변화 감지
- 반복 업무 자동 처리
- 외부 시스템 연동
- 문서 생성/알림/분류/기록

### 대표 시스템
- n8n
- OpenClaw cron / wake / orchestration
- webhook 기반 실행기

### 원칙
- 자동화는 원장을 기준으로 읽고 쓴다.
- 자동화 실패 시 복구 가능한 이벤트/로그를 남긴다.
- 자동화만 알고 있는 숨은 상태를 만들지 않는다.

---

## L4. 서비스/포털 계층

### 역할
- 외부 사용자용 기능 제공
- 회원/파트너/강사/수강생 인터페이스 제공
- 서비스형 기능 구현

### 대표 시스템 예시
- 파트너클래스 포털
- 강사/파트너용 전용 페이지
- 신청/조회/자료 접근 서비스

### 기술 방향
- 필요 시 Supabase 또는 별도 앱 백엔드 사용 가능

### 원칙
- 핵심 마스터는 원장과 연결 가능한 구조로 설계한다.
- 서비스 편의 때문에 회사 핵심 데이터를 별도 진실로 고립시키지 않는다.
- 인증/세션/스토리지/캐시 데이터와 핵심 마스터 데이터를 구분한다.

---

## L5. 판단/브리핑/오케스트레이션 계층

### 역할
- 대표 요청 해석
- 업무 우선순위 결정
- 상태 요약 및 브리핑
- 다음 액션 제안
- 자동화 및 서브시스템 호출

### 대표 시스템
- 플로라 Frontdoor
- 전문 비서 모드들
- 향후 대표용 종합 브리핑 시스템

### 원칙
- 플로라는 원장을 읽고 판단한다.
- 플로라가 생성하는 분석/초안/판단 결과도 추적 가능해야 한다.
- 플로라는 여러 시스템의 분산 상태를 임의로 해석하는 것이 아니라,
  통합된 기준 데이터 위에서 작동해야 한다.

## 6. 절대 분산하면 안 되는 데이터

아래 데이터는 회사 종합 OS 기준에서 **원천 데이터 분산 금지 대상**이다.

- 고객 마스터
- 거래처 마스터
- 파트너/강사 마스터
- 클래스 핵심 마스터
- 신청 핵심 상태
- 견적/명세/입금/정산 핵심 데이터
- 공통 상태코드 체계
- 담당자/권한 기준 정보
- 자동화 기준이 되는 마스터 데이터

이 데이터들은 여러 도구에서 보일 수는 있지만,
각 도구가 자기 DB 안에서 별도 진실처럼 관리하는 구조는 피한다.

---

## 7. 허용 가능한 분리

모든 데이터를 무조건 하나의 물리DB에만 둬야 한다는 뜻은 아니다.
아래 범위의 분리는 허용 가능하다.

## 7.1 서비스 보조 데이터
- 세션
- 캐시
- 임시 업로드 메타
- 인증 토큰
- 실시간 연결 상태
- 서비스 전용 로그

## 7.2 로컬 실험 데이터
- 프로토타입용 샘플 데이터
- 로컬 테스트 레코드
- UI 실험용 SQLite 데이터

## 7.3 분석/집계용 파생 데이터
- 리포트용 집계 테이블
- 브리핑용 캐시
- 검색 최적화용 인덱싱 데이터

### 단, 조건
- 원천 데이터와 구분되어야 한다.
- 재생성 가능해야 한다.
- 주 원장으로 오인되면 안 된다.

## 8. 시스템별 역할 정의

## 8.1 CRM
### 역할
- 내부 운영 실행의 중심 화면
- 거래/명세/고객/정산 흐름을 관리

### 원칙
- 핵심 데이터는 원장 기준
- 운영 상태/문서 흐름/검수 규칙을 명확히 가져감
- 보고와 자동화의 기준점 역할 수행

## 8.2 NocoDB
### 역할
- 실무 데이터 관리 인터페이스
- 빠른 장부 편집/조회

### 원칙
- 원장을 보는 UI
- 별도 독립 진실을 만들지 않음

## 8.3 n8n
### 역할
- 반복 업무 자동화
- 외부 시스템 연동
- 경보/알림/문서 생성

### 원칙
- 원장 읽기/쓰기 기반
- 숨은 상태를 만들지 않음

## 8.4 파트너클래스
### 역할
- 외부 사용자/파트너/강사/교육 흐름 서비스
- 회사 신규 성장축 후보

### 원칙
- 서비스형 UX는 독립적으로 발전 가능
- 핵심 마스터는 통합 가능한 구조 유지
- 제품 전용 상태와 회사 공통 상태를 구분

## 8.5 플로라 OS
### 역할
- 대표와 회사 시스템 사이의 종합 인터페이스
- 요청 분류, 요약, 판단, 우선순위화, 실행 연결

### 원칙
- 원천데이터 기반으로 판단
- 업무 맥락을 통합해 보여줌
- 장기적으로 회사 전체 운영 브리퍼 역할 수행

## 9. Claude Code / Codex 설계 지침

## 9.1 설계 시작 전 반드시 답할 질문

새 기능/시스템 설계 시 아래를 먼저 답한다.

1. 이 기능은 원천데이터를 생성/수정하는가?
2. 이 데이터는 핵심 마스터인가, 파생/보조 데이터인가?
3. 이 기능은 운영 실행용인가, 외부 서비스용인가, 자동화용인가?
4. 이 데이터는 다른 시스템에서도 참조될 가능성이 높은가?
5. 플로라가 이 데이터를 기준으로 판단/브리핑할 가능성이 있는가?
6. 나중에 CRM, 파트너클래스, 자동화와 통합될 가능성이 있는가?

핵심 마스터/공통 참조 가능성이 높다면,
먼저 원장 계층에서 모델링한다.

## 9.2 기본 설계 순서

Claude Code와 Codex는 아래 순서로 설계한다.

1. 엔터티 분류
2. 원천데이터 여부 판단
3. 상태코드/관계구조 정의
4. 원장 반영 방식 결정
5. 운영 UI 또는 서비스 UI 결정
6. 자동화 연결점 정의
7. 플로라가 읽을 수 있는 요약 포인트 정의

## 9.3 금지 규칙

아래는 기본 금지/비권장이다.

- 서비스 편의 때문에 핵심 마스터를 별도 DB에 중복 운영하는 것
- 각 프로젝트가 고객/파트너/클래스를 자기 방식으로 따로 정의하는 것
- 자동화 내부 상태를 원장보다 더 믿는 구조
- 원천데이터와 파생데이터 구분 없이 설계하는 것
- 로컬 프로토타입 DB가 그대로 운영 원장이 되는 것

## 9.4 추천 출력 형식

새 기능 설계 제안 시 아래 형식으로 설명한다.

- **기능 분류:** 운영 / 자동화 / 서비스 / 브리핑
- **핵심 엔터티:** 예) 고객, 신청, 클래스
- **원천데이터 여부:** 원장 / 파생 / 캐시
- **저장 위치 권장:** PostgreSQL 원장 / NocoDB UI / Supabase 서비스층 / 보조 저장소
- **연결 시스템:** CRM / n8n / 플로라 / 파트너클래스
- **통합 리스크:** 중복마스터, 상태불일치, 벤더종속 등

## 10. PRESSCO21 장기 목표 상태

이 문서 기준에서 PRESSCO21의 장기 목표 상태는 다음과 같다.

### 목표 상태 A. 단일 원천데이터 체계 확보
- 핵심 마스터가 통일됨
- 상태코드 체계가 통일됨
- 중복 장부 감소

### 목표 상태 B. 운영 실행 체계 정리
- CRM/NocoDB가 역할 분리된 상태로 안정화
- 운영 데이터 입력/수정 흐름 명확화

### 목표 상태 C. 자동화 체계 정리
- n8n/OpenClaw가 원장 기준으로 작동
- 실패 복구/로그 체계 정리

### 목표 상태 D. 서비스 계층 정리
- 파트너클래스 등 외부 서비스가 통합 가능한 구조로 정리

### 목표 상태 E. 플로라 종합 OS 완성
- 대표가 텔레그램에서 회사 운영 상태를 통합 조회 가능
- 우선순위/미처리 건/리스크/다음 행동을 브리핑 가능
- 필요한 자동화와 운영 시스템을 오케스트레이션 가능

## 11. 최종 운영 문장

PRESSCO21는 앞으로 아래 원칙으로 시스템을 설계한다.

> 하나의 원천데이터 체계를 중심으로  
> 운영, 자동화, 서비스, AI 판단을 연결한다.  
> UI와 서비스는 나눌 수 있어도  
> 핵심 원장과 핵심 마스터는 최대한 통일한다.  
> 플로라는 이 통합 원장 위에서 작동하는 회사 종합 OS로 발전시킨다.

이 문서는 PRESSCO21의 상위 시스템 설계 원칙이며,
Claude Code와 Codex는 신규 개발과 구조 제안 시 본 문서를 우선 기준으로 따른다.
