# PRESSCO21 역할/권한 모델 가이드 v0.2 (현실반영 초안)

_상태: 현실반영 초안 / 지호님 추가 검증 필요 / 실무 적용 전_

## 1. 목적
이 문서는 PRESSCO21의 사람/시스템/플로라 권한 구조를 **단순하지만 운영 현실에 맞게** 정리한 역할/권한 모델 초안이다.

목적은 아래와 같다.

1. 누가 무엇을 보고, 수정하고, 승인할 수 있는지 정한다.
2. 대표님과 지호님의 권한 구조를 현실에 맞게 반영한다.
3. 직원 권한은 과하게 복잡하지 않게 유지한다.
4. 플로라는 사용자에 따라 다른 권한 컨텍스트로 동작하게 한다.
5. 향후 CRM / NocoDB / 파트너클래스 / 플로라 OS 권한 설계의 기준으로 삼는다.

## 2. 핵심 현실 전제

### 2.1 대표와 지호님 권한
- 대표님(이진선)과 지호님은 **사실상 동등한 최상위 권한**으로 본다.
- 지호님은 예비대표이자 실질 운영/시스템 관리자이므로, 전 영역을 다룰 수 있어야 한다.
- 대표님도 필요 시 전체를 볼 수 있어야 한다.

즉,
**최상위 권한은 2인 체제**로 본다.

### 2.2 직원 권한은 단순하게
직원 권한은 너무 세분화하면 오히려 운영이 복잡해진다.
따라서 현재는 아래 2개 실무 특화군만 둔다.

- **물류/생산 특화**
- **디자인 특화**

### 2.3 플로라는 사용자 컨텍스트를 따라간다
플로라는 하나의 고정 권한이 아니라,
**누가 쓰고 있는지에 따라 허용 범위가 달라지는 구조**가 현실적이다.

- 지호님이 쓰면: 거의 전체 권한
- 대표님이 쓰면: 대표님 용도에 맞는 축소/단순화된 전체 조회 + 필요한 실행
- 실무자가 쓰면: 자기 업무 범위 중심 제한 권한

즉 플로라는 **권한 자체의 주체가 아니라, 사용자 권한을 반영하는 인터페이스**에 가깝다.

## 3. 현재 권장 역할 구조 (단순화 버전)

PRESSCO21는 현재 단계에서 아래 4개 역할군으로 시작하는 것이 가장 현실적이다.

### Role A. 최상위 관리자 (대표 / 지호님)
- 대표님
- 지호님

### Role B. 물류/생산 특화 실무자
- 물류 담당
- 생산관리 담당

### Role C. 디자인 특화 실무자
- 디자인기획
- 웹디자인
- 영상기획 / 콘텐츠 담당

### Role D. 플로라 / 자동화 시스템
- 플로라
- n8n
- OpenClaw 자동화

---

## 4. 역할별 권한 모델

## 4.1 Role A. 최상위 관리자 (대표 / 지호님)

### 성격
실질 운영/시스템/의사결정 전체를 다룰 수 있는 권한

### 권한 범위
- 전체 데이터 읽기
- 핵심 상태 수정
- 자동화 규칙 수정
- 승인/확정
- 문서/정산/대외 이슈 확인
- 플로라 사용 시 최고 권한 컨텍스트

### 비고
대표와 지호님은 권한상 동등하게 두되,
실제 사용 패턴은 다를 수 있다.

즉,
- 대표님은 "필요할 때 전체를 보고 판단할 수 있는 권한"
- 지호님은 "상시 운영/시스템을 다루는 권한"
으로 이해하면 된다.

---

## 4.2 Role B. 물류/생산 특화 실무자

### 포함 대상
- 물류 운영
- 출고/CS 담당
- 생산관리

### 권한 범위 (초안)
- 주문/출고/재고 관련 데이터 읽기
- 자기 업무 범위 내 저위험 상태 수정
- 고객 응대 보조용 정보 조회
- 생산/재고/출고 관련 Task 처리

### 제한 권한
- 가격 확정
- 정산 확정
- 대외 공식 발송
- 핵심 승인 처리
- 전사 자동화 규칙 수정

### 주의
물류와 생산을 세부적으로 또 나누는 건 현재 단계에선 과할 수 있으므로,
처음엔 한 그룹으로 시작하는 것이 현실적이다.

---

## 4.3 Role C. 디자인 특화 실무자

### 포함 대상
- 디자인기획
- 웹디자인
- 영상기획
- 콘텐츠 제작

### 권한 범위 (초안)
- 콘텐츠 자산/요청/초안 읽기
- 상세페이지/배너/영상/기획 초안 관련 정보 접근
- 자기 업무 범위 내 문서/콘텐츠 초안 수정
- 작업 요청 / 진행 상태 확인

### 제한 권한
- 가격 확정
- 정산 확정
- 핵심 상태 변경
- 대외 공식 발송 최종 실행
- 시스템 규칙 수정

### 비고
영상기획팀장과 웹디자이너는 현재 운영상 같은 "디자인 특화" 권한군으로 묶는 것이 현실적이다.

---

## 4.4 Role D. 플로라 / 자동화 시스템

### 성격
플로라는 독립 권한 주체라기보다,
**사용자 권한을 반영하는 AI 인터페이스**로 본다.

### 기본 원칙
- 넓은 읽기 가능
- 초안/요약/브리핑/Task 생성 가능
- 민감한 최종 수정/확정은 제한
- 실제 허용 범위는 사용자 컨텍스트에 따라 달라짐

### 사용자별 컨텍스트
#### 지호님이 사용할 때
- 거의 전체 읽기
- 넓은 초안 생성
- 운영/구조/자동화 제안 및 실행 보조
- 핵심 수정까지 열 수 있음

#### 대표님이 사용할 때
- 전체 조회 가능
- 쉬운 언어 브리핑
- 복잡한 수정은 최소화
- 필요한 승인/확인 중심

#### 실무자가 사용할 때
- 자기 업무 범위 중심 읽기
- 초안/요약/체크리스트/Task 보조
- 민감 데이터/전사 수정 권한 없음

## 5. 권한 레벨 단순화 모델

현재 단계에서는 권한 레벨도 너무 세밀하지 않게 아래 5개로 시작하는 것이 좋다.

- `R` = 읽기
- `W-basic` = 자기 업무 범위 내 수정
- `W-ops` = 운영 구조/핵심 상태 수정
- `Approve` = 승인/확정
- `Admin` = 시스템/자동화/권한 전체 관리

### 권장 매핑
- 대표: `R + W-ops + Approve + Admin`
- 지호님: `R + W-ops + Approve + Admin`
- 물류/생산: `R + W-basic`
- 디자인 특화: `R + W-basic`
- 플로라: 사용자 컨텍스트를 따름

## 6. 현실적 구현 권장안

이 문서는 구조 문서지만,
실제 시스템 구현은 더 단순하게 시작하는 것이 좋다.

### 1차 구현 권장안
- `super_admin` = 대표 + 지호님
- `ops_staff` = 물류/생산
- `creative_staff` = 디자인/콘텐츠
- `ai_proxy` = 플로라

즉 문서는 조금 더 풍부하게 유지하되,
실제 구현은 **4개 권한군**으로 먼저 시작하는 것이 좋다.

## 7. 플로라 권한 설계 원칙

### 원칙 1
플로라는 항상 "누가 쓰는가"를 먼저 본다.

### 원칙 2
지호님 컨텍스트에선 넓게 열어도 된다.

### 원칙 3
실무자 컨텍스트에선 읽기와 초안 위주로 제한한다.

### 원칙 4
대표님 컨텍스트에선 전체를 볼 수 있으되,
사용 경험은 쉽게 단순화한다.

## 8. 지금 단계에서 너무 과한 것

아래는 현재 단계에서 너무 정교하게 들어가면 오히려 운영이 힘들 수 있다.

- 직원별 세부 permission 수십 개로 쪼개기
- 생산/물류를 또 세분화하기
- 디자인/영상/웹을 각각 따로 role로 빼기
- 플로라 권한을 독립 고정 role처럼 만들기

현재 단계에선
**단순한 권한군 + 지호님 중심 운영**이 더 현실적이다.

## 9. 지호님 현실 체크 포인트

다음은 추가로 확인하면 좋은 항목이다.

1. 대표와 지호님을 시스템상 완전히 같은 role로 둘지, 같은 권한이지만 표현만 다르게 둘지
2. 물류/생산을 장기적으로도 한 그룹으로 둘지
3. 디자인 특화 안에서 영상/웹을 나중에 나눌 필요가 생길지
4. 플로라가 실무자 대신 상태를 바꾸는 범위를 어디까지 열어둘지

## 10. 최종 운영 문장 (현실반영 초안)

> PRESSCO21 권한 모델은 대표와 지호님을 동등한 최상위 관리자 권한으로 두고, 실무자는 물류/생산 특화와 디자인 특화 두 그룹으로 단순화한다.  
> 플로라는 고정 권한 주체가 아니라, 누가 쓰는지에 따라 권한이 달라지는 AI 인터페이스로 본다.  
> 처음 구현은 단순하게 시작하고, 실제 운영 필요가 생길 때만 세분화한다.
