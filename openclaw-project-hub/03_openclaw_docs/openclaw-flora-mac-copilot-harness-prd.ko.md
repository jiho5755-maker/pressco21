# Flora Mac Copilot Harness PRD

## 1. 문서 개요

- 문서명: Flora Mac Copilot Harness PRD
- 버전: v1.0
- 작성일: 2026-03-31
- 기준 대화: 2026-03-31 텔레그램 `플로라` direct 세션

## 2. 문제 정의

현재 플로라는 텔레그램 기반 메모, 리마인더, 운영 보조, 답변 초안 작성은 가능하지만 아래 한계가 있다.

1. 맥북 로컬의 실제 개발 폴더와 문서 구조를 지속적으로 읽지 못한다.
2. 현재 어떤 프로젝트가 살아 있고 무엇이 중복되며 어디가 병목인지 구조적으로 파악하지 못한다.
3. 로그인된 운영 시스템과 로컬 개발 산출물 사이를 하나의 업무 하네스로 연결하지 못한다.
4. PRD, 코드, 운영 메모, 관리자 페이지 분석이 한 세션 안에서 이어지지 않는다.

## 3. 목표

플로라를 텔레그램 응답형 비서에서 다음 수준으로 끌어올린다.

1. 맥북 로컬 업무 폴더를 안전한 범위 안에서 인벤토리화한다.
2. 로컬 프로젝트 상태를 서버 OpenClaw가 참조 가능한 컨텍스트로 동기화한다.
3. 텔레그램에서 "이 프로젝트 분석", "중복 구조 정리", "다음 개발 우선순위 제안" 같은 요청을 바로 처리할 수 있게 한다.
4. 향후 로그인 필요한 운영 시스템 분석까지 연결 가능한 운영 하네스 기반을 마련한다.

## 4. 비목표

이번 v1에서 바로 하지 않는 것:

1. 맥북 전체 파일 시스템 전체 공개
2. 비밀번호를 텔레그램 채팅에 직접 전달받아 로그인하는 방식
3. 개인 사진, 개인 문서, 인증서, 키체인 등 민감 자산 자동 수집
4. 관리자 페이지 자동 조작까지 포함한 완전 자율 운영

## 5. 핵심 사용자 스토리

### 5.1 인벤토리 기반 분석

- 대표가 텔레그램에서 "현재 개발 중인 폴더 전부 분석해서 중복 시스템과 우선순위 정리해줘"라고 요청한다.
- 플로라는 최신 맥북 인벤토리를 읽고 프로젝트별 목적, 상태, 위험요소, 중복도를 정리한다.

### 5.2 프로젝트 코파일럿

- 대표가 텔레그램에서 특정 프로젝트를 지정한다.
- 플로라는 해당 프로젝트의 스택, 최근 변경 흔적, 연관 문서, 자동화 연결 지점을 근거로 PRD/실행안을 제안한다.

### 5.3 운영 시스템 확장 기반

- 추후 메이크샵 관리자, 사방넷, 쿠팡 서플라이어 허브 같은 로그인 시스템을 다루더라도
- 먼저 로컬 인벤토리와 접근 정책표가 준비되어 있어 어떤 시스템을 어떤 방식으로 붙일지 결정할 수 있다.

## 6. 요구사항

### 6.1 기능 요구사항

1. 맥북에서 지정한 최상위 폴더 목록을 읽어 프로젝트 인벤토리 JSON을 생성해야 한다.
2. 인벤토리는 프로젝트 루트, 스택 힌트, 핵심 파일, 최근 수정 시각, 위험 플래그를 포함해야 한다.
3. 제외 경로 규칙을 지원해야 한다.
4. 생성된 인벤토리를 OpenClaw 서버의 워크스페이스 컨텍스트 폴더로 동기화할 수 있어야 한다.
5. 텔레그램 세션에서 참조 가능한 문서 경로로 저장되어야 한다.
6. 설정 파일은 사람이 손으로 수정 가능한 JSON 형식이어야 한다.

### 6.2 비기능 요구사항

1. 민감 폴더는 기본 제외 규칙이 있어야 한다.
2. 인벤토리 생성은 읽기 전용이어야 한다.
3. 대용량 폴더에서도 무한 순회 없이 `maxDepth`와 제외 규칙으로 제어되어야 한다.
4. 생성 산출물은 JSON으로 검증 가능해야 한다.
5. 서버 동기화는 SSH 키 기반이어야 한다.

## 7. 정보 구조

### 7.1 로컬 입력

- 하네스 설정 JSON
- 허용 루트 폴더 목록
- 제외 glob 목록
- 프로젝트 판별 기준 파일

### 7.2 생성 산출물

- `inventory.json`
- `summary.json`
- 이후 단계에서 추가될 `access-policy.json`, `meta-prompt.md`

### 7.3 서버 적재 위치

- `/home/ubuntu/.openclaw/workspace-owner/context/flora-mac-harness/`

## 8. 보안 원칙

1. `.env`, `.secrets`, 인증서, 키체인, 개인 문서/사진은 기본 제외한다.
2. 텔레그램으로 아이디/비밀번호를 직접 주고받는 흐름은 금지한다.
3. 로그인 시스템은 추후 세션 유지 또는 브라우저 프로필 연결 방식으로 다룬다.
4. 우선은 "읽기/분석" 하네스부터 구축한다.

## 9. 단계별 로드맵

### Phase 1. 로컬 인벤토리 하네스

1. 최상위 폴더 목록 정의
2. 제외 규칙 정의
3. 인벤토리 생성 스크립트 구현
4. 서버 동기화 스크립트 구현
5. OpenClaw에서 참조 가능한 컨텍스트 경로 확보

### Phase 2. 분석 프롬프트 하네스

1. 인벤토리 기반 메타 프롬프트 생성
2. 프로젝트 우선순위/중복도/위험도 리포트 자동 생성
3. 텔레그램 요청 템플릿 정리

### Phase 3. 로그인 시스템 분석 하네스

1. 접근 정책표 작성
2. 시스템별 세션 유지 방식 설계
3. 직접 조작보다 우선 읽기/캡처/분석 흐름 구축

## 10. 성공 기준

1. 맥북에서 인벤토리를 생성해 서버로 보낼 수 있다.
2. 플로라가 텔레그램 요청 시 최신 인벤토리를 기준으로 프로젝트 구조를 설명할 수 있다.
3. "어디부터 손대야 하는지"를 코드베이스 근거와 함께 제안할 수 있다.
4. 민감 정보 없이도 실무 분석 품질이 눈에 띄게 올라간다.

## 11. 이번 턴에서 구현하는 범위

1. 최종 PRD 문서화

3. 로컬 인벤토리 생성 스크립트
4. 서버 컨텍스트 동기화 스크립트
