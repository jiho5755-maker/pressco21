## 2. 최종 목표

PRESSCO21 OpenClaw의 최종 목표는 다음 두 가지다.

1. 회사 문서와 현장 노하우를 직원이 바로 쓸 수 있는 공용 AI 계층으로 만든다.
2. 부족한 전문 인력을 OpenClaw 기반 전문 에이전트 팀으로 보완해, 상세페이지, 콘텐츠, CS, 운영분석 병목을 줄인다.

이 프로젝트는 "챗봇 하나 배포"가 아니라 아래 구조를 만드는 일이다.

- 공용 지식 계층
- 직군별 코치 계층
- 회사 비서 계층
- 전문 에이전트 스쿼드
- 관리자 통제 계층
- 읽기 전용 자동화 연동

## 3. 복구 후 목표 구조

### 3-1. 제어 계층

| 계층 | 역할 | 주 사용자 | 목적 |
|------|------|----------|------|
| `company-core` | 회사 지식 계층 | 전 직원 | 브랜드, 상품, 고객, 채널, 정책 근거 제공 |

| `executive-assistant` | 회사 비서/코디네이터 | 장지호, 이진선, 부서장 | 회의 준비, 액션 정리, 핸드오프, 후속 추적 |
| `company-admin` | 운영 관리자 | 장지호 | 승인 경계 관리, 로그 요약, 에이전트 호출 통제 |

### 3-2. 전문/지원 에이전트 팀

| 에이전트 | 메우는 공백 | 산출물 | 최종 승인자 |

| `video-content-planner` | 콘텐츠 플래너 | 8개 촬영 계획안, 제목 초안, 재활용 우선순위 | 장다경 |

| `operations-analyst` | 운영 분석가 | 운영 다이제스트, 병목, KPI 신호 | 장지호, 이진선 |
| `knowledge-curator` | 문서 큐레이터 | FAQ 후보, 문서 갱신 후보, SOP 변경 목록 | 장지호 |
| `executive-assistant` | 회사 비서/운영 코디네이터 | 회의 브리프, 액션리스트, 담당자 추적, 핸드오프 메모 | 장지호, 이진선 |

| 스쿼드 | 구성 | 쓰는 순간 | 최종 산출물 |
|------|------|---------|------------|
| `product-launch-squad` | coach + detail-page-planner + knowledge-curator + admin | 신상품, 키트상품, 상세페이지 기획 | 상세 브리프, 누락 자료, 문서 반영 후보 |
| `content-growth-squad` | coach + video-content-planner + knowledge-curator + admin | 월간 촬영 계획, 숏츠/릴스 운영 | 8개 촬영안, 제목 초안, 재활용 플랜 |
| `customer-response-squad` | core + coach + cs-triage-specialist + admin | 배송, 재고, 사용법, 전화주문 | 1차 응대 초안, 위험도, 담당자 경로 |
| `operations-improvement-squad` | operations-analyst + knowledge-curator + admin | 주간 리뷰, 반복 병목 증가 | 운영 요약, FAQ 후보, 다음 개선 액션 |
| `executive-support-squad` | executive-assistant + core + knowledge-curator + admin | 회의 준비, 대표 지시 정리, 부서 간 핸드오프 | 회의 브리프, 액션리스트, 후속 추적표 |

## 4. 회사 기준 우선순위

현재 회사 자료 기준 AI 도입 우선순위는 다음으로 복구한다.

1. 상세페이지 기획 자동화
2. 영상 콘텐츠 기획
3. 재고/출고 읽기 전용 보조
4. 고객 응대 초안/분류
5. 신제품 탐색/키트 구성 보조

이 순서를 유지하는 이유:

- 조승해의 가장 큰 병목은 디자인 자체가 아니라 기획 정리다.
- 장다경의 가장 큰 병목은 촬영 계획 부재다.
- 이 두 병목이 풀려야 대표의 키트 등록, 영상 소재 확보, 매출 선순환이 시작된다.
- 재고/출고와 CS는 중요하지만, 초기에는 자동 실행보다 읽기 전용 보조로 접근하는 편이 안전하다.

### 5-2. 이번 복구 비범위

- 주문/환불/가격/재고 직접 변경 자동화

## 6. 3단계 복구 전략

### 단계 A. 구조 복구

목표:

- 남아 있는 문서와 구현 자산을 "회사용 OpenClaw 운영 체계"로 다시 연결한다.

완료 조건:

- 제어 계층 4종, 전문 에이전트 5종, 지원 비서 1종, 스쿼드 5종이 문서상 일관되게 정리된다.
- 참조 문서 우선순위가 고정된다.
- `notifications.openclaw` 예시 설정과 워크포스 구성이 현재 문서와 맞아떨어진다.

즉시 할 일:

1. 이 복구 계획서를 기준 문서로 추가 유지
2. `design-draft`, `PRD`, `agent-workforce` 간 용어 차이 점검
3. 실제 운영 시 필요한 문서 경로를 단일 목록으로 고정

### 단계 B. 파일럿 복구

목표:

- 실제로 시간 절감이 나는 3개 유스케이스를 복구해 직원이 써볼 수 있게 만든다.

우선 파일럿:

1. 상세페이지 브리프 생성
2. 8개 영상 촬영 계획안 생성
3. CS 1차 트리아지 초안 생성

완료 조건:

- 한 건 입력으로 실사용 가능한 JSON 또는 텍스트 초안이 나온다.
- 승인 필요 항목이 항상 분리된다.
- 누락 자료/추가 질문이 3개 이하로 정리된다.

### 단계 C. 운영 복구

목표:

- 에이전트 호출 결과가 문서 갱신과 주간 개선으로 이어지게 만든다.

완료 조건:

- 세션 종료 후 `company-admin`이 운영 기록을 남긴다.
- `operations-analyst`가 주간 병목을 정리한다.
- `knowledge-curator`가 FAQ/SOP 갱신 후보를 뽑는다.

## 9. 에이전트별 실제 활용 방식

### detail-page-planner

가장 먼저 활용한다.

- 입력: 상품명, 카테고리, 핵심 특징, 자산 보유 여부
- 출력: 섹션 구조, 카피 포인트, 누락 자료, 채널 메모
- 기대 효과: 조승해의 "기획 공백"을 먼저 줄임

### video-content-planner

두 번째로 활용한다.

- 입력: 시즌, 제품 후보, 채널 목표, 기존 영상
- 출력: 4~8주 촬영 계획, 재활용 플랜, 플랫폼 메모
- 기대 효과: 장다경의 "무엇을 찍지?" 공백 제거

### cs-triage-specialist

세 번째로 활용한다.

- 입력: 전화/카톡/게시판 문의 원문
- 출력: 분류, 위험도, 응대 초안, 담당자
- 기대 효과: 이재혁의 반복 전화 대응 시간 절감

### operations-analyst

파일럿이 시작되면 붙인다.

- 입력: 일일 로그, 반복 질문, 실패 사례
- 출력: 다이제스트, 병목, KPI 신호
- 기대 효과: 감으로 운영하지 않고 개선 루프 형성

### knowledge-curator

반드시 같이 붙인다.

- 입력: 확정된 결정, 수정된 규칙, 반복 질문
- 출력: FAQ 후보, SOP 갱신 후보
- 기대 효과: OpenClaw가 쓸수록 회사 문서가 좋아지게 만듦

### executive-assistant

가장 범용적인 프론트도어로 둔다.

- 입력: 회의 메모, 대표 지시, 우선순위, 참여자, 마감일
- 출력: 회의 브리프, 액션리스트, 담당자별 추적표, 후속 질문
- 기대 효과: 장지호와 대표가 흩어진 메모를 다시 정리하는 시간을 줄임
- 운영 역할: 다른 전문 에이전트로 넘길 작업을 먼저 분류하고 정리함

## 10. OpenClaw를 회사에서 최대 활용하는 방식

### 10-1. "답변봇"이 아니라 "준비봇"으로 쓴다

초기 OpenClaw는 고객과 직접 대화하는 봇보다, 직원의 준비 시간을 줄이는 봇으로 쓰는 것이 가장 가치가 높다.

우선순위:

- 기획안 준비
- 촬영안 준비
- 응대 초안 준비
- 운영 요약 준비
- 회의와 후속 액션 준비

### 10-2. 모든 요청은 공용 지식 계층을 거치게 한다

- `company-core` 없이 전문 에이전트가 단독 판단하지 않게 한다.
- 모든 에이전트는 회사 기준 문서에서 근거를 끌어오게 한다.
- 답변마다 승인 필요 여부를 남긴다.

### 10-3. 사람별 체감 가치를 명확히 만든다

| 사용자 | OpenClaw가 즉시 줄여야 할 시간 |
|------|----------------------------|
| 조승해 | 상세페이지 기획 준비 시간 |
| 장다경 | 촬영 아이템 선정 시간 |
| 이재혁 | 반복 전화/문의 분류 시간 |
| 이진선 | 키트/신제품 정리 전 준비 시간 |
| 장지호 | 문서 취합과 운영 정리 시간 |
| 대표/부서장 공통 | 회의 준비와 후속 조치 정리 시간 |

### 10-4. 로그를 반드시 남긴다

- 어떤 질문이 자주 나왔는지
- 어떤 문서가 부족했는지
- 어떤 승인 경계에서 자주 멈췄는지
- 어떤 스쿼드가 가장 효과가 좋았는지

이 기록이 없으면 OpenClaw는 "가끔 쓰는 도구"에서 멈춘다.

## 11. n8n 연동 우선순위

### 1순위

- 상세페이지 브리프 생성 워크플로우
- 입력 폼 -> OpenClaw -> JSON 결과 -> 검토용 문서 저장

### 2순위

- 영상 촬영 계획안 워크플로우
- 시즌/채널 목표 입력 -> 촬영안 출력 -> 재활용 링크 정리

### 3순위

- CS 트리아지 워크플로우
- 문의 입력 -> 분류/위험도/초안/담당자 반환

### 4순위

- executive-assistant 회의 브리프/액션 트래커 워크플로우
- 원문 메모 -> 액션리스트 -> 담당자/기한 정리

### 5순위

- 운영 다이제스트 워크플로우
- 일일 로그 -> 주간 요약 -> FAQ 후보 저장

원칙:

- n8n은 실행 자동화보다 입력 정리와 기록 저장에 먼저 쓴다.
- 재고/주문/가격은 읽기 전용 전까지 쓰지 않는다.

## 12. 리스크와 통제

| 리스크 | 통제 방식 |
|------|----------|
| AI가 정책을 확정해버림 | 모든 출력에 승인 필요 섹션 고정 |
| 문서가 오래됨 | `knowledge-curator`로 갱신 후보 정례화 |
| 에이전트가 많아져 관리가 복잡해짐 | 초기에는 3개 파일럿 + 2개 운영 보조만 유지 |
| 고객-facing으로 너무 빨리 확장 | 내부 전용 30일 파일럿 전제 유지 |
| 재고/가격 자동화 욕심이 커짐 | 읽기 전용 조회 전까지 차단 |

## 13. 복구 성공 기준

아래 상태가 되면 복구는 성공으로 본다.

1. OpenClaw 프로젝트 B의 목적, 역할, 유스케이스가 다시 한 문서 체계로 연결된다.
2. 조승해, 장다경, 이재혁이 각각 하나씩 실제로 써볼 수 있는 플로우가 생긴다.
3. 장지호가 세션 로그와 문서 갱신 후보를 주간 단위로 볼 수 있다.
4. OpenClaw가 회사에서 "전문가 에이전트 팀"처럼 작동하기 시작한다.

## 14. 바로 다음 액션

1. 상세페이지 브리프 플로우를 실제 테스트 입력 2건으로 검증

6. `executive-assistant` 예시 산출물을 실제 파일럿 입력값으로 전환
