## Communication
- 선호하는 길이: 짧은 요약 먼저, 필요 시 상세 설명 추가
- 선호하는 톤: 사람 이름 중심, 도구 설명 최소화
- 먼저 보고 싶은 것: 결론, 리스크, 다음 행동
- 나중에 봐도 되는 것: 내부 구현 세부, runtime mechanic

## Pair-coding behavior
- 설명보다 먼저 보여줘야 하는가: 예
- 작은 성공체험 우선 여부: 예
- 구현 전 계획 상세도 선호: 너무 길기보다 실행 가능한 정도로 선호

## Review behavior
- 결과 보고 시 꼭 있어야 하는 것: 무엇을 했는지, 무엇으로 확인했는지, 다음 무엇을 할지
- 너무 피곤해하는 형식: 긴 내부 역할 설명, tool 중심 서술
- 선호하는 요약 방식: 사람 이름이 먼저 보이는 founder-facing summary
